# Task 04.01.00: Sharing + Visibility Backend and Frontend

## Business Requirements

### Problem
Users need to share issues with other users at specific permission levels (view, comment, edit) and control whether an issue is publicly visible or private.

### User Story
As an issue owner, I want to share my issue with specific users at different permission levels, so I can collaborate while controlling access.

### Acceptance Criteria
- [ ] `POST /api/issues/{issue}/shares` — share an issue with a user (email + permission level)
- [ ] `DELETE /api/issues/{issue}/shares/{share}` — remove a share
- [ ] `PATCH /api/issues/{issue}/shares/{share}` — update share permission level
- [ ] `GET /api/issues/{issue}/shares` — list shares for an issue
- [ ] Only issue owner can manage shares (IssuePolicy::share already enforced)
- [ ] Permission levels: view, comment, edit (Permission enum already exists)
- [ ] Visibility toggle: PATCH `/api/issues/{id}` with `visibility` field already works
- [ ] Shared issues appear in the recipient's Kanban board (scopeAccessibleBy already handles this)
- [ ] All tests pass

### Business Rules
- Cannot share with yourself
- Cannot share with same user twice (upsert permission instead)
- Removing a share revokes all access for that user
- Public issues are viewable by any authenticated user (already handled by IssuePolicy::view)

### Out of Scope
- Share UI in slide-over (04.04)
- Email notifications on share
