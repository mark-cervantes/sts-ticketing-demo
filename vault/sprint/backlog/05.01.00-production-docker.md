# Task 05.01.00: Production Dockerfile + Docker Compose

## Business Requirements

### Problem
Sail's `compose.yaml` is dev-only — it mounts source code, runs Vite HMR, and lacks production optimizations. Deployment to 192.168.254.140 requires a production-ready Docker setup with pre-built assets, optimized PHP, and proper service orchestration.

### User Story
As a deployer, I want to run `docker compose up -d` on the production host and have the entire app running with migrations, seeds, and all services.

### Acceptance Criteria
- [ ] `Dockerfile` (multi-stage): composer install → npm build → PHP 8.4 FPM production image
- [ ] `docker-compose.prod.yml` with services: app (Laravel + FPM), nginx (or Caddy), postgres, redis, horizon, scheduler
- [ ] `.dockerignore` excludes dev artifacts (node_modules, .git, tests, vendor)
- [ ] Entrypoint script: runs migrations, caches config/routes/views, optionally seeds
- [ ] `APP_ENV=production`, `APP_DEBUG=false` in production compose
- [ ] Horizon runs as a separate service (not in the app container)
- [ ] Scheduler runs `php artisan schedule:work` as a separate service
- [ ] Health check on app service
- [ ] Frontend assets pre-built and served from `public/build/`
- [ ] Total image size < 250MB (no node_modules or dev dependencies in final stage)
- [ ] `docker compose -f docker-compose.prod.yml up -d` starts everything
- [ ] All tests pass in dev (no breakage to existing setup)

### Business Rules
- Dev setup (Sail) must remain untouched — production compose is a separate file
- Postgres data persisted via named volume
- Redis ephemeral (cache/queue — acceptable to lose on restart)
- Queue connection must be redis in production
- Session driver must be redis in production (not file)

### Out of Scope
- Caddy reverse proxy config (05.02)
- SSL/TLS certificates
- CI/CD pipeline
- Kubernetes / cloud orchestration
