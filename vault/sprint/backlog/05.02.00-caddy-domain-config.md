# Task 05.02.00: Caddy Reverse Proxy + Domain Config

## Business Requirements

### Problem
The production host (192.168.254.140) already runs Caddy as a reverse proxy. The app needs a Caddyfile entry to route `sts-demo.betamaxgroup.tech` to the app container.

### User Story
As a deployer, I want the app accessible at `https://sts-demo.betamaxgroup.tech` via the existing Caddy proxy on the host.

### Acceptance Criteria
- [ ] `Caddyfile.prod` or deployment docs with Caddy config for `sts-demo.betamaxgroup.tech`
- [ ] Reverse proxy to app container's nginx/FPM port
- [ ] Automatic HTTPS via Caddy's built-in ACME (if DNS points to host)
- [ ] WebSocket/SSE passthrough for `/api/issues/{id}/stream` (no buffering)
- [ ] Trusted proxies configured in Laravel (`TrustProxies` middleware) for correct `X-Forwarded-*` headers
- [ ] `.env.production.example` with all required production env vars documented
- [ ] Deployment README section explaining how to deploy to the host
- [ ] All tests pass (no breakage)

### Business Rules
- Caddy runs on the HOST, not in Docker — the Caddyfile is a host-level config file
- App container exposes port only on localhost (not 0.0.0.0) — Caddy proxies to it
- SSE endpoint must NOT be buffered by proxy (needs `flush_interval -1` or similar)

### Out of Scope
- DNS configuration (assumed already pointing)
- SSL certificate management (Caddy handles this automatically)
- CI/CD
