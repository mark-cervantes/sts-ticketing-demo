# Task 06.01: Fix Horizon/Scheduler Entrypoint to Respect CMD Override

## Business Requirements

### Problem
The Docker entrypoint script (`docker/entrypoint.sh`) always ends with `exec php-fpm`, ignoring the `command:` override in `docker-compose.prod.yml`. This means the Horizon container (which should run `php artisan horizon`) and the scheduler container (which should run `php artisan schedule:work`) both run `php-fpm` instead of their intended commands. Result: **queue jobs never process** — AI summaries stay stuck at "Generating summary…" forever.

### Root Cause
`docker/entrypoint.sh` line 45: `exec php-fpm` is hardcoded. Docker's entrypoint/cmd contract requires the entrypoint to `exec "$@"` to forward the CMD from docker-compose. The current script ignores `$@` entirely.

### Acceptance Criteria
- [ ] Entrypoint script uses `exec "$@"` when arguments are passed (CMD override), falls back to `exec php-fpm` when no arguments given (default app container)
- [ ] `docker compose exec horizon ps aux` shows `php artisan horizon` running, NOT `php-fpm`
- [ ] `docker compose exec scheduler ps aux` shows `php artisan schedule:work` running, NOT `php-fpm`
- [ ] `php artisan horizon:status` returns "running" (not "inactive")
- [ ] Existing app container still runs `php-fpm` (no CMD override → default)
- [ ] 283 PHPUnit tests still pass

### Business Rules
- Horizon must process queued jobs (GenerateSummaryJob) within seconds, not stuck forever
- Scheduler must run `schedule:work` for recurring tasks (needs_attention checks, etc.)
- The app container must continue running php-fpm as before

### Out of Scope
- AI summary generation logic changes (separate task)
- LLM driver configuration
