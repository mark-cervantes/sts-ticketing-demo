# Task 09.01: AI Chat — Backend (Model, API, Streaming Endpoint)

## Business Requirements

### Problem
Users can see AI-generated summaries on issues, but they can't ask the AI follow-up questions. When a summary says "Carol confirmed the gateway timeout," there's no way to ask "What specific timeout value?" or "Which gateway endpoint?" without reading the entire comment thread. Users want a conversational interface where they can discuss the issue with AI in context.

### User Story
As a user viewing an issue, I want to chat with AI about the issue so that I can get quick answers, drill into details, and explore implications without re-reading the full thread.

### Architecture

#### Data Model
```
issue_chat_messages
├── id (bigint, PK)
├── issue_id (FK → issues.id, ON DELETE CASCADE)
├── user_id (FK → users.id, nullable — null = AI response)
├── role (enum: 'user' | 'assistant')
├── content (text)
├── created_at
└── updated_at
```

#### API Endpoints
| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/issues/{issue}/chat` | Load chat history (paginated, oldest first) |
| `POST` | `/api/issues/{issue}/chat` | Send user message → get streamed AI response |

#### Streaming Strategy
The `POST` endpoint returns a streamed response (SSE or chunked text). The AI response tokens are streamed as they arrive from the LLM, giving users real-time feedback. The same `{base_url}/chat/completions` endpoint used by `LlmDriver` and `TriageService` is used here, but with `stream: true`.

#### Context Window
The AI receives:
1. **System prompt** — configurable from DB (like summary prompts), stored as `chat_system_prompt` on `ai_settings`
2. **Issue context** — title, description, category, priority, status
3. **Recent comments** — the last N comments with author names and timestamps
4. **Chat history** — previous messages in this conversation (up to context limit)

### Acceptance Criteria
- [ ] Migration creates `issue_chat_messages` table
- [ ] `IssueChatMessage` model with `issue()` and `user()` relations
- [ ] `GET /api/issues/{issue}/chat` returns paginated messages (oldest first, 50 per page)
- [ ] `POST /api/issues/{issue}/chat` accepts `{ message: string }`, creates user message, streams AI response
- [ ] AI response is saved to DB after streaming completes
- [ ] Streaming uses `text/event-stream` content type with `data: {token}` format
- [ ] Final SSE event: `data: [DONE]` (matches OpenAI convention)
- [ ] If LLM is unavailable, returns a non-streaming JSON error (don't leave the stream hanging)
- [ ] Chat messages respect issue authorization — only users who can view the issue can chat
- [ ] `ai_settings` gets a `chat_system_prompt` column (nullable, falls back to default)
- [ ] Default system prompt instructs the AI: "You are a helpful assistant discussing a support ticket. Answer based on the issue details and conversation provided. Be concise and specific."
- [ ] PHPUnit tests cover: message creation, history retrieval, authorization, streaming format
- [ ] 380+ tests pass

### Business Rules
- Chat is per-issue — each issue has its own conversation thread
- Only users who can view the issue (owner, shared-with, or admin) can chat
- AI responses are attributed to `user_id: null, role: 'assistant'`
- Chat history is append-only (no edit/delete for v1)
- Rate limit: max 10 messages per user per issue per hour (prevent abuse)
- If no AI provider is configured, the POST endpoint returns 503 with a clear message

### Out of Scope
- Frontend UI (task 09.02)
- Multi-turn function calling / tool use
- Image/file attachments in chat
- Chat message editing or deletion
