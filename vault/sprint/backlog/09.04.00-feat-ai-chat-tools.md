# Task 09.04: AI Chat Tools — Extensible Tool Architecture + Discovery UX

## Business Requirements

### Problem
Users chatting with AI about an issue often want to take action — "create a ticket for the retry queue." Currently the chat is text-only. There's also no way for users to discover that the AI can do more than answer questions.

### User Story
As a user chatting with AI, I want the AI to create tickets based on our conversation, and I want to naturally discover this capability without reading documentation.

## Part A: Backend — Extensible Tool System

### Architecture: Tool Registry Pattern

```
app/Services/Ai/Tools/
├── ChatToolInterface.php      ← contract every tool implements
├── ChatToolResult.php         ← value object for tool execution results
├── ChatToolRegistry.php       ← discovers + registers all tools, provides schema to LLM
└── CreateTicketTool.php       ← first tool
```

**`ChatToolInterface`:**
```php
interface ChatToolInterface
{
    public function name(): string;                    // e.g., 'create_ticket'
    public function description(): string;             // for LLM system prompt
    public function parameterSchema(): array;          // OpenAI function calling JSON schema
    public function execute(array $parameters, Issue $issue, User $user): ChatToolResult;
    public function requiresConfirmation(): bool;      // true = render card, wait for user click
    public function suggestionChip(): ?string;         // e.g., "Create a follow-up ticket" — null if no chip
}
```

The `suggestionChip()` method lets each tool register its own discovery prompt for the frontend empty state.

**`ChatToolResult`** value object:
```php
class ChatToolResult
{
    public function __construct(
        public readonly string $toolName,
        public readonly bool $success,
        public readonly string $message,
        public readonly ?array $data = null,        // e.g., { id: 42, title: "...", url: "/issues/42" }
        public readonly bool $pendingConfirmation = false,
    ) {}
}
```

**`ChatToolRegistry`:**
- Registered via service provider (explicit, not auto-discovery — predictable ordering)
- `getToolDefinitions(): array` — OpenAI function calling format for LLM request
- `getSuggestionChips(): array` — collects non-null chips from all registered tools
- `executeTool(string $name, array $params, Issue $issue, User $user): ChatToolResult`

**`CreateTicketTool`:**
- Name: `create_ticket`
- Description: "Create a new support ticket based on the conversation context."
- Parameters: `{ title: string, description: string, priority: enum, category: string }`
- Requires confirmation: YES
- Suggestion chip: "Create a follow-up ticket"
- Execute: Creates `Issue` via existing pipeline, returns ID + title + URL

### ChatService Updates
- Include `tools` param in `/chat/completions` request when tools are registered
- When LLM responds with `tool_calls`: parse, execute via registry, feed result back as `tool` role message
- **Fallback for non-function-calling models:** Include tool descriptions in system prompt, instruct LLM to output `{"tool_call": {"name": "...", "arguments": {...}}}` JSON block, parse response for this pattern
- Stream tool call events as distinct SSE: `data: {"type":"tool_call","tool":"create_ticket","arguments":{...},"requires_confirmation":true}\n\n`

### New Endpoint
- `POST /api/issues/{issue}/chat/tool-confirm` — accepts `{ tool: string, arguments: object }`, executes via registry, returns `ChatToolResult`

### API: Suggestion Chips
- `GET /api/chat/suggestions` — returns `ChatToolRegistry::getSuggestionChips()` as JSON array. Frontend calls once on load.

## Part B: Frontend — Tool Discovery UX (4 Layers)

### Layer 1: Empty State Suggestion Chips (highest impact)
When the chat panel has no messages, show clickable suggestion chips instead of plain placeholder text:

```
┌─────────────────────────────────────┐
│  💡 Try asking:                     │
│  [Summarize the discussion]         │
│  [What's the root cause?]           │
│  [Create a follow-up ticket]  📋    │
│  [What should we do next?]          │
├─────────────────────────────────────┤
│  [Ask about this issue...    ] [⏎]  │
└─────────────────────────────────────┘
```

- **Static chips** (hardcoded): "Summarize the discussion", "What's the root cause?", "What should we do next?"
- **Tool chips** (from API): fetched from `GET /api/chat/suggestions`, each gets a subtle icon
- Clicking a chip = sends that text as the first message
- Chips disappear once the first message is sent

### Layer 2: Contextual Nudge After AI Insight
When AI response contains action-oriented language (keywords: "recommend", "suggest", "should", "need to", "follow-up"), show a subtle hint below the message:

```
💡 You can ask me to create a ticket for this.
```

- Rendered as a muted, italic line below the AI message — NOT a button
- Only shows on the first qualifying AI message per session (don't spam)
- Tracked via a `hasShownNudge` ref in the composable

### Layer 3: Input Placeholder Rotation
The chat input placeholder rotates between hints every 8 seconds:
- "Ask about this issue..."
- "Try: create a ticket for this"
- "Ask anything about this issue..."

Subtle — teaches through repetition. Stops rotating once the user starts typing.

### Layer 4: First-Time Tooltip (One-Time Education)
On the user's first-ever chat panel open (tracked via `localStorage` key `ai-chat-onboarded`):
- Show a small popover/tooltip above the input:
  > "💡 You can ask the AI to create tickets and more. Just describe what you need."
- Auto-dismisses after 8 seconds or on click
- Never shows again

### Tool Call Rendering in Chat
When AI responds with a tool call (SSE type `tool_call`):

**For confirmation-required tools (like CreateTicketTool):**
```
┌──────────────────────────────────────┐
│ 📋 Create Ticket                     │
│                                      │
│ Title: Add retry queue for gateway   │
│ Priority: High · Category: Backend   │
│                                      │
│ Description:                         │
│ Implement a local retry queue to     │
│ buffer peak-hour gateway timeouts... │
│                                      │
│        [Create]  [Edit & Create]     │
└──────────────────────────────────────┘
```

- **"Create"** → POST to `/api/issues/{issue}/chat/tool-confirm`, show success: "✓ Created — #42" with link
- **"Edit & Create"** → opens `CreateIssueDialog` with `prefill` prop (already exists)
- Card updates in-place after creation (no re-render flicker)

### Component Changes
- `IssueChatPanel.vue` — add suggestion chips in empty state, first-time tooltip, placeholder rotation
- `ChatMessage.vue` — add tool call card rendering, contextual nudge detection
- New: `SuggestionChips.vue` — clickable chip grid component
- New: `ToolCallCard.vue` — generic tool result card (CreateTicket is first, but the component should accept tool name + arguments generically and render per-tool templates)
- `useIssueChat.ts` — add tool call SSE parsing, suggestion chip loading, nudge tracking, tool confirm API call

### Business Rules
- Tools respect user permissions (can't create ticket without permission)
- Confirmation-required tools NEVER auto-execute
- Adding a new tool = new class implementing interface + register in provider + optional frontend template. No service/controller/core-component changes needed.
- Suggestion chips are extensible — each tool provides its own chip text
- Future tools to support (architecture only, don't implement):
  - `update_priority`, `assign_issue`, `search_issues`, `summarize_thread`, `add_comment`

### Out of Scope
- Implementing any tool beyond CreateTicketTool
- Multi-step tool chains
- Tool usage analytics
