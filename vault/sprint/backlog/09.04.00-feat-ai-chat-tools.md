# Task 09.04: AI Chat Tools — Extensible Tool Architecture

## Business Requirements

### Problem
Users chatting with AI about an issue often want to take action based on the conversation — "create a ticket for the retry queue," "change the priority to critical," "assign this to Carol." Currently the chat is read-only: the AI can only respond with text. There's no way for the AI to trigger actions in the system.

### User Story
As a user chatting with AI about an issue, I want the AI to be able to suggest and execute actions (like creating a follow-up ticket) so that I can act on insights without leaving the conversation.

### Architecture: Extensible Tool System

#### Design Principle
The ticket generation capability is the **first tool** in a system designed for many. The architecture must make adding new tools trivial — each tool is a self-contained class that registers itself, declares its schema, and handles execution independently.

#### Backend: Tool Registry Pattern

```
app/Services/Ai/Tools/
├── ChatToolInterface.php      ← contract every tool implements
├── ChatToolRegistry.php       ← discovers + registers all tools, provides schema to LLM
├── CreateTicketTool.php       ← first tool: generates a ticket from chat context
├── (future) UpdatePriorityTool.php
├── (future) AssignIssueTool.php
├── (future) SearchIssuesTool.php
└── (future) SummarizeThreadTool.php
```

**`ChatToolInterface`:**
```php
interface ChatToolInterface
{
    /** Tool name as the LLM knows it (e.g., 'create_ticket') */
    public function name(): string;

    /** Human-readable description for the LLM system prompt */
    public function description(): string;

    /** JSON Schema for the tool's parameters (OpenAI function calling format) */
    public function parameterSchema(): array;

    /** Execute the tool, return a result to feed back to the LLM */
    public function execute(array $parameters, Issue $issue, User $user): ChatToolResult;

    /** Whether this tool requires user confirmation before executing */
    public function requiresConfirmation(): bool;
}
```

**`ChatToolResult`** value object:
```php
class ChatToolResult
{
    public function __construct(
        public readonly string $toolName,
        public readonly bool $success,
        public readonly string $message,         // human-readable result
        public readonly ?array $data = null,     // structured data (e.g., created ticket ID/title)
        public readonly bool $pendingConfirmation = false,  // needs user to approve
    ) {}
}
```

**`ChatToolRegistry`:**
- Auto-discovers all classes implementing `ChatToolInterface` in the `Tools/` directory
- Or uses explicit registration in a service provider
- Provides `getToolDefinitions(): array` — returns the OpenAI-format function definitions array for the system prompt
- Provides `executeTool(string $name, array $params, Issue $issue, User $user): ChatToolResult`

#### LLM Integration: Function Calling
The `ChatService` already builds messages for the LLM. When tools are available:
1. Include `tools` parameter in the `/chat/completions` request (OpenAI function calling format)
2. When the LLM responds with a `tool_calls` choice, execute via `ChatToolRegistry`
3. Feed the tool result back as a `tool` role message
4. Let the LLM generate the final user-facing response incorporating the tool result

**Fallback for non-function-calling models:** If the LLM doesn't support function calling (e.g., some Ollama models), fall back to prompt-based tool invocation:
- Include tool descriptions in the system prompt
- Instruct the LLM to output a JSON block `{"tool_call": {"name": "...", "arguments": {...}}}` when it wants to use a tool
- Parse the response for this pattern, execute the tool, feed result back

#### First Tool: `CreateTicketTool`

**Name:** `create_ticket`
**Description:** "Create a new support ticket based on the conversation. Use when the user asks to create a ticket, report a bug, or log a follow-up."
**Parameters:**
```json
{
    "title": { "type": "string", "description": "Brief ticket title" },
    "description": { "type": "string", "description": "Detailed description incorporating conversation context" },
    "priority": { "type": "string", "enum": ["low", "medium", "high", "critical"], "description": "Suggested priority" },
    "category": { "type": "string", "description": "Category name (must match existing categories)" }
}
```
**Requires confirmation:** YES — the AI suggests the ticket, the user confirms before it's actually created.
**Execution:** Creates an `Issue` via the existing issue creation pipeline (respects validation, authorization, triggers summary generation).

#### Frontend: Tool Call Rendering

When the AI response includes a tool call:
1. **Pending confirmation tools** → render an interactive card in the chat:
   ```
   AI: Based on our discussion, here's a ticket I'd suggest:
   ┌──────────────────────────────────────┐
   │ 📋 Create Ticket                     │
   │ Title: Add retry queue for gateway   │
   │ Priority: High · Category: Backend   │
   │ Description: Implement a local...    │
   │                                      │
   │          [Create] [Edit & Create]    │
   └──────────────────────────────────────┘
   ```
2. **"Create"** → POST to existing issue creation endpoint with the AI-suggested data
3. **"Edit & Create"** → opens `CreateIssueDialog` pre-filled (reuse existing `prefill` prop)
4. **Auto-executed tools** (future, no confirmation) → render result inline: "✓ Priority updated to Critical"

### Business Rules
- Tool execution respects the user's permissions (can't create a ticket if you don't have permission)
- Tool calls are logged in the conversation (visible to other users in saved conversations)
- Confirmation-required tools NEVER auto-execute — always wait for user click
- The tool system is additive — adding a new tool should require only creating a new class implementing `ChatToolInterface` and registering it. No changes to ChatService, controller, or frontend rendering logic.
- Future tools to plan for (don't implement, just ensure the architecture supports):
  - `update_priority` — change issue priority
  - `assign_issue` — assign/reassign to a user
  - `search_issues` — search related tickets
  - `summarize_thread` — re-summarize with specific focus
  - `add_comment` — post a comment to the issue

### Out of Scope
- Implementing any tool beyond `CreateTicketTool`
- Multi-step tool chains (tool A's output feeds tool B)
- Tool permissions beyond issue-level auth
- Tool usage analytics
