### 2. Vite Config for Sail HMR

Breeze generates `vite.config.ts`. It may not include the Sail-specific HMR settings. The container listens on port `5175` (`VITE_PORT` in `.env`), and the compose service maps `5175:5175`.

**Required final state of `vite.config.ts` `server` block:**

```ts
server: {
    host: '0.0.0.0',               // bind inside container to all interfaces
    hmr: {
        host: 'localhost',          // browser reaches HMR via localhost (host machine)
    },
    watch: {
        usePolling: true,           // needed for Docker volume mounts
    },
},
```

- `server.host` **must** be `0.0.0.0` — without it, Vite binds to `127.0.0.1` inside the container and the port-forward from Docker never reaches it.
- `hmr.host` must be `localhost` — the browser on the host connects to HMR through the forwarded port.
- `usePolling: true` ensures file-change detection works over Docker bind mounts (Linux host → container can skip this, but it's harmless and safer to include).
- Port is already `5175` via `VITE_PORT` env — Breeze/Sail read it automatically. Do **not** hardcode `port: 5175` unless Breeze's config omits it.

**If Breeze's generated config already has `server.host: '0.0.0.0'` and `hmr.host: 'localhost'`, no edit needed.** If it doesn't, patch only the `server` block. Do not remove `watch.ignored` if Breeze sets one.

**Verify §2:**

### 8. Pitfalls & Warnings

1. **Vite port binding:** Container AND host both use `5175`. The `server.host` in `vite.config.ts` MUST be `'0.0.0.0'` for Sail containers. If missing, Vite binds to `127.0.0.1` inside the container → HMR unreachable from host browser.

2. **Boost MCP bootability:** Boost MCP invokes `php artisan boost:mcp` — if the app fails to boot (e.g., missing migration, broken config), MCP tools in this project will fail. Commit `wip:` after each section so the app stays bootable.

3. **Existing tests are sacred:** `tests/Feature/ExampleTest.php` and `tests/Unit/ExampleTest.php` must NOT be modified or deleted. Breeze adds new test files alongside them. If Breeze overwrites `ExampleTest`, restore it from git: `git checkout HEAD -- tests/Feature/ExampleTest.php tests/Unit/ExampleTest.php`.

4. **Breeze overwrites `resources/css/app.css`:** The original Tailwind v4 `@theme` block with `--font-sans` will be replaced. That's expected — shadcn-vue init and our §4 theming replace it entirely. Order is: Breeze → shadcn-vue init → our theming (§4).

5. **Breeze rewrites `vite.config.js` → `vite.config.ts`:** The original `vite.config.js` (with `bunny()` font plugin and `@tailwindcss/vite`) gets replaced. Breeze's generated `vite.config.ts` uses `laravel-vite-plugin` and `@vitejs/plugin-vue`. The `@tailwindcss/vite` plugin should still be present — verify after install.

6. **shadcn-vue `components.json` alias:** The `--template laravel` flag should set `aliases.components` → `@/components` and `aliases.utils` → `@/lib/utils`. Verify this matches the `@` alias in `vite.config.ts` → `resolve.alias` (Breeze sets `@` → `./resources/js`). Mismatch = import errors at build time.

7. **Horizon REQUIRES Redis driver:** `QUEUE_CONNECTION=database` (current `.env`) makes Horizon non-functional. The `.env` must be updated to `QUEUE_CONNECTION=redis` BEFORE testing Horizon. The Redis container is already in the Sail stack.

8. **Sail user/group:** `WWWUSER=1000`, `WWWGROUP=1000` in `.env`. File ownership should be fine since the host UID typically matches. If `npm install` or `artisan` commands produce permission errors, run `./vendor/bin/sail root-shell` and `chown -R sail:sail /var/www/html/node_modules`.

9. **HMR verification is a one-shot check:** The coder verifies HMR by running `sail npm run dev` in a separate terminal, confirming the Vite client is accessible via curl, then stopping it. Do NOT leave a long-running `npm run dev` process from within the agent.

10. **Pint after PHP changes:** After Breeze and Horizon install, run `./vendor/bin/sail pint --dirty --format agent` to fix formatting on any modified PHP files per Boost guidelines.
