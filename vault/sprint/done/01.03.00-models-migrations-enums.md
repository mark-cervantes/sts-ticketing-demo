
### 2. Enum Design (app/Enums/)

All four enums: backed string enums per PHP 8.4, explicit return types, PHPDoc blocks.

**`Status`** ‚Äî cases: `Open = 'open'`, `InProgress = 'in_progress'`, `Resolved = 'resolved'`. No methods needed.

**`Priority`** ‚Äî cases: `Low = 'low'`, `Medium = 'medium'`, `High = 'high'`, `Critical = 'critical'`.
- `needsAttention(): bool` ‚Äî returns `true` for `High` and `Critical` only (the priority component of BR-03; deadline component lives on Issue model per ADR-005).
- Implementation: `return $this === self::High || $this === self::Critical;`

**`Visibility`** ‚Äî cases: `Private = 'private'`, `Public = 'public'`. No methods needed.

**`Permission`** ‚Äî cases: `View = 'view'`, `Comment = 'comment'`, `Edit = 'edit'`.
- Ladder: `view < comment < edit` (SPEC ¬ß3.2).
- `canComment(): bool` ‚Äî true for `Comment` and `Edit`: `return $this === self::Comment || $this === self::Edit;`
- `canEdit(): bool` ‚Äî true for `Edit` only: `return $this === self::Edit;`
- Explicit equality comparisons are fine here ‚Äî three cases don't warrant a numeric ordering system. The explicitness is self-documenting.

‚öÝÔ∏è **Discrepancy flag:** SPEC ¬ß6.5 says the share API only accepts `view` or `edit` (not `comment`). ADR-007 also only mentions `view | edit`. But SPEC ¬ß4.5 schema and ¬ß3.2 authorization define the full three-value ladder including `comment`. **Recommendation: implement all three in the enum and schema (SPEC ¬ß4.5 is the column definition ground truth), but note that the share API endpoint may restrict input to `view` and `edit` only via Form Request validation.** Resolve with PM before building the share endpoint (later task).

### 3. Model Design (app/Models/)

Generate with factories:
```bash
./vendor/bin/sail artisan make:model Issue --factory
./vendor/bin/sail artisan make:model Comment --factory
./vendor/bin/sail artisan make:model Category --factory
./vendor/bin/sail artisan make:model IssueShare --factory
```

**`Issue`** model:
- Traits: `HasFactory`, `SoftDeletes`
- Use `#[Fillable]` attribute (matches existing `User` model convention): `#[Fillable(['title', 'description', 'priority', 'category_id', 'status', 'visibility', 'summary', 'suggested_next_action', 'summary_status', 'needs_attention', 'deadline_at'])]` ‚Äî whitelist matching FR-02 input + server-set fields. Do NOT use `$guarded = []`.
- `casts()` method (Laravel 13 style, matching existing `User::casts()`):
  ```php
  protected function casts(): array
  {
      return [
          'priority' => Priority::class,
          'status' => Status::class,
          'visibility' => Visibility::class,
          'deadline_at' => 'immutable_datetime',
          'needs_attention' => 'boolean',
      ];
  }
  ```
  Note: `summary_status` left as string intentionally ‚Äî no enum class for it in this task (no methods needed; add `SummaryStatus` enum later if methods arise).
- Relationships: `user(): BelongsTo`, `category(): BelongsTo`, `comments(): HasMany`, `shares(): HasMany` (to `IssueShare`).
- **`computeNeedsAttention()` static helper** per ADR-005 + BR-03:
  ```php
  public static function computeNeedsAttention(Priority $priority, ?CarbonImmutable $deadlineAt): bool
  {
      if ($priority->needsAttention()) {
          return true;
      }
      if ($deadlineAt !== null) {
          $threshold = config('issues.attention_threshold_minutes', 60);
          return $deadlineAt->lte(now()->addMinutes($threshold));
      }
      return false;
  }
  ```
- **Boot `saving` event** to auto-set `needs_attention`:
  ```php
  protected static function booted(): void
  {
      static::saving(function (Issue $issue): void {
          $issue->needs_attention = self::computeNeedsAttention(
              $issue->priority,
              $issue->deadline_at,
          );
      });
  }
  ```
  This fires on both create and update, satisfying BR-03's "recomputed on issue create/update" requirement.

**`Comment`** model:
- `#[Fillable(['body'])]` ‚Äî only user-supplied field; `issue_id` and `user_id` set in controller/service.
- Relationships: `issue(): BelongsTo`, `user(): BelongsTo`.
- No `$table->timestamps()` in migration, so set `public $timestamps = false;` and add `const CREATED_AT = 'created_at'; const UPDATED_AT = null;` ‚Äî OR simply set `const UPDATED_AT = null;` and keep `$timestamps = true` which will only write `created_at`. The cleaner approach: `$timestamps = true` with `const UPDATED_AT = null;`.

**`Category`** model:
- `#[Fillable(['name', 'slug'])]`
- Relationship: `issues(): HasMany`
- No `updated_at` ‚Äî same pattern as Comment: `$timestamps = true` with `const UPDATED_AT = null;`
- **Slug auto-generation** on `creating` event in `booted()`:
  ```php
  protected static function booted(): void
  {
      static::creating(function (Category $category): void {
          $category->slug = static::generateUniqueSlug($category->name);
      });
  }
  ```
- **Collision-handling algorithm** for `generateUniqueSlug(string $name): string`:
  1. Base slug = `Str::slug($name)` (e.g., "Bug Reports" ‚Üí "bug-reports")
  2. Query `categories` for `slug LIKE '{base}%'` to find existing matches
  3. If base doesn't exist ‚Üí use base
  4. If base exists ‚Üí find highest numeric suffix, increment: `bug-reports-2`, `bug-reports-3`, etc.
  5. Edge case: first collision is `-2` (not `-1`), matching conventional behavior

**`IssueShare`** model:
- `#[Fillable(['issue_id', 'user_id', 'permission'])]`
- Relationships: `issue(): BelongsTo`, `user(): BelongsTo`
- `casts()`: `'permission' => Permission::class`
- No `updated_at` ‚Äî same `const UPDATED_AT = null;` pattern
- The unique constraint `(issue_id, user_id)` is enforced at DB level (migration). Service layer will handle upsert logic (ADR-007) ‚Äî not needed in this task.

**Soft-delete cascade note:** Laravel `SoftDeletes` sets `deleted_at` without firing DB-level `ON DELETE CASCADE`. When an issue is soft-deleted, its comments and shares **remain intact** in the DB ‚Äî this is correct per ADR-007 (comments preserved for audit trail). Hard-delete via `forceDelete()` triggers the DB cascade and removes comments + shares. No additional cascade logic needed in models.

### 7. Pitfalls

- **Postgres native `enum` type is an ALTER-headache.** Postgres `CREATE TYPE ... AS ENUM` requires `ALTER TYPE ... ADD VALUE` to add new values later ‚Äî which can't run inside a transaction. Use `string` columns with PHP enum `$casts` instead. Laravel handles hydration/dehydration transparently. The `$table->enum()` blueprint method creates a `CHECK` constraint on some DBs but a native type on Postgres ‚Äî **avoid it entirely**; use `$table->string()`.
- **`SoftDeletes` + unique indexes:** If you ever need uniqueness on a soft-deleted column (e.g., unique title per user), you'd need a partial unique index (`WHERE deleted_at IS NULL`). Not needed now (no unique constraints on issue columns), but flag for future.
- **`category_id` must be `NOT NULL` + `onDelete('restrict')`** ‚Äî SPEC ¬ß4.2 says required, BR-05 says categories with issues can't be deleted. `foreignId()->constrained()` is non-nullable by default. The `restrict` enforces at DB level as a safety net beyond the app-layer guard.
- **`deadline_at` is nullable** ‚Äî tests must cover both branches (`null` ‚Üí only priority signal, non-null ‚Üí check threshold). Ensure factory has `deadline_at => null` as default with a state for `withDeadline`.
- **`casts()` method form** ‚Äî Laravel 13 uses the `casts()` method (not `$casts` property). Existing `User` model already uses this pattern. Follow it consistently.
- **Composite index column order** ‚Äî `[status, priority]` is correct: status has ~3 values (lower cardinality but it's the leading filter in "show all open, then sort by priority"). This matches the dominant query pattern per FR-03.
- **`IssueShare` unique constraint must be a DB-level unique index** (`$table->unique(['issue_id', 'user_id'])`), not just model-level validation. The upsert for re-sharing (ADR-007) relies on this constraint for `updateOrCreate` correctness.
- **`comments` and `issue_shares` have no `updated_at`** per SPEC ¬ß4.4 and ¬ß4.5 ‚Äî set `const UPDATED_AT = null;` on those models. Using `$table->timestamps()` will create both columns; use `$table->timestamp('created_at')->nullable()` instead in migrations.
- **Factories are required** for all 4 models ‚Äî `IssueFactory`, `CommentFactory`, `CategoryFactory`, `IssueShareFactory`. The `--factory` flag on `make:model` generates the stub. Populate with realistic defaults matching SPEC ¬ß4 columns. `IssueFactory` should default to `Priority::Low`, `Status::Open`, `Visibility::Private`, `needs_attention => false`, `deadline_at => null`, with states like `highPriority()`, `withDeadline()`, `public()`.

### Deviation 1: `categories.name` NOT unique at DB level

**Ruling: (b) ‚Äî Accept with mandatory follow-up task to rework the test**

**Evidence:**
- SPEC ¬ß4.3 line 96: `name: Required, unique` ‚Äî explicit.
- ADR-006 line 15: `name string, unique` ‚Äî reaffirms SPEC.
- Coder dropped DB unique to accommodate QA's `test_slug_collision_handled` which creates identically-named categories.

**Reasoning:** The SPEC and ADR are unambiguous ‚Äî `name` must be unique. The QA test is testing the right behavior (slug collision handling) but using the wrong setup. Identical names that produce the same slug is a valid collision scenario, but the test should achieve this through *near-identical* names that normalize to the same slug via `Str::slug()`, not by violating the name uniqueness contract.

Example: `"Bug Reports"` and `"Bug  Reports"` (double space) or `"Bug-Reports"` all produce slug `"bug-reports"` through `Str::slug()` normalization, triggering the collision path without requiring duplicate names.

**Required follow-up:** Backlog task to:
1. Add `->unique()` to `categories.name` in migration
2. Rewrite `test_slug_collision_handled` to use near-identical names (e.g. `"Bug Reports"` + `"Bug  Reports"`) that collide through `Str::slug()` normalization
3. Add a separate test verifying that duplicate exact names are rejected at DB level

This does NOT block merge ‚Äî the current state is safe (slug uniqueness still enforced, app-layer validation will come with the Category API endpoint). The DB unique on `name` is a defense-in-depth addition.
