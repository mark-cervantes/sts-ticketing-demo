## What To Build

Create model factories and database seeders that produce realistic demo data.

### Factories (database/factories/)
- **UserFactory** — realistic names/emails via Faker
- **CategoryFactory** — realistic category names
- **IssueFactory** — configurable states for each status, priority, visibility. Respects enum values. Can generate with/without deadline.
- **CommentFactory** — realistic body text
- **IssueShareFactory** — configurable permission level

### Seeders (database/seeders/)
- **CategorySeeder** — creates default categories: billing, technical, account, general, bug, feature-request
- **UserSeeder** — 5 users with distinct realistic names/emails
- **IssueSeeder** — 15-20 issues spanning:
  - All priority levels
  - All status values
  - Mix of private/public visibility
  - Some with deadlines, some without
  - At least 2 with summary_status=ready (pre-filled summary)
  - At least 3 with needs_attention=true
- **CommentSeeder** — 2-4 comments per issue (30+ total)
- **IssueShareSeeder** — 2-3 issues shared between users
- **DatabaseSeeder** — orchestrates all above in correct order

### Requirements per SRS §6
- Minimum: 3+ users, 5+ issues, comments across issues
- At least 1 issue with summary_status=ready
- At least 1 with needs_attention=true
- All seeded categories populated

## Technical Guidance

### 1. IssueFactory States — Extend, Don't Rewrite

The existing `IssueFactory` (line 27-43) already defines `definition()` with safe defaults and has `highPriority()`, `critical()`, `withDeadline()`, `public()` states. ADD the following states — do not touch `definition()`:

- `->status(Status $status)` — generic setter: `$this->state(fn () => ['status' => $status])`
- `->open()`, `->inProgress()`, `->resolved()` — semantic shortcuts calling `->status()`
- `->priority(Priority $priority)` — generic setter (complements existing `highPriority()`/`critical()`)
- `->private()` — sets `Visibility::Private` (counterpart to existing `->public()`)
- `->withDeadline(?Carbon $deadline = null)` — overload the existing method signature to accept a `Carbon` instance OR default to `now()->addMinutes(fake()->numberBetween(15, 180))`
- `->summaryReady(string $summary, string $actionItem)` — sets `summary_status => SummaryStatus::Ready`, `summary => $summary`, `suggested_next_action => $actionItem`; both args MUST be non-null
- `->summaryProcessing()` — sets `summary_status => SummaryStatus::Processing`
- `->summaryFailed()` — sets `summary_status => SummaryStatus::Failed`
- `->needsAttention()` — convenience: `->critical()->withDeadline(now()->addMinutes(15))` (both triggers hit)

Design rule: semantic states (`->critical()`, `->summaryReady()`, `->needsAttention()`) are reusable in future feature tests — prefer them over raw field arrays.

### 2. Realistic Summary Text Bank

Define a private array of 4-5 summary/action pairs in `IssueFactory` for `summaryReady()` to draw from. Example entries:

```php
private static array $summaryBank = [
    ['summary' => 'The user reports intermittent 502 errors when accessing the billing portal. Logs indicate upstream timeout from the payment gateway after 30s. This correlates with peak-hour traffic spikes observed in the last 7 days.', 'action' => 'Increase payment gateway timeout to 60s and add retry logic with exponential backoff.'],
    ['summary' => 'Account login fails for users with special characters in passwords. Root cause traced to overly strict input sanitization on the authentication endpoint that strips non-ASCII characters before hash comparison.', 'action' => 'Update the authentication middleware to use raw password bytes for hashing without pre-sanitization.'],
    ['summary' => 'Feature request for bulk export of issue history as CSV. Multiple enterprise customers have raised this in the last quarter. Current workaround involves manual copy-paste from the issue detail view.', 'action' => 'Implement a CSV export endpoint at GET /issues/export with date-range and status filters.'],
    ['summary' => 'The dashboard performance degrades noticeably when a user has more than 200 assigned issues. Profiling shows N+1 queries on the comments relationship during the summary card render.', 'action' => 'Add eager loading for comments and category relationships in the dashboard query.'],
];
```

When `summaryReady()` is called without arguments, it can pick a random pair from this bank. When called with explicit args, use those. This keeps demo data readable and non-gibberish.

### 3. needs_attention — Let the Model Compute It

`Issue::booted()` (app/Models/Issue.php:63-71) fires a `saving` event that calls `computeNeedsAttention()` based on `Priority` and `deadline_at`. Factories MUST NOT manually set `needs_attention` in state methods — the model event handles it. The existing `definition()` sets `'needs_attention' => false` but this gets overwritten by the saving event on create. The factory states just need to set `priority` and/or `deadline_at` to the right values; the saving event does the rest.

**CRITICAL PITFALL**: `DatabaseSeeder` currently uses `WithoutModelEvents` trait (line 11). This SUPPRESSES the `saving` event, meaning `needs_attention` will never be computed AND `Category::creating` won't fire (no slug auto-generation). **Remove `use WithoutModelEvents;`** from `DatabaseSeeder` — both `Issue` and `Category` depend on model events.

### 4. Seeder Order and Design

Generate seeders: `sail artisan make:seeder CategorySeeder`, etc. `DatabaseSeeder::run()` calls in this order:

1. **CategorySeeder** — the 6 names from SPEC §4.3 (line 100): `billing`, `technical`, `account`, `general`, `bug`, `feature-request`. Use `Category::firstOrCreate(['name' => $name])` for idempotent re-runs. Slug auto-generates via the `creating` event — do NOT set it manually.
2. **UserSeeder** — 5 users via `User::factory()->create()`. First user: `['name' => 'Demo User', 'email' => 'demo@example.com']` with the default `'password'` password (already in `UserFactory::$password`). Developer-experience choice: predictable login credentials for demo/dev.
3. **IssueSeeder** — orchestrate `IssueFactory` states to produce 15-20 issues distributed across the 5 users and 6 categories. Minimum coverage: every `Status` case, every `Priority` case, ≥3 with `needs_attention=true` (use `->critical()` or `->highPriority()` or tight deadline), ≥2 with `->summaryReady(...)`, mix of `->public()`/`->private()`, mix of with/without deadline.
4. **CommentSeeder** — for each seeded issue, create 2-4 comments via `Comment::factory()`. Author (`user_id`) should differ from the issue's `user_id` — pick a random user from the seeded set excluding the owner. Use `fake()->paragraph(2)` for body (1-3 sentences).
5. **IssueShareSeeder** — create 2-3 shares. Vary `Permission` values (`View`, `Comment`, `Edit`). Respect the `unique(issue_id, user_id)` constraint (migration line 25) — use `IssueShare::firstOrCreate(['issue_id' => ..., 'user_id' => ...], ['permission' => ...])` or track used pairs. The shared user must NOT be the issue owner.

**Transaction**: Wrap `DatabaseSeeder::run()` body in `DB::transaction(fn () => ...)` for atomicity on re-seed. Remove the existing `WithoutModelEvents` trait (see §3 above).

### 5. Faker Locale and Determinism

Use `fake('en_US')` — matches `APP_FAKER_LOCALE` in `.env`. Do NOT pin a seed value in `DatabaseSeeder`; randomness is the point for demo data. Devs who want reproducibility can use `fake()->seed($n)` manually or `--seed` on the CLI.

### 6. IssueShareFactory States

Add a `->permission(Permission $permission)` state to `IssueShareFactory` (currently bare). Also add `->view()`, `->comment()`, `->edit()` convenience methods for seeder readability.

### 7. Tests — SeederTest

Create `tests/Feature/SeederTest.php` via `sail artisan make:test --phpunit SeederTest`. Uses `RefreshDatabase` trait so seeded data is rolled back after each test — no pollution of other test classes. Tests:

| Test method | Assertion |
|---|---|
| `test_database_seeder_runs_cleanly` | `Artisan::call('migrate:fresh', ['--seed' => true])` returns 0 |
| `test_seeded_data_meets_minimum_counts` | `User::count() >= 5`, `Category::count() === 6`, `Issue::count() >= 15`, `Comment::count() >= 30`, `IssueShare::count() >= 2` |
| `test_seeded_data_includes_summary_ready_issue` | `Issue::where('summary_status', SummaryStatus::Ready)->exists()` is true AND that issue's `summary` is not null |
| `test_seeded_data_includes_needs_attention_issues` | `Issue::where('needs_attention', true)->count() >= 3` |
| `test_seeded_categories_match_spec` | All 6 names (`billing`, `technical`, `account`, `general`, `bug`, `feature-request`) exist in `categories` table |

Note: `RefreshDatabase` re-runs migrations between tests — these tests call `Artisan::call('db:seed')` to seed within the transaction, then assert. Import `SummaryStatus` enum for the assertion.

### 9. Pitfalls

- **`WithoutModelEvents` on `DatabaseSeeder`** — MUST be removed; `Issue::saving` and `Category::creating` both depend on model events (see §3)
- **`IssueShareSeeder` unique constraint** — migration enforces `unique(issue_id, user_id)` (migration line 25); seeder must avoid duplicate pairs or use `firstOrCreate`
- **Comment author = issue owner** — allowed by schema but looks wrong in demo; seeder should pick a different user as comment author
- **`summaryReady()` nullability** — `summary` and `suggested_next_action` must both be non-null when `summary_status = ready`; the factory state must enforce this via required parameters or the summary bank fallback
- **Existing factory `definition()` methods** — do NOT modify; only ADD `state()` methods to existing factory files
- **Generator usage** — use `sail artisan make:seeder {Name}Seeder` for each new seeder file; do not hand-write boilerplate
