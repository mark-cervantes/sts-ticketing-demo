## What To Build

Full Issue CRUD with validation, business rules, and soft delete.

### Endpoints
| Method | Route           | Action                    |
| ------ | --------------- | ------------------------- |
| POST   | /issues         | Create issue              |
| GET    | /issues         | List (paginated, filtered)|
| GET    | /issues/{id}    | View with comments        |
| PATCH  | /issues/{id}    | Update                    |
| DELETE | /issues/{id}    | Soft delete               |

### Service Layer
**IssueService** (app/Services/IssueService.php):
- `create(User $user, array $data): Issue` — set defaults, compute needs_attention, dispatch job
- `update(Issue $issue, array $data): Issue` — validate transitions, recompute needs_attention, re-trigger summary if description changed
- `delete(Issue $issue): void` — soft delete

### Form Requests
- **StoreIssueRequest**: title (required, max:255, trimmed), description (required, trimmed), priority (required, valid enum), category_id (required, exists:categories), visibility (optional, valid enum), deadline_at (optional, after:now)
- **UpdateIssueRequest**: same fields but all optional. Includes `updated_at` for optimistic locking check.

### Controller
- Thin controller delegates to IssueService
- Applies IssuePolicy on every action
- List uses `scopeAccessibleBy($user)` + filters + pagination

### List Filtering (SPEC §5.6)
- `?status=open` — filter by status enum value
- `?priority=high` — filter by priority enum value  
- `?category=billing` — filter by category **slug** (resolve to category_id)
- All combinable. Invalid values → ignored (no 422).
- Pagination: 15 per page, `?page=2`
- Sort: `?sort=created_at&direction=desc` (default: needs_attention desc, priority desc, created_at desc)

### Business Rules
- Create: status=open, visibility=private, summary_status=pending (defaults)
- Create: compute needs_attention immediately
- Create: dispatch GenerateSummaryJob
- Update description: reset summary_status=pending, dispatch new GenerateSummaryJob
- Update status only: do NOT re-trigger summary
- Update priority: recompute needs_attention
- Optimistic locking: client sends updated_at, mismatch → 409

### Response Shape
Issue response includes:
- All issue fields
- `category`: { id, name, slug } (not just category_id)
- `user`: { id, name } (owner)
- `comments_count` on list (not full comments)
- `comments` with `comments.user` on single view

## Technical Guidance

1. **API routing does not exist yet.** `bootstrap/app.php` has no `api:` parameter in `withRouting()` — add `api: __DIR__.'/../routes/api.php'` and create `routes/api.php` with `Route::apiResource('issues', IssueController::class)->middleware('auth:sanctum')` (or `auth` if staying session-based via Inertia — match Breeze's session guard, not Sanctum tokens; confirm with `config/auth.php` defaults).

2. **`app/Services/` directory does not exist.** This is the first service class in the project — create `app/Services/IssueService.php`. Keep `needs_attention` computation out of the service: it's already handled by `Issue::booted()` saving event (line 66–72 of `app/Models/Issue.php`), so just setting `priority` or `deadline_at` and calling `save()` triggers it automatically.

3. **`GenerateSummaryJob` does not exist yet** (built in task 02.04.00). Create a minimal stub at `app/Jobs/GenerateSummaryJob.php` whose `handle()` is empty — enough for `dispatch()` to succeed and `Queue::fake()` + `Queue::assertPushed()` to work in tests. Task 02.04.00 will fill in the real implementation.

4. **`app/Http/Resources/` directory does not exist.** Create `IssueResource` extending `JsonApiResource`. For list responses, include `comments_count` (use `withCount('comments')` on the query, not a nested load). For show, include `comments` with `comments.user`. Use conditional loading: `$this->whenLoaded('comments')` and `$this->whenCounted('comments')` — SRS §FR-03 line 84 / §FR-04 line 94.

5. **Category filter by slug, not ID.** The `?category=billing` filter must resolve slug → `category_id` via `Category::where('slug', $slug)->value('id')`. If the slug doesn't match, silently ignore the filter (SRS §FR-03 — invalid filter values → ignored, no 422). Apply the same silent-ignore rule for invalid `status` and `priority` enum values: use `tryFrom()` on the enum, skip the where clause if null.

6. **Optimistic locking is manual, not Laravel's built-in.** `UpdateIssueRequest` must accept `updated_at` (required on update). In `IssueService::update()`, compare `$issue->updated_at->equalTo(Carbon::parse($data['updated_at']))` — if false, return/throw a 409 response. The comparison must handle timezone normalization (DB stores UTC).

7. **Eager loading matrix for N+1 prevention (SRS §NFR-01).** List: `Issue::with(['category', 'user'])->withCount('comments')`. Show: `Issue::with(['comments.user', 'category', 'user'])`. Verify with `DB::enableQueryLog()` + count assertion in integration tests (SRS §8.5).

8. **IssuePolicy is already complete** (`app/Policies/IssuePolicy.php`) — all five abilities (viewAny, view, create, update, delete) are implemented and tested. Wire in controller via `$this->authorize('viewAny', Issue::class)` / `$this->authorize('view', $issue)` etc. The policy auto-discovers via naming convention; verify with `php artisan policy:list` if needed.

9. **Default sort order** must match SPEC §5.6 / task spec: `needs_attention desc, priority desc, created_at desc`. Priority sort requires a `CASE` expression or `orderByRaw()` since it's a string enum (`low` < `medium` < `high` < `critical`). Use `orderByRaw("CASE priority WHEN 'critical' THEN 4 WHEN 'high' THEN 3 WHEN 'medium' THEN 2 WHEN 'low' THEN 1 END DESC")` — or add a helper on the `Priority` enum.

10. **Test expectations for QA.** Integration tests I-01, I-04, I-05 require `Queue::fake()` to assert `GenerateSummaryJob` dispatch. I-12 requires two sequential updates with stale `updated_at`. I-13 requires seeding 15 issues across statuses/priorities/categories and asserting exact filter results. I-16 must assert soft-deleted issue returns 404 on show and is absent from list. I-17 seeds 30 issues and asserts page 1 has 15 items + page 2 has 15 items with no ID overlap.
