## What To Build

Comment creation endpoint on issues.

### Endpoint
| Method | Route                  | Action      |
| ------ | ---------------------- | ----------- |
| POST   | /issues/{id}/comments  | Add comment |

### Controller
**CommentController** (thin):
- `store(StoreCommentRequest $request, Issue $issue)` → delegates to service or direct create
- Applies authorization: user must have `comment` level access (IssuePolicy::comment)

### Form Request
**StoreCommentRequest**:
- `body`: required, string, trimmed, reject empty after trim

### Authorization (ADR-004 ladder)
- Owner → can comment (implicit edit includes comment)
- Shared(comment) → can comment
- Shared(edit) → can comment
- Shared(view) → CANNOT comment
- Public (no share) → CANNOT comment

### Response
- 201 with comment data including `user: { id, name }` (display as author)
- Comment body stored as-is (after trim)

### Business Rules
- `user_id` set from authenticated user — never from request input
- Issue must exist and not be soft-deleted
- UI displays `user.name` as author name

## Tests Required
Integration: I-03 (comment thread + N+1 verification)
Feature:
- Add comment with valid body → 201
- Add comment with empty body → 422
- Add comment with spaces-only body → 422
- Comment user_id set from auth (not request)
- Comment on non-existent issue → 404
- Comment on inaccessible private issue → 403
- Comment on public issue without share → 403 (view-only, no comment)
- Comment with shared(view) → 403
- Comment with shared(comment) → 201
- Comment with shared(edit) → 201
- Comment response includes user.name

## Technical Guidance

### Files to create
1. **`app/Http/Controllers/CommentController.php`** — generate via `sail artisan make:controller CommentController --no-interaction`; single `store` method only. Follow `IssueController` pattern: inject nothing (comment is simple enough for inline create), authorize via `$this->authorize('create', [Comment::class, $issue])` which hits `CommentPolicy::create()` → `IssuePolicy::comment()`.
2. **`app/Http/Requests/StoreCommentRequest.php`** — generate via `sail artisan make:request StoreCommentRequest --no-interaction`. Use `prepareForValidation()` to trim body (see `StoreIssueRequest` lines 48-57 for the exact pattern). Rules: `'body' => ['required', 'string', 'min:1']` — the `min:1` catches spaces-only *after* the trim in `prepareForValidation`. `authorize()` returns `true` (policy check is in controller).

### Files to modify
3. **`routes/api.php`** — add a nested route: `Route::middleware('auth')->post('issues/{issue}/comments', [CommentController::class, 'store'])`. The `{issue}` binding will 404 on missing issues and exclude soft-deleted (SoftDeletes default). Do NOT use `apiResource` — there is only one action.
4. **No new Resource class needed** — the comment 201 response shape (`id, body, created_at, user: {id, name}`) is identical to what `IssueResource` already renders in its `whenLoaded('comments')` block (lines 61-71). Build an inline array in the controller to match that shape, or extract a `CommentResource` if the coder prefers — either is acceptable as long as the shape matches.

### Files that already exist (no changes needed)
5. `Comment` model (`app/Models/Comment.php`) — already has `BelongsTo user()`, `#[Fillable(['body'])]`, and `UPDATED_AT = null`. `user_id` and `issue_id` are NOT in `$fillable` — they must be set explicitly (not via mass-assignment), which is exactly correct for security.
6. `CommentPolicy` (`app/Policies/CommentPolicy.php`) — `create(User, Issue)` already delegates to `IssuePolicy::comment()`. The policy signature takes `Issue` as the second parameter, so authorization must be called as `$this->authorize('create', [Comment::class, $issue])`.
7. `IssuePolicy::comment()` — already implements the full ADR-004 ladder: owner → yes, share with `canComment()` → yes, else → no. `Permission::canComment()` returns true for `Comment` and `Edit`, false for `View`.
8. `CommentFactory` — exists with `issue_id`, `user_id`, `body`. `IssueShareFactory` has `->comment()` and `->edit()` and `->view()` state helpers.

### Implementation pitfalls
- **`user_id` must be set explicitly**, not from request input: `$comment->user_id = $request->user()->id` (or `$issue->comments()->create([...] + ['user_id' => ...])`) — never trust `$request->validated()` for this field since `body` is the only fillable.
- **`issue_id` via relationship** — use `$issue->comments()->create(...)` so the FK is set by Eloquent, not by mass-assignment.
- **N+1 on show endpoint** — `IssueController::show()` (line 103) already does `$issue->load(['comments.user', ...])`, and `IssueResource` maps `comments.user.name` in `whenLoaded`. No changes needed for the *read* path, but integration test I-03 must assert query count after creating 3 comments then hitting show.
- **Soft-delete guard** — Laravel's `{issue}` route-model binding excludes soft-deleted by default when the model uses `SoftDeletes`. No manual check needed.
- **Public issue without share** — the policy denies comment for non-owners without a share row, even on public issues. This is correct per the task ladder, but test it explicitly (line 51 in Tests Required).

### Testing expectations
- **Feature tests** go in `tests/Feature/CommentApiTest.php` (new file, `RefreshDatabase`). Use `$this->actingAs($user)->postJson("/api/issues/{$issue->id}/comments", ['body' => '...'])` pattern matching `IssueCrudApiTest`.
- **IssueShareFactory states**: `->view()`, `->comment()`, `->edit()` are ready for permission-ladder tests.
- **I-03 integration** — create issue, add 3 comments via API, GET show, assert: (a) response has 3 comments each with `user.name`, (b) use `DB::listen()` or `DB::getQueryLog()` to assert bounded query count (expect ≤5 queries: issue, comments, users, category, shares — not 3N+1).
- **422 on empty body** — post `{'body': ''}` and `{'body': '   '}` (spaces-only) — both must return 422 after trim.
- **Malicious user_id in request** — post `{'body': 'x', 'user_id': 999}` → the stored comment's `user_id` must equal auth user, not 999.
