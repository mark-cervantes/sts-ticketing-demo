# Task 03.02.00: Kanban Board with Drag-Drop Status Transitions

## Business Requirements

### Problem
Users need a visual Kanban board to see all issues organized by status (Open, In Progress, Resolved) and move them between columns via drag-and-drop.

### User Story
As a user, I want to see my issues in a Kanban board with drag-and-drop between columns, so I can quickly triage and update issue status.

### Acceptance Criteria
- [x] Three columns: Open, In Progress, Resolved (from Status enum)
- [x] Issues fetched from `GET /api/issues` with filters applied from sidebar
- [x] Issue cards show: title, priority badge, needs_attention flag, category, summary preview (truncated), deadline
- [x] Cards sortable within column: needs_attention first → priority desc → created_at desc
- [x] Drag card between columns triggers `PATCH /api/issues/{id}` with status update
- [x] Optimistic update: card moves immediately, reverts on server rejection (409/422)
- [x] Sidebar filters (status, priority, category) filter the board in real-time
- [x] Pagination: infinite scroll or "Load more" within each column
- [x] Loading state: skeleton cards during initial fetch
- [x] Empty state per column when no matching issues

### Business Rules
- Drag = status change only (no priority/category change via drag)
- Optimistic locking: send `updated_at` on PATCH; 409 → revert card + toast error
- Filter by status hides/shows entire columns; filter by priority/category filters within visible columns
- needs_attention badge: visual indicator (e.g., orange dot or flame icon)

### Out of Scope
- Issue create modal (03.03)
- Issue detail slide-over (03.04)
- Comment UI (03.05)
- SSE (03.06)

## Technical Guidance

### Architecture Notes

- **Composable pattern:** Create `useKanbanBoard` composable that owns all board state — issues map, column arrays, active filters, loading/pagination cursors. Single source of truth consumed by `Dashboard.vue` and sidebar. Follow the `useDarkMode` pattern (`composables/useDarkMode.ts`) for module-scoped shared refs.
- **Data flow:** Fetch `GET /api/issues` via `fetch()`/axios (NOT Inertia `router.visit`) — these are JSON API endpoints behind `apiResource` (`routes/api.php:8`), not Inertia page routes. Parse the `IssueResource` JSON shape (see `IssueResource.php:25-72`) into typed frontend models, then bucket into three column arrays keyed by `Status` enum values (`open`, `in_progress`, `resolved` — see `app/Enums/Status.php`).
- **Drag-drop:** Use `vue-draggable-plus` (already installed — `package.json:31`, `^0.6.1`). Wrap each column's card list with `<VueDraggable>`. On `end` event, extract the card's `id`, the target column's status, and the card's `updated_at`, then fire `PATCH /api/issues/{id}` with `{ status, updated_at }`. The `updated_at` field is **required** by `UpdateIssueRequest.php:36`.
- **Optimistic update:** On drag-end, immediately move the card into the target column array. On server success, update the card's `updated_at` from the response (the server bumps it by 1 second — `IssueService.php:88`). On 409, revert the card to its original column position and show a Sonner toast (`Toaster` is already mounted in `AppLayout.vue:52`). On 422, revert and show validation error toast.
- **Filter wiring:** `AppSidebar.vue` currently renders skeleton placeholders for Status/Priority/Category filters (lines 20-55). Replace skeletons with checkbox/toggle groups. Status and Priority values are derived from the PHP enums (hardcode the TS enum mirrors — the values are stable: SPEC §4.2). Category list comes from `GET /api/categories` (`routes/api.php:12`). Bind filter state into `useKanbanBoard` — when filters change, re-fetch with query params `?status=X&priority=X&category=slug`.
- **Column sort:** The API default sort is `needs_attention desc → priority desc → created_at desc` (`IssueController.php:73-77`), which matches SPEC §9.4. Do NOT pass `?sort=` — let the server default handle it. Only pass explicit sort params if the user activates a custom sort control.
- **Pagination:** Each column fetches independently with `?status=<column_status>&per_page=15`. Use cursor-based "Load more" button per column. The API returns standard Laravel pagination with `next_page_url` in meta — use that for the next fetch. `appends()` preserves filter params in pagination links (`IssueController.php:85`).

### Gotchas

- **Inertia vs fetch:** The issue endpoints are `apiResource` routes (`routes/api.php:8`) returning JSON, NOT Inertia pages. Do NOT use `router.visit()` or `router.patch()` from `@inertiajs/vue3` — those expect Inertia page responses. Use `fetch()` or axios directly. CSRF cookie is handled by Sanctum's cookie-based auth.
- **Optimistic locking on drag:** Every PATCH must include `updated_at` (required — `UpdateIssueRequest.php:36`). Cache the card's `updated_at` locally and update it from the response after each successful PATCH. Stale `updated_at` → server returns 409 → revert + toast. The server advances `updated_at` by exactly 1 second (`IssueService.php:88`) regardless of wall clock.
- **Column sort is server-side:** The default API sort (`IssueController.php:73-77`) already matches SPEC §9.4 ordering. Don't re-sort client-side — just render cards in API response order. If you re-sort client-side, you risk inconsistency with pagination boundaries.
- **Status filter hides columns:** Per business rules, filtering by status should hide/show entire columns, not filter cards within them. Priority and category filters narrow cards within visible columns. This is a UX distinction — implement it in `useKanbanBoard`, not in the API query (the API always returns all statuses unless `?status=` is specified).
- **Per-column fetching vs single fetch:** For initial simplicity, fetch once with no status filter, then bucket client-side. For pagination, you'll need per-column fetches with `?status=open&page=2`. Design `useKanbanBoard` to support both — initial bulk fetch, then per-column pagination appends.
- **Category slugs:** The filter-by-category API expects `?category=slug` (not ID) — see `Issue::scopeFilterByCategory` (`Issue.php:174-186`) which resolves slug → ID internally. Fetch category list from `GET /api/categories` for the sidebar filter UI.
