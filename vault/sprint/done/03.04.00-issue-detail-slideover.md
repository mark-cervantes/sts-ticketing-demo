# Task 03.04.00: Issue Detail Slide-Over Panel

## Business Requirements

### Problem
Users need to view and edit issue details, see the AI summary, manage sharing, and change settings — all without leaving the Kanban dashboard.

### User Story
As a user, I want to click an issue card to open a slide-over panel showing full details, so I can view and edit issues in context.

### Acceptance Criteria
- [x] Click card → right-side slide-over panel (dashboard stays visible underneath)
- [x] URL updates to `/dashboard?issue={id}` — back button closes, direct link opens
- [x] Shows: title (editable inline), description (editable), priority (editable select), status (editable select), category (editable select), deadline (editable date picker), visibility toggle
- [x] AI summary section: shows summary text + suggested_next_action when status=ready; shows spinner when processing; shows "Summary unavailable" when failed
- [x] Edit fields trigger `PATCH /api/issues/{id}` with optimistic locking (send `updated_at`)
- [x] 409 conflict → toast error + reload fresh data from server
- [x] Delete action via "..." menu → confirm dialog → `DELETE /api/issues/{id}` → close panel, remove card
- [x] Loading state: skeleton loader while fetching `GET /api/issues/{id}`
- [x] Close via X button, Escape key, or clicking outside panel

### Business Rules
- Editing description re-triggers AI summary (status goes to "processing" — handled by backend)
- Status change via select in slide-over should also update the Kanban column position
- needs_attention flag displayed prominently if true
- Share section placeholder (wired in 04.01)

### Out of Scope
- Comment thread UI (03.05)
- SSE live updates (03.06)
- Share management UI (04.04)

## Technical Guidance

### Architecture Notes

- **Composable `useIssueDetail.ts`**: new composable owning fetch/patch/delete + reactive `issue` ref, `loading`, `saving`, `error` state. The `apiFetch`/`apiPatch` helpers in `useKanbanBoard.ts:47-79` are module-private — extract a shared `useApiFetch.ts` utility or duplicate the CSRF+fetch pattern (prefer extraction; both composables need identical `X-XSRF-TOKEN` + 401 handling).
- **Sheet component, not Dialog**: use `Sheet` + `SheetContent side="right"` from `@/components/ui/sheet` (reka-ui `DialogContent` under the hood). The default `sm:max-w-sm` in `SheetContent.vue:44` is too narrow for issue detail — override with `class="sm:max-w-lg"` or `sm:max-w-xl` on the `SheetContent` instance. Mobile sidebar already uses `side="left"` (`AppLayout.vue:46`), so no collision.
- **URL query param sync**: use `window.history.replaceState` (not `router.visit`) to set `?issue={id}` without Inertia page reload. On `Dashboard.vue` mount, check `new URLSearchParams(window.location.search).get('issue')` to auto-open on direct link / back-button. `watch` the Sheet's `open` model — on close, `replaceState` back to `/dashboard`.
- **Kanban state sync on status change**: `columnMap` in `useKanbanBoard.ts:15` is module-scoped singleton. When slide-over PATCH changes `status`, the composable must splice the issue out of old column and unshift into new column in that same `columnMap`. Export a `updateIssueInBoard(issue: Issue, oldStatus: IssueStatus)` function from `useKanbanBoard` for this. When PATCH changes any other field (title, priority, etc.), update the matching issue object in-place within `columnMap`.
- **Kanban state sync on delete**: after successful `DELETE`, splice the issue from its column in `columnMap`. Export a `removeIssueFromBoard(issueId: number, status: IssueStatus)` function from `useKanbanBoard`.
- **Optimistic locking**: every PATCH must include `updated_at` from the current `issue` ref (`IssueService.php:62-67` compares against DB). On success, update the local `issue.updated_at` from the response (`responseData.data.updated_at`). On 409, toast + re-fetch `GET /api/issues/{id}` to get fresh data.
- **Summary section states**: key off `issue.summary_status` (type `SummaryStatus` in `types/issue.ts:10` — values: `pending`, `processing`, `completed`, `failed`). Show summary text + `suggested_next_action` when `completed`; show spinner when `pending` or `processing`; show "Summary unavailable" when `failed`. The AC says "ready" but the actual enum is `completed` — use the enum.
- **Description edit → summary re-trigger**: when user edits description and PATCH succeeds, backend resets `summary_status=pending` and dispatches `GenerateSummaryJob` (`IssueService.php:81-95`). The PATCH response already reflects `summary_status: 'pending'` — update the local ref so the UI immediately shows the spinner state. SSE push is 03.06; until then, the user sees "processing" until they re-open.
- **Inline editing pattern**: use controlled inputs that PATCH on blur/enter, not a form submit. Debounce is NOT needed — each field patches independently on commit. Disable the field briefly while the PATCH is in flight to prevent double-submit.
- **IssueCard click handler**: `IssueCard.vue` currently has no `@click` — add `$emit('select', issue.id)` on the card wrapper `div`. The drag-start (`@dragstart`) already fires first, so gate: only emit `select` if the pointer didn't move (use `@mousedown`/`@mouseup` position delta or `@click` which doesn't fire after drag).
- **Delete confirmation**: use `AlertDialog` from `@/components/ui/alert-dialog` (not `Dialog` — AlertDialog is for destructive confirmations per shadcn-vue convention).

### Gotchas

- **URL sync without full page reload**: `router.visit()` or `router.get()` would trigger a full Inertia page cycle and remount the kanban board (losing scroll position, column state). Use `window.history.replaceState` / `pushState` + `popstate` listener instead. Do NOT use Inertia's `router.replace` — it also triggers a server request.
- **Sheet vs Dialog z-index**: Sheet and Dialog both use `z-50` (`SheetContent.vue:44`). The delete `AlertDialog` inside the Sheet will stack correctly because reka-ui portals manage stacking order — but verify visually that the AlertDialog overlay covers the Sheet.
- **Kanban singleton state**: `columnMap` is module-scoped (`useKanbanBoard.ts:15`), so any import of `useKanbanBoard()` in the slide-over composable accesses the same reactive object. This is intentional — but means mutations must be careful not to break reactivity (use `splice`, not reassignment of array items by index).
- **Summary status enum mismatch**: The AC says "status=ready" but the `SummaryStatus` enum (`types/issue.ts:10`) uses `completed`. Use the actual enum value `completed` in the template, not `ready`.
- **SheetContent max-width**: default is `sm:max-w-sm` (24rem / 384px) — far too narrow for issue editing. Must pass a wider class override. Test that the wider panel doesn't overlap the sidebar on `md` breakpoint (sidebar is `var(--sidebar-width)` = 16rem).
- **Mobile: Sheet full-screen**: on `< 640px`, the Sheet's `w-3/4` still leaves a narrow strip visible. Consider overriding to `w-full` on small screens so the panel is a full takeover, matching mobile UX expectations.
- **Click vs drag on cards**: `IssueCard.vue` wrapper has `cursor-grab`. A naive `@click` fires after drag release. Use a threshold (e.g., track `mousedown` position, compare on `mouseup`, only emit `select` if delta < 5px) or rely on the native `click` event which browsers suppress after a drag — test both in Chrome and Firefox.
