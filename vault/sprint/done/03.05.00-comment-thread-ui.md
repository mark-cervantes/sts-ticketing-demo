# Task 03.05.00: Comment Thread UI

## Business Requirements

### Problem
Users need to read and add comments on issues directly within the slide-over panel.

### User Story
As a user, I want to see a comment thread at the bottom of the issue detail panel and add new comments, so I can collaborate on issue resolution.

### Acceptance Criteria
- [x] Comment thread displayed at bottom of issue slide-over (03.04)
- [x] Comments loaded from `GET /api/issues/{id}` response (already includes comments with user)
- [x] Each comment shows: author name, timestamp (relative, e.g., "2 hours ago"), body text
- [x] "Add comment" input at bottom: textarea + submit button
- [x] Submit sends `POST /api/issues/{issue}/comments`; on success: optimistically append to thread, clear input
- [x] Validation: empty body ŌåÆ inline error (422 mapped)
- [x] Loading state on submit
- [x] Scroll to bottom after new comment added
- [x] Empty state: "No comments yet" when thread is empty

### Business Rules
- Comments ordered chronologically (oldest first)
- Only users with comment permission see the input (owner or share with comment/edit permission)
- Comment count badge on issue card updates after successful comment

### Out of Scope
- Comment editing/deletion
- @mentions
- Rich text / markdown in comments

## Technical Guidance

### Architecture Notes

- **New component: `CommentThread.vue`** ŌĆö extract into `resources/js/components/issues/CommentThread.vue`; receives `comments` array + `issueId` + `canComment` as props; `IssueDetailSheet.vue:524-527` placeholder becomes `<CommentThread>`.
- **New TS interface: `IssueComment`** ŌĆö add to `resources/js/types/issue.ts` matching the shape in `IssueResource.php:62-69`: `{ id, body, created_at, user: { id, name } }`. Extend the `Issue` interface with optional `comments?: IssueComment[]` (present on show, absent on list).
- **`apiPost` helper needed** ŌĆö `useApiFetch.ts` has `apiFetch`/`apiPatch`/`apiDelete` but no POST; add `apiPost(url, body): Promise<Response>` following the `apiPatch` pattern (same CSRF + credentials).
- **Permission gate: `can_comment` must come from backend** ŌĆö the frontend cannot derive share-based permission from `issue.user_id` alone (`IssuePolicy::comment` at `IssuePolicy.php:86` checks `$share->permission->canComment()`). Add `'can_comment' => $request->user()->can('comment', $this->resource)` to `IssueResource.php:toArray()` and a matching `can_comment?: boolean` field to the TS `Issue` type. Hide the input textarea when falsy.
- **Optimistic append** ŌĆö on 201 from `CommentController::store` (which returns the same `{ data: { id, body, created_at, user } }` shape at `CommentController.php:37-47`), push into the local comments array and `nextTick(() => scrollContainer.scrollTo({ top: scrollContainer.scrollHeight, behavior: 'smooth' }))`. No re-fetch needed.
- **Kanban card `comments_count` sync** ŌĆö after successful POST, call `useKanbanBoard().updateIssueInBoard()` with an incremented `comments_count` so the badge updates without a full board reload (same pattern as `useIssueDetail.ts:75`).

### Technical DoD

**Visual:**
- Empty state ("No comments yet") visible when `comments` array is empty.
- Comment thread scrolls independently if it exceeds available height (use `max-h-` + `overflow-y-auto` on thread container).
- Input hidden when `can_comment` is false; no flicker on load (v-if on resolved value).
- Loading spinner on submit button while POST is in-flight (reuse `Loader2Icon` pattern from `IssueDetailSheet.vue:554`).

**Code:**
- `CommentThread.vue` receives typed props ŌĆö no `any`.
- Relative timestamps via `Intl.RelativeTimeFormat` or a small util ŌĆö do not add `date-fns` / `dayjs` (check existing deps first; if already present, use it).
- 422 from POST ŌåÆ show inline validation error below textarea (map `response.json().errors.body[0]`), do not use toast for field validation.
- 403 from POST ŌåÆ toast "Permission denied" (should not happen if `can_comment` gate is correct, but defend).

### Gotchas

- **`Issue.comments()` has no `orderBy`** (`Issue.php:114-116`) ŌĆö Eloquent returns by PK ascending which is chronological, but if IDs are ever non-sequential (e.g., UUID migration), this breaks. Consider adding `->orderBy('created_at')` to the relation or sorting in `IssueResource`. Low risk now but worth noting.
- **`StoreCommentRequest` trims body** (`StoreCommentRequest.php:39`) ŌĆö the frontend should also trim before checking empty to avoid user confusion (submit appears enabled but server rejects).
- **`SheetContent` scroll** ŌĆö `IssueDetailSheet.vue:248` uses `overflow-y-auto` on the entire sheet; the comment thread's own scroll container must not fight this. Use a fixed `max-h` on the thread list, not the outer sheet.
- **No `updated_at` locking on comments** ŌĆö unlike issue PATCH, comment POST is create-only (no conflict scenario), so no optimistic lock header needed.
