# Task 03.06.00: SSE Client for Live Summary Updates

## Business Requirements

### Problem
When an AI summary is processing, the user shouldn't need to refresh to see the result. The summary section should update live.

### User Story
As a user, I want to see the AI summary appear automatically when it's ready, without refreshing the page.

### Acceptance Criteria
- [x] Backend: `GET /api/issues/{id}/stream` SSE endpoint (auth required)
- [x] Emits events on `summary_status` transitions (processing → ready, processing → failed)
- [x] Event payload includes: summary_status, summary text (when ready), suggested_next_action (when ready)
- [x] Frontend: when issue detail slide-over is open and summary_status is "processing", connect to SSE
- [x] On "ready" event: update summary section with new text, show success indicator
- [x] On "failed" event: show "Summary unavailable" message
- [x] Auto-reconnect on disconnect (EventSource built-in, or manual retry)
- [x] Close SSE connection when slide-over closes or issue changes
- [x] Graceful degradation: if SSE connection fails, fall back to polling every 10s

### Business Rules
- SSE stream is per-issue, auth-gated
- Only connect when summary is actually processing (don't open unnecessary connections)
- SummaryCompleted event (02.04) triggers the SSE push

### Out of Scope
- Real-time comment updates
- Presence indicators
- WebSocket upgrade

## Technical Guidance

### Architecture Notes

- **Backend SSE controller:** Create `app/Http/Controllers/IssueSseController.php` with a single `__invoke(Request $request, Issue $issue)` method returning a `StreamedResponse` (Symfony). Set headers `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `X-Accel-Buffering: no` (disables nginx/Caddy buffering). Inside the loop: poll `$issue->fresh()->summary_status` every 2s; when status transitions to `ready` or `failed`, emit one SSE event and `return` (close the connection). ADR-001 chose SSE over WebSocket specifically because summary push is unidirectional.
- **No broadcasting needed:** `SummaryCompleted` event (`app/Events/SummaryCompleted.php:15`) does not implement `ShouldBroadcast` — keep it that way. The SSE controller polls the DB directly; it does not depend on Laravel Echo or Pusher. This is intentional per ADR-001 line 34–36: no WebSocket server, no channel auth layer.
- **Route registration:** Add `Route::middleware('auth')->get('issues/{issue}/stream', IssueSseController::class)` in `routes/api.php` (SPEC §5.5). The `{issue}` binding gives implicit model binding; authorize with `$this->authorize('view', $issue)` which calls `IssuePolicy::view()` (`app/Policies/IssuePolicy.php:32`).
- **Frontend composable:** Create `resources/js/composables/useSummaryStream.ts` — wraps `EventSource` or `fetch` with streaming. Accepts `issueId: Ref<number | null>` and `enabled: Ref<boolean>`. Returns `{ status, summary, suggestedNextAction, connectionError }`. On event received, update the reactive refs. On error after 3 retries, set `connectionError = true` and fall back to polling `GET /api/issues/{id}` every 10s.
- **Connection lifecycle in IssueDetailSheet:** In `IssueDetailSheet.vue`, call `useSummaryStream(issueId, isProcessing)` where `isProcessing = computed(() => issue.value?.summary_status === 'processing' || issue.value?.summary_status === 'pending')`. Watch for SSE data to arrive and merge into the `issue` ref via the existing `useIssueDetail` composable. Close connection on `onUnmounted` and when `props.open` becomes `false` (slide-over close watcher at line 150–154).

### Gotchas

1. **SSE + session auth:** `EventSource` cannot set custom headers. Since this app uses Laravel Breeze session auth (ADR-001), cookies are sent automatically — this works. However, `EventSource` does not support `withCredentials` in all contexts; if the API is cross-origin, set `new EventSource(url, { withCredentials: true })`. For same-origin (our case), this is automatic.
2. **Connection timeout / keepalive:** Caddy and most reverse proxies will kill idle connections after ~60s. The SSE controller must send a keepalive comment (`: keepalive\n\n`) every 15–20s to prevent premature disconnect. Without this, the client's `onerror` fires and reconnects, creating unnecessary churn.
3. **`ob_end_clean()` in controller:** PHP output buffering will prevent SSE events from flushing. Call `ob_end_clean()` at the start of the streamed callback and `flush()` after each `echo`. Without this, the client receives nothing until the connection closes.
4. **EventSource auto-reconnect:** `EventSource` reconnects automatically on network errors — but it does NOT reconnect on HTTP 4xx/5xx. If the endpoint returns 403 (auth expired), the `onerror` handler fires once and `readyState` becomes `CLOSED`. The composable must detect this and switch to polling fallback rather than retrying SSE infinitely.
5. **`SummaryStatus` type mismatch (`issue.ts:10`):** The frontend type uses `'completed'` but the backend sends `'ready'`. This will cause `summaryState` computed (`IssueDetailSheet.vue:117`) to never match the completed branch. Fix the type to `'ready'` and update the computed check. This is a latent bug from 03.04.00 — the seeder likely sends `'ready'` too, so the summary section may already be broken in production for ready-state issues.
6. **Queue timing:** The SSE controller polls the DB; `GenerateSummaryJob` updates `summary_status` in the same DB. If the queue worker and SSE controller read from different DB connections (read replica), there could be replication lag. For this project (single Postgres via Docker Compose), this is not an issue, but document the assumption.
