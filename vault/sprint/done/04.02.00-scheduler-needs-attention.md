# Task 04.02.00: Scheduler for needs_attention Recalculation

## Business Requirements

### Problem
The `needs_attention` flag is computed on issue create/update, but deadlines pass with time. An issue that wasn't urgent yesterday may need attention today if its deadline is approaching. A scheduled job should periodically recalculate.

### User Story
As a user, I want issues approaching their deadline to automatically get flagged as needing attention, even if I haven't edited them recently.

### Acceptance Criteria
- [x] Artisan command `issues:recalculate-attention` that recalculates `needs_attention` for all open/in-progress issues
- [x] Laravel scheduler runs the command every 15 minutes
- [x] Only issues where `needs_attention` actually changed get updated (no unnecessary writes)
- [x] Command outputs count of issues updated
- [x] PHPUnit test for the command
- [x] All tests pass

### Business Rules
- `needs_attention` = true when: priority is critical OR deadline is within 24 hours (SPEC §6.6)
- Resolved issues are excluded from recalculation
- Soft-deleted issues are excluded

### Out of Scope
- Real-time push of attention changes to frontend (would need websocket/SSE extension)

## Technical Guidance

- Generate the command with `sail artisan make:command RecalculateAttentionCommand --no-interaction`; set `$signature = 'issues:recalculate-attention'`
- Query scope: `Issue::whereIn('status', [Status::Open, Status::InProgress])` — excludes `Resolved` per BR; `SoftDeletes` trait on `Issue` already excludes trashed rows automatically
- Reuse `Issue::computeNeedsAttention()` (`app/Models/Issue.php:82`) for the recalculation — it's a pure static method, keeps logic in one place
- Compare `computeNeedsAttention()` result against current `$issue->needs_attention`; only call `$issue->updateQuietly(['needs_attention' => $new])` when the value changed — avoids unnecessary writes AND skips the `saving` event (which would redundantly recompute)
- The threshold is `config('issues.attention_threshold_minutes', 60)` (`config/issues.php:4`), not a hardcoded 24h — the command must not reimplement this
- Register the schedule in `routes/console.php` (Laravel 13 has no `Kernel.php`): `Schedule::command('issues:recalculate-attention')->everyFifteenMinutes()`
- Use `$this->info("Updated {$count} issues.")` for command output; return `Command::SUCCESS`
- Chunk the query (`->chunkById(200, ...)`) to avoid loading all issues into memory at once
- Test in `tests/Feature/Console/RecalculateAttentionCommandTest.php`: create issues via `Issue::factory()->needsAttention()` and `Issue::factory()` states, freeze time with `$this->travel()`, assert DB changes and command output; set `Config::set('issues.attention_threshold_minutes', 60)` for hermetic tests (see `tests/Unit/Models/IssueTest.php:25`)
- Resolved issues must remain untouched even if they have approaching deadlines — assert this explicitly in the test
