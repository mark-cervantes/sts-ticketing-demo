# Task 04.04.00: Share UI in Issue Detail Slide-Over

## Business Requirements

### Problem
Issue owners need a UI to manage sharing directly from the issue detail slide-over.

### User Story
As an issue owner, I want to add, modify, and remove shares from the issue detail panel, so I can control who has access.

### Acceptance Criteria
- [x] Share section in issue detail slide-over (replaces "Sharing — coming soon" placeholder)
- [x] Shows list of current shares: user email/name, permission level badge
- [x] "Add share" input: email field + permission dropdown (view/comment/edit) + add button
- [x] Permission dropdown to change existing share level (PATCH)
- [x] "x" button to remove a share (DELETE with confirmation)
- [x] Only visible to issue owner
- [x] Inline validation errors (invalid email, user not found, already shared)
- [x] Toast on success/failure
- [x] All tests pass

### Business Rules
- Share section only visible when current user is the issue owner
- Permission levels display as human-readable labels: "Can view", "Can comment", "Can edit"
- Adding a share for an already-shared user should update the permission (upsert behavior from 04.01)

### Out of Scope
- Email notifications
- Batch sharing

## Technical Guidance

- Create `useIssueShares.ts` composable (new file) following the `useIssueDetail.ts` pattern — module-scoped refs for `shares`, `loading`, `saving`; methods `fetchShares(issueId)`, `addShare(issueId, email, permission)`, `updatePermission(shareId, permission)`, `removeShare(shareId)`; use `apiFetch`/`apiPost`/`apiPatch`/`apiDelete` from `useApiFetch.ts`
- Add `Share` and `SharePermission` types to `types/issue.ts` — `SharePermission = 'view' | 'comment' | 'edit'`; `Share = { id: number; permission: SharePermission; created_at: string; user: { id: number; name: string; email: string } }` — mirrors the shape returned by `ShareController::index()` lines 37–46
- Owner gate in the template: `issue.user_id` is already on the `Issue` type (line 45 of `types/issue.ts`); compare against `usePage().props.auth.user.id` to conditionally render the share section — no new API flag needed
- Replace the placeholder at `IssueDetailSheet.vue:564` (`"Sharing — coming soon"`) with a new `<ShareSection>` component that receives `issueId` and encapsulates all share CRUD; keeps the slide-over under control
- API routes are shallow: index/store nest under `/api/issues/{issue}/shares`, but update/destroy use `/api/shares/{share}` — the composable must use the correct base URL for each operation (see `ShareController` class docblock lines 16–19)
- Store endpoint returns 422 with `errors.email` array for both "not found" and "cannot share with self" — render these inline under the email input, not as toast; toast only for network/unexpected errors (consistent with `StoreShareRequest::messages()`)
- `Permission` enum values are lowercase strings (`view`, `comment`, `edit` — see `Permission.php:12–14`); map to display labels "Can view" / "Can comment" / "Can edit" in the UI per business rule
- Delete confirmation: reuse the `AlertDialog` pattern already in `IssueDetailSheet.vue:586–607`; track `deleteShareId` ref to know which share the dialog targets
- The composable should call `toast.success()` on successful add/remove (pattern established in `useIssueDetail.ts:110`); avoid success toasts on permission updates (inline select change is self-evident)
- Fetch shares on slide-over open alongside `fetchIssue` — add `fetchShares(id)` call inside the existing `watch([props.open, props.issueId])` at `IssueDetailSheet.vue:165–180`; clear share state in the `!isOpen` branch
