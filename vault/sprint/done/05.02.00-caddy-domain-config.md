# Task 05.02.00: Caddy Reverse Proxy + Domain Config

## Business Requirements

### Problem
The production host (192.168.254.140) already runs Caddy as a reverse proxy. The app needs a Caddyfile entry to route `sts-demo.betamaxgroup.tech` to the app container.

### User Story
As a deployer, I want the app accessible at `https://sts-demo.betamaxgroup.tech` via the existing Caddy proxy on the host.

### Acceptance Criteria
- [x] `Caddyfile.prod` or deployment docs with Caddy config for `sts-demo.betamaxgroup.tech`
- [x] Reverse proxy to app container's nginx/FPM port
- [x] Automatic HTTPS via Caddy's built-in ACME (if DNS points to host)
- [x] WebSocket/SSE passthrough for `/api/issues/{id}/stream` (no buffering)
- [x] Trusted proxies configured in Laravel (`TrustProxies` middleware) for correct `X-Forwarded-*` headers
- [x] `.env.production.example` with all required production env vars documented
- [x] Deployment README section explaining how to deploy to the host
- [x] All tests pass (no breakage)

### Business Rules
- Caddy runs on the HOST, not in Docker ‚Äî the Caddyfile is a host-level config file
- App container exposes port only on localhost (not 0.0.0.0) ‚Äî Caddy proxies to it
- SSE endpoint must NOT be buffered by proxy (needs `flush_interval -1` or similar)

### Out of Scope
- DNS configuration (assumed already pointing)
- SSL certificate management (Caddy handles this automatically)
- CI/CD

## Technical Guidance

- Create `Caddyfile.prod` at repo root; `reverse_proxy localhost:8080` targets the nginx service port mapping in `docker-compose.prod.yml:57` (`"8080:80"`)
- Add a `handle_path /api/issues/*/stream` block with `flush_interval -1` ‚Äî `IssueSseController` (line 34) streams for up to 120s and Caddy's default buffering will break SSE `text/event-stream` delivery
- `docker/nginx/default.conf:46` sets `fastcgi_read_timeout 60s` but `IssueSseController::TIMEOUT` is 120s ‚Äî add an nginx `location ~ ^/api/issues/\d+/stream$` block with `fastcgi_read_timeout 130s` and `proxy_buffering off` to prevent nginx from killing the SSE connection before the controller's own timeout
- Laravel 13 has no `app/Http/Middleware/TrustProxies.php` ‚Äî configure trusted proxies via `TRUSTED_PROXIES=127.0.0.1,::1` in `.env.production.example` (covers host-local Caddy); alternatively register `$middleware->trustProxies(at: '*')` in `bootstrap/app.php:16` but wildcard is less secure
- Set `APP_URL=https://sts-demo.betamaxgroup.tech` in `.env.production.example:6` ‚Äî must match the Caddy domain so `url()` and Inertia's `ziggy` generate correct HTTPS URLs behind the proxy
- `docker-compose.prod.yml:57` binds `"8080:80"` on all interfaces (`0.0.0.0`); for security, change to `"127.0.0.1:8080:80"` so only host-local Caddy can reach it (task requirement: "port only on localhost")
- ADR-008 (line 38-43) has a skeleton Caddy block missing `flush_interval`, `header_up`, and `transport` directives ‚Äî update it or reference the new `Caddyfile.prod` as the authoritative source
- README.md line 349 lists a `caddy` Docker service that doesn't exist in `docker-compose.prod.yml` ‚Äî update the "Production services" table to remove `caddy` row and add a note that Caddy runs on the host
- Caddy should forward `Host`, `X-Real-IP`, and `X-Forwarded-*` headers ‚Äî Caddy v2 does this by default with `reverse_proxy`, but explicitly set `header_up X-Forwarded-Proto {scheme}` to ensure HTTPS detection works through the Inertia/Ziggy stack
- Add `TRUSTED_PROXIES` to `docker-compose.prod.yml` environment blocks for the `app` service (alongside existing env vars at line 9-31) so the runtime container picks it up without a separate `.env` file inside the container
