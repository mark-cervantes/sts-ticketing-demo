# Task 05.03.00: README + Architecture Documentation

## Business Requirements

### Problem
The project needs comprehensive documentation for submission/evaluation — explaining the architecture, how to run it, what was built, and design decisions made.

### User Story
As an evaluator, I want to read a clear README that explains what this project is, how to run it, and what architectural decisions were made, so I can assess the work.

### Acceptance Criteria
- [x] `README.md` at project root with:
  - Project overview (what it does, key features)
  - Tech stack summary
  - "Try It Locally" section (prerequisites, one-command setup, seeded credentials)
  - Architecture overview with Mermaid diagram (request flow, service layer, AI pipeline)
  - API endpoint summary table
  - Testing section (how to run, what's covered, test counts)
  - Design decisions / ADR summary (link to vault/docs/adr/)
  - Screenshots of the Kanban dashboard (light + dark mode)
- [x] Architecture diagram shows: Browser → Inertia → Controller → Service → Model, Queue → GenerateSummaryJob → SummaryManager → Drivers, SSE stream
- [x] All existing ADRs referenced and summarized
- [x] Screenshots captured and committed to `docs/screenshots/`
- [x] All tests pass (no breakage)

### Business Rules
- README must be self-contained — evaluator should not need to dig into vault/ to understand the project
- Screenshots should show real seeded data, not empty states
- Mermaid diagrams render in GitHub markdown

### Out of Scope
- API documentation (Swagger/OpenAPI)
- Inline code documentation (PHPDoc is already in place)

## Technical Guidance

- The existing `README.md` (322 lines) already covers setup, tour, troubleshooting, project layout, and AI workflow — **do not rewrite from scratch**; surgically add the missing sections (Mermaid diagram, API table, ADR summary, screenshots, explicit features list).
- Replace the ASCII architecture box (lines 43–65 in current README) with a Mermaid `graph TD` diagram that traces: `Browser --> InertiaMiddleware --> Controller --> Service --> Model`, plus a parallel path `QueueWorker --> GenerateSummaryJob --> SummaryManager --> LlmDriver/RulesDriver`, plus `IssueSseController --> Browser` — these are the exact class names in `app/Services/Summary/`, `app/Jobs/`, `app/Http/Controllers/IssueSseController.php`.
- Add a second Mermaid `graph LR` for the AI fallback chain specifically: `SummaryManager --> LlmDriver --fail--> LlmDriver(OpenRouter) --fail--> RulesDriver` — this matches `app/Services/Summary/SummaryManager.php` driver resolution and ADR-002.
- Build the API endpoint table from `sail artisan route:list --except-vendor` — there are 35 routes, but group only the 15 app routes (exclude auth/profile/verification); columns: Method, URI, Name, Controller.
- For ADR summaries, read the first 5 lines of each `vault/docs/adr/00{1..9}*.md` and `010*.md` — the context line in each file is the one-line summary; format as a table: `| ADR | Title | Decision |` with links to `vault/docs/adr/<file>`.
- Screenshots: `mkdir -p docs/screenshots/` then use `make verify-visual` (which runs Playwright smoke) to capture; if that doesn't produce the right screenshots, run Playwright directly against `http://localhost` as `demo@example.com` / `password` — capture dashboard (light mode), dashboard (dark mode), issue detail modal, and share dialog.
- Add a `## Key Features` section near the top listing: Kanban dashboard with drag-and-drop, async AI summary pipeline with 3-tier fallback (SPEC §5), SSE real-time updates, Google Docs-style sharing with ladderized permissions (SPEC §3.2), optimistic locking (`lock_version` on issues model), deadline + needs_attention auto-flagging (ADR-005).
- Current test count is **283 tests, 649 assertions** (PHPUnit 12) — use this exact number in the testing section rather than a vague "comprehensive suite".
- Keep the existing `## When Things Break` troubleshooting table — it's evaluator-friendly and not duplicated by any new section.
