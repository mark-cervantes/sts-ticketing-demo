# Task 05.04.00: Final Seed Verification + Polish

## Business Requirements

### Problem
Before submission, the seeded demo data should be verified end-to-end through the production Docker setup. Any rough edges in the demo experience should be polished.

### User Story
As an evaluator, I want the demo data to tell a realistic story when I log in, so I can see all features in action without manual setup.

### Acceptance Criteria
- [x] `docker compose -f docker-compose.prod.yml up -d` → app boots with seeded data
- [x] Login as `demo@example.com` / `password` → Kanban board shows issues across all 3 columns
- [x] Issues have realistic titles, descriptions, priorities, categories
- [x] At least 2 issues have AI summaries (summary_status=ready)
- [x] At least 1 issue has needs_attention=true (visible flag on card)
- [x] At least 1 issue has comments
- [x] At least 1 issue is shared with another user
- [x] Drag-drop between columns works
- [x] Create new issue via modal works
- [x] Issue detail slide-over shows summary + comments
- [x] Dark mode toggle works
- [x] No console errors in browser
- [x] All tests pass

### Business Rules
- Demo data must be reproducible: `migrate:fresh --seed` produces identical state
- Demo user must have enough issues to demonstrate filtering and pagination

### Out of Scope
- Performance optimization
- Load testing

## Technical Guidance

- **3-column coverage already met:** Demo user (`$users[0]`) owns issues #1 (InProgress), #6 (Resolved), #11 (Open), #16 (Open) in `IssueSeeder.php` — all 3 Kanban columns populated.
- **AI summaries already met:** Issues #1 and #3 use `summaryReady()` state (`IssueFactory.php:187-199`), which sets `summary_status=SummaryStatus::Ready`, `summary`, and `suggested_next_action` from the `$summaryBank` — but neither belongs to the demo user; confirm the slide-over can display them via share or public visibility.
- **needs_attention already met:** Issues #1-#5 trigger `needs_attention=true` via `critical()`/`highPriority()` in the `Issue::saving` event (`Issue.php:67`); issue #1 belongs to the demo user.
- **IssueShareSeeder gap:** `IssueShareSeeder.php:27` uses `$issues->shuffle()->take(3)` — non-deterministic; the demo user may never appear as share recipient; pin at least one share so that `demo@example.com` receives a shared issue (use `firstOrCreate` with the demo user's ID explicitly).
- **CommentSeeder coverage OK but cosmetic:** Every issue gets 2-4 comments (`CommentSeeder.php:25`), so the "≥1 issue has comments" criterion passes; body text is `fake()->paragraph()` — consider replacing with realistic comment text on 1-2 demo-user issues if the evaluator will read the slide-over.
- **Verification via Sail, not prod Docker:** Run `./vendor/bin/sail artisan migrate:fresh --seed` then manually check the Kanban at `http://localhost` — no need to build `docker-compose.prod.yml` for seed data verification; the prod compose can be smoke-tested separately.
- **Reproducibility check:** After `migrate:fresh --seed`, run a second `migrate:fresh --seed` and compare issue counts via `./vendor/bin/sail artisan tinker --execute 'echo Issue::count() . " " . Comment::count() . " " . IssueShare::count();'` — counts must be identical across runs (the `shuffle()` in `IssueShareSeeder` makes share _targets_ vary but count stays 3).
- **summary_status on demo-user issues:** If the evaluator expects to see an AI summary in the demo user's slide-over, either move one `summaryReady()` call to a demo-user issue (e.g., issue #11 or #16), or ensure the share pin from bullet 4 targets a `summaryReady()` issue.
- **No seeder changes should break existing tests:** Run `./vendor/bin/sail test --compact` after any seeder modification — the `DatabaseSeeder` runs inside a transaction (`DatabaseSeeder.php:26`) so partial failures are atomic.
- **Acceptance items 8-12 (drag-drop, modal, dark mode, console errors) are UI verification only:** No seeder changes needed — verify manually in the browser after seeding; if `make verify-visual` is available, use it to cover console-error and screenshot checks.
