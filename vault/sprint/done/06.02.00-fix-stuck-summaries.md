# Task 06.02: Reset Stuck Summaries and Verify Generation Pipeline

## Business Requirements

### Problem
Due to Horizon being inactive (fixed in 06.01), 29 issues are stuck in `pending` status and 2 in `processing` status. After Horizon is fixed, these existing stuck issues need their summaries regenerated. New issues created after the fix will process normally, but the backlog of 31 stuck issues must be cleared.

### Current State (production)
- `pending`: 29 issues (job was never dispatched or dispatched but never picked up)
- `processing`: 2 issues (job started but Horizon died mid-process)
- `ready`: 4 issues (generated during seed via sync queue)
- `failed`: 2 issues

### Acceptance Criteria
- [ ] Artisan command exists: `php artisan summaries:retry-stuck` — resets `processing`→`pending`, then dispatches `GenerateSummaryJob` for all `pending` issues
- [ ] After running the command with Horizon active, all previously-pending issues reach `ready` or `failed` status (no stuck `pending`/`processing`)
- [ ] The rules driver generates deterministic summaries (no LLM key needed)
- [ ] SSE endpoint properly streams `summary.ready` events when a summary completes
- [ ] The "Generating summary…" spinner in the UI resolves to actual summary text after job completes
- [ ] 283+ PHPUnit tests pass (add test for the new artisan command)

### Business Rules
- The retry command should be idempotent — safe to run multiple times
- Issues already in `ready` or `failed` status should NOT be re-processed
- Failed issues from the retry should log the error but not crash the command

### Out of Scope
- LLM driver setup (rules driver is the production default)
- Changing the summary regeneration trigger logic
