# Task 06.03: Wire Kanban Drag-and-Drop to Update Issue Status via API

## Business Requirements

### Problem
Dragging a Kanban card from one column to another (e.g., "In Progress" → "Open") visually moves the card but **does not update the issue's status** in the database. Refreshing the page reverts the card to its original column. The drag-and-drop is cosmetic only.

### Root Cause
`KanbanColumn.vue` defines a `moveIssue` emit (line 19) but **never fires it**. The `VueDraggable` component has `v-model="localIssues"` and `group="kanban"` which allows cross-column dragging, but there is no `@end` or `@update` handler that detects when a card lands in a different column and triggers the API call.

### Architecture
- `KanbanColumn.vue` → needs `@end` handler on `VueDraggable` that detects cross-column drops
- `KanbanBoard.vue` → already has `moveIssue(issueId, fromStatus, toStatus)` wired
- `useKanbanBoard.ts` composable → already has `moveIssue()` that calls `PATCH /api/issues/{id}` with `{ status: toStatus }`

### Acceptance Criteria
- [ ] Dragging a card from "Open" to "In Progress" updates the issue status to `in_progress` via API
- [ ] Dragging from "In Progress" to "Resolved" updates status to `resolved`
- [ ] Dragging from "Resolved" back to "Open" updates status to `open`
- [ ] Column counts update immediately after drop (optimistic UI)
- [ ] On API failure, card reverts to original column with toast error
- [ ] Page refresh after drag shows the card in its new column (persisted)
- [ ] PHPUnit tests pass; visual verification via `make verify-visual`

### Business Rules
- Status transitions are unrestricted (any status → any status) per SRS
- Only the card's status field changes — title, priority, description unchanged
- The `data-from-status` attribute on each card div should be used to detect the source column

### Out of Scope
- Status transition restrictions (e.g., preventing Resolved → Open)
- Drag-and-drop reordering within the same column
