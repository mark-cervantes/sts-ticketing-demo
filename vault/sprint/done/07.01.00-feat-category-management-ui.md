# Task 07.01: Category Management UI — Create and Delete Categories

## Business Requirements

### Problem
Users cannot create new issue categories from the UI. The backend API supports `POST /api/categories` and `DELETE /api/categories/{id}`, but there is no UI to use them. Users are limited to the categories seeded at setup time.

### User Story
As a user, I want to create custom categories for my issues so that I can organize them according to my team's workflow.

### Acceptance Criteria
- [ ] Category management is accessible from the sidebar or settings area
- [ ] User can create a new category by typing a name (slug auto-generated by backend)
- [ ] New category immediately appears in the "Category" dropdown on the Create Issue modal
- [ ] User can delete a category that has no issues attached (API returns 409 with count if in use)
- [ ] Inline error message shown when trying to delete a category in use
- [ ] Duplicate category name shows validation error
- [ ] Visual verification via `make verify-visual`

### Business Rules
- Categories are global (shared across all users) per SRS §FR-08
- Category names must be unique (case-insensitive, handled by backend)
- Categories with existing issues cannot be deleted (foreign key constraint)
- No category editing (rename) — create new + migrate issues if needed

### Out of Scope
- Category editing/renaming
- Category-level permissions
- Category colors or icons

### Architecture Notes
- Place category management at `resources/js/Pages/Settings/CategorySettings.vue` with an Inertia route `GET /settings/categories` (name: `settings.categories`) in `routes/web.php`, mirroring the existing `/settings/statuses` pattern — `SRS §FR-08 UX` calls for a dedicated management surface, not a modal
- Add the Inertia route in `routes/web.php` inside the existing `auth` middleware group, following the same closure pattern as `settings.statuses` at line 31
- Add `deleteCategory(id: number)` to `resources/js/composables/useCategories.ts`; call `apiDelete('/api/categories/{id}')`, check `response.status === 409`, parse `{ message }` from the body; remove from the local `categories` ref only on `204` — mirrors the exact pattern in `StatusSettings.vue:280–316`
- The 409 response body from `CategoryController::destroy` is `{ "message": "Cannot delete: {N} issues use this category" }` — display this string verbatim as an inline per-row error below the row, styled `text-xs text-destructive`, matching `StatusSettings.vue:579–584`
- Do **not** add a `deleteCategory` call to `useCategories.ts` that silently swallows the 409; the page component must capture the error message string and store it in a `deleteErrors` ref keyed by category id — identical to `StatusSettings.vue:57`
- The `useCategories` module state is a singleton (refs declared outside the composable function at `useCategories.ts:4–6`); any category created or deleted on `CategorySettings.vue` will be reflected in `CreateIssueDialog.vue` without additional event plumbing — no Vuex/Pinia store needed
- Wrap delete action in `AlertDialog` (confirm before calling the API) — reuse `AlertDialog`, `AlertDialogTrigger`, `AlertDialogContent`, `AlertDialogAction`, `AlertDialogCancel` from `@/components/ui/alert-dialog`, exactly as `StatusSettings.vue:543–575` demonstrates; the 409 error appears *after* the confirmed delete attempt
- The `StoreCategoryRequest` trims whitespace and checks case-insensitively before slug generation; the frontend should surface 422 `errors.name[0]` on the create input — same pattern as `StatusSettings.vue:116–119`
- Display each category row's auto-generated `slug` in `font-mono text-xs text-muted-foreground` beneath the name — slug is returned in all three API responses (`index`, `store`) per `CategoryController::index` (`get(['id', 'name', 'slug'])`) and `::store` (`$category->only(['id', 'name', 'slug'])`)
- Empty state: when `categories.length === 0` render a `p.text-sm.text-muted-foreground` message ("No categories yet. Create one above.") — `StatusSettings.vue:399` shows the exact pattern; this covers first-run and post-delete-all scenarios

### Gotchas
- `useCategories.ts` declares `categories`, `isLoading`, and `isCreating` refs **at module scope** (outside the exported function), making them a shared singleton across all component instances — this is intentional for cross-component reactivity, but it means `deleteCategory` must mutate that same module-level `categories` ref (not a locally scoped copy)
- `Category` model has `UPDATED_AT = null` (`Category.php:26`) — the `categories` table has no `updated_at` column; do not attempt to use or display `updated_at` on any category row
- `StoreCategoryRequest` enforces uniqueness via a `LOWER(name)` raw query, not the Laravel `unique` rule — the error key is `errors.name[0]` but the message text is "The name has already been taken." (not the standard Laravel unique message); surface it as-is
- The deletion guard at `CategoryController::destroy:50` fires a `COUNT` query **before** the `DELETE` — the FK `restrictOnDelete()` on `issues.category_id` is a safety net, not the primary guard; the 409 will always arrive with the count in the message string, never as a DB exception from this controller
- PHPUnit tests must use `make:test --phpunit`, extend `Tests\TestCase`, and use `public function test_*` naming — per AGENTS.md Boost rules; do not use Pest `it()` / `describe()` syntax
