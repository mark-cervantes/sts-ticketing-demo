# Task 07.02: Show Issue Creator on Kanban Cards and Detail Sheet

## Business Requirements

### Problem
Users cannot see who created a ticket from the Kanban board. The issue creator's name only appears at the bottom of the detail sheet in small meta text ("Created by Alice Johnson"). On the Kanban cards, there is no indication of ownership — users can't tell which tickets are theirs vs. others'. This causes confusion about commenting permissions (users try to comment on others' tickets and find the input missing).

### User Story
As a user, I want to see who created each ticket at a glance on the Kanban board so that I can quickly identify my own tickets and understand the ownership context.

### Acceptance Criteria
- [ ] Kanban card shows the creator's name or initial avatar (e.g., "AJ" circle for Alice Johnson)
- [ ] Creator display is compact — does not crowd the card layout
- [ ] "My" tickets (owned by current user) have a subtle visual indicator (e.g., small "You" badge or outline)
- [ ] Detail sheet still shows "Created by {name}" in the meta section
- [ ] Visual verification via `make verify-visual`

### Business Rules
- The `user` relation is already eager-loaded and included in the API response (`user.id`, `user.name`)
- No additional API calls needed — data is already available on the card
- The avatar should use initials (no profile picture system exists)

### Out of Scope
- User profile pictures / avatar uploads
- Filtering by creator (separate feature)

## Technical Guidance

### Architecture Notes

- **Card avatar is already implemented.** `resources/js/components/kanban/IssueCard.vue` (lines 7–8, 28–35, 149–162) already imports `Avatar`/`AvatarFallback` from `resources/js/components/ui/avatar/`, computes `isOwnIssue` via `usePage<PageProps>().props.auth.user.id === issue.user.id`, computes `creatorInitials` (first char of first + last word), and renders the avatar in the footer row with `bg-primary text-primary-foreground` for own issues and `bg-muted text-muted-foreground` for others. The card acceptance criteria are already satisfied — verify before writing any new code.

- **Detail sheet "You" indicator is the remaining gap.** `resources/js/components/issues/IssueDetailSheet.vue` line 691 renders `<p>Created by {{ issue.user.name }}</p>` with no ownership distinction. This is the only line that needs to change: when `issue.user.id === page.props.auth.user.id`, append a `(you)` suffix or a small inline badge alongside the name, keeping it in the same meta `<p>` element so the existing layout is undisturbed.

- **`user.id` is available in the detail sheet.** `app/Http/Resources/IssueResource.php` line 55–58 always emits `user: { id, name }` on both list and show responses. `issue.user.id` is safe to compare against `page.props.auth.user.id` in `IssueDetailSheet.vue` without any backend change.

- **Auth user is available in the detail sheet via `usePage`.** `IssueCard.vue` already imports `usePage` from `@inertiajs/vue3` and types it with `PageProps` (`resources/js/types/index.ts`). The same import pattern applies to `IssueDetailSheet.vue` — no new composable or prop is needed.

- **`ui/avatar/` primitives are the correct primitives; no custom `UserAvatar` component is warranted at this scope.** `resources/js/components/ui/avatar/` provides `Avatar`, `AvatarFallback`, `AvatarBadge`, `AvatarGroup`, and `AvatarGroupCount`. The initials logic in `IssueCard.vue` (two lines of computed) is simple enough to inline. Extract a shared `UserAvatar` component only if a third callsite appears in this sprint; otherwise, the inlining in `IssueCard.vue` is the established pattern.

- **No color hash palette needed.** `IssueCard.vue` uses two states — `bg-primary` for own, `bg-muted` for others — which is the correct minimal design. Introducing a hash-to-palette scheme would break visual consistency with the existing card and would be out of scope per the task's "Out of Scope" section.

- **Avatar size constraint is already satisfied.** `Avatar` is rendered with `size="sm"` (line 150 of `IssueCard.vue`). The `avatarVariants` in `resources/js/components/ui/avatar/index.ts` drive the size; the `sm` variant is ≤24px. Do not introduce a custom inline `size` style — use the variant API.

- **No backend or API change required.** `IssueResource.php` already emits `user.id` and `user.name` in both list and show contexts. `Issue` type (`resources/js/types/issue.ts` lines 44–47, 99) already declares `user: IssueUser` with `id` and `name`. Zero schema, migration, or resource changes.

- **`make verify-visual` is the required gate.** The acceptance criterion calls it explicitly. The coder must run `make verify-visual` and keep the Playwright screenshot artifacts from `test-results/playwright/smoke/` before requesting review — per the Visual Verification Contract in `AGENTS.md`.

### Gotchas
- **Verify `IssueCard.vue` before editing it.** The card avatar, `isOwnIssue`, and `creatorInitials` are already wired (lines 28–35, 149–162). Rewriting working code to match the task description would be a regression. Confirm visually first, then scope any edits to gaps only.
- **`AvatarFallback` default class sets `bg-muted`.** `resources/js/components/ui/avatar/AvatarFallback.vue` line 17 hard-codes `bg-muted text-muted-foreground` in the base `cn()` call — a passed `:class` on `AvatarFallback` is appended, not substituted. For own-issue styling, `bg-primary text-primary-foreground` must override the defaults; confirm Tailwind specificity works as expected in the running app rather than assuming CSS cascade order.
- **Do not introduce `usePage` as a prop.** Pass `page.props.auth.user.id` via a `computed` local to the component, consistent with how `IssueCard.vue` handles it. Drilling `authUserId` as a prop would be a pattern inconsistency and would require touching parent components without task justification.
