# Task 07.03: Improve Comment Permissions UX — Explain Why Commenting is Disabled

## Business Requirements

### Problem
When viewing someone else's issue without comment permission, the comment input simply doesn't appear — there's no explanation. Users interpret this as a bug ("I cannot drop new comment on my own tickets") when they're actually looking at someone else's ticket. The current permission model is correct (owner can always comment; shared users need `comment` or `edit` permission), but the UI gives zero feedback about why the input is missing.

### User Story
As a user, I want to understand why I can't comment on a ticket so that I know whether to request access or whether it's expected behavior.

### Acceptance Criteria
- [ ] When `can_comment` is `false`, a disabled state message appears: "You don't have permission to comment on this issue. Ask the owner to share it with you."
- [ ] The message includes the owner's name (e.g., "Ask Alice Johnson to share it with you")
- [ ] When `can_comment` is `true`, the comment input appears as before (no change)
- [ ] On the user's own tickets, the comment input always appears (existing behavior preserved)
- [ ] Visual verification via `make verify-visual`

### Business Rules
- Permission model is unchanged — this is UX only, no backend changes
- Owner always has `can_comment: true`
- Shared users with `comment` or `edit` permission have `can_comment: true`
- Shared users with `view` permission have `can_comment: false`
- Non-shared users viewing public issues have `can_comment: false`

### Out of Scope
- Changing the permission model
- Inline "request access" button (would require notification system)
