# Task 08.02: Kanban-Centric Inline Status Management

## Business Requirements

### Problem
Status management lives on a separate Settings page, disconnected from where statuses are actually used — the Kanban board. Users should be able to manage statuses directly on the board for a more intuitive workflow.

### User Story
As a user, I want to manage statuses directly on the Kanban board so I can organize my workflow without leaving the dashboard.

### Acceptance Criteria
- [ ] **Lock/Unlock toggle** — a lock icon on the Kanban board header. When locked (default), the board behaves normally (drag issues between columns). When unlocked, enters "edit mode":
  - [ ] Columns can be **reordered** via drag-and-drop
  - [ ] Each column header shows a **delete button** (×)
  - [ ] An **"+ Add Status"** button appears at the end of the columns
- [ ] **Add status** — clicking "+" opens an inline form (name + color) that creates a new column immediately
- [ ] **Delete status column** — clicking × on a column with existing issues shows a modal:
  - "This status has N issues. Move them to:" + dropdown of other statuses + "Delete all issues" option
  - On confirm: migrate issues to chosen status (or delete), then delete the status
  - Columns with 0 issues delete immediately (with confirmation)
- [ ] **Column rename** — double-click column header to inline-edit the name (when unlocked)
- [ ] The default status column cannot be deleted (show tooltip explaining why)
- [ ] Lock icon returns to locked state after changes, re-enabling normal drag behavior
- [ ] Visual verification via `make verify-visual`

### Business Rules
- Lock state is local (per-session, not persisted) — default is locked
- Reorder persists via `PUT /api/statuses/{id}` with updated `sort_order`
- Delete with migration: `POST /api/statuses/{id}/migrate-and-delete` — new endpoint that moves issues then deletes
- Column drag-and-drop in edit mode is for columns only (issues not draggable in edit mode)
- The Settings → Statuses page remains as a fallback but the Kanban is the primary UX

### Out of Scope
- Column color editing inline (use Settings page for that)
- Status transition rules

## Technical Guidance

### Architecture Notes

- **`vue-draggable-plus` is already installed** (`KanbanColumn.vue` line 6: `import { VueDraggable } from 'vue-draggable-plus'`). Column drag-and-drop for edit mode must use the same library with a separate `VueDraggable` wrapping the column list in `KanbanBoard.vue` — do NOT introduce a second drag library. Use `group="kanban-columns"` (distinct from issues' `group="kanban"`) and `@end` to emit the reordered column array — ADR-003.

- **Edit mode is a boolean toggle on `KanbanBoard.vue`, not a new composable.** A local `ref<boolean> editMode` on `KanbanBoard.vue` is sufficient; lock state is explicitly per-session (task §Business Rules). No Pinia store. When `editMode` is true: column drag enabled, issue drag disabled (set `:disabled="editMode"` on the issue `VueDraggable` in `KanbanColumn.vue` via prop).

- **`POST /api/statuses/{status}/migrate-and-delete` is a new route on `StatusController`.** The existing `destroy` returns 409 when issues exist (`StatusController.php` lines 83–89). The new endpoint must accept `{ target_status_id: number } | { delete_issues: true }`, run inside `DB::transaction()`, bulk-update or bulk-delete issues, then call `$status->delete()`. Do NOT reuse the existing `destroy` — it must stay intact for zero-issue deletes.

- **Column reorder persists via `PUT /api/statuses/{id}` with `sort_order` only.** `UpdateStatusRequest` already validates `sort_order` (task §Business Rules). After drag-end emit, send one PATCH per reordered status with its new `sort_order` index — same pattern as issue `moveIssue` in `useKanbanBoard.ts` lines 212–293.

- **`useStatuses` cache must be busted after add/rename/delete/reorder.** `useStatuses.ts` line 54 exposes `refresh()` which resets `fetched = false` and re-fetches. Call `refresh()` after any mutation that changes the statuses list so `useKanbanBoard`'s `columns` computed (`useKanbanBoard.ts` line 127) sees the updated `statuses.value`. Without this, column titles stay stale.

- **Default-status column guard is frontend-only for the delete ×button.** The backend already enforces `is_default` at `StatusController.php` line 75 (returns 409). Frontend should hide the × button (not just disable it) for `column.statusId === defaultStatusId`, with a `Tooltip` from `resources/js/components/ui/tooltip/` explaining why.

- **Inline rename on double-click uses a conditional `<input>` swapping the `<h3>` in `KanbanColumn.vue`.** Use `contenteditable` only as fallback — an `<input>` with `:value="column.label"` and `@blur`/`@keydown.enter` to commit is easier to control. Commit via `PUT /api/statuses/{id}` with `{ name }`. The slug is auto-generated server-side on create; rename does NOT change the slug (slug is stable — `IssueStatus.php` line 38: slug only set on `creating` event).

- **`KanbanColumnDef` type in `resources/js/types/issue.ts` line 145 needs two new fields for edit mode**: `isDefault: boolean` (to hide × button) and optionally `issueCount: number` (to decide whether to show migration modal vs. immediate delete). Populate from `statuses.value` in the `columns` computed in `useKanbanBoard.ts`.

- **Migration modal uses `resources/js/components/ui/dialog/` (shadcn Dialog, already installed).** It needs a `<Select>` (already in `resources/js/components/ui/select/`) for target status, plus a "Delete all issues" option. The modal must close and call `useStatuses.refresh()` + `loadInitial()` on confirmed delete.

- **Tests: integration test covering the `migrate-and-delete` endpoint** — verify issues moved to target, verify original status deleted, verify 409 when target_status_id does not exist, verify 422 when neither key provided. Use PHPUnit 12 (`make:test --phpunit MigrateAndDeleteStatusTest`), extending `TestCase` with `use RefreshDatabase` — per ADR-009.

### Gotchas

- **Issue drag must be disabled in edit mode**, not just visually hidden. `VueDraggable` in `KanbanColumn.vue` line 77 accepts `:disabled="editMode"` prop — pass `editMode` down as a prop and bind it. Without this, users can simultaneously drag issues and columns, causing race conditions in the optimistic update logic (`moveIssue` in `useKanbanBoard.ts`).
- **`useStatuses` is a module-scoped singleton** (`useStatuses.ts` lines 8–12). Calling `refresh()` in one component updates `statuses.value` everywhere — this is intentional and correct, but the coder must call it after every status mutation to keep `KanbanBoard`'s columns computed in sync.
- **`migrate-and-delete` must run in a `DB::transaction()`**. A partial failure (issues moved, status delete fails) would leave orphaned issues pointing at a missing status — the board would render empty columns. Wrap the entire operation in `DB::transaction()` matching the existing pattern in `StatusController.php` lines 54–59.
- **Slug stability**: renaming a status changes `name` only, NOT `slug` (IssueStatus booted event fires on `creating`, not `updating`). The frontend uses `slug` as the drag-and-drop `data-status` key and as the column map key in `useKanbanBoard.ts` line 19. A slug change would break the column map; confirm the controller's `update` never touches `slug` after creation.
