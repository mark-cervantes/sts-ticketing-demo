# Task 08.03: AI Model Picker — Autocomplete Dropdown

## Business Requirements

### Problem
The AI Settings page model field is a plain text input. For OpenRouter (which has 300+ models), users must know the exact model ID to type. For presets, the model override is also a plain text input. Users need a searchable dropdown with autocomplete.

### User Story
As a user, I want to pick an AI model from a searchable dropdown so I don't have to remember exact model IDs.

### Acceptance Criteria
- [ ] **OpenRouter custom mode**: model field becomes a Combobox with:
  - Searchable text input with autocomplete
  - Dropdown showing matching models from `GET /api/settings/ai/models`
  - Each item shows: model name, context length, pricing (per-token cost)
  - "Popular" models section at top (existing curated list)
  - Free-text fallback: user can still type an arbitrary model ID not in the list
- [ ] **Preset mode model override**: same Combobox for the model override field
  - Pre-populated with the preset's default model
  - Dropdown shows models relevant to that provider
- [ ] **Non-OpenRouter custom mode** (Ollama, Custom): plain text input remains (no model list API available), but with placeholder suggestions
- [ ] Loading state while models are fetching
- [ ] The existing model search/filter logic in AiSettings.vue should be reused
- [ ] Visual verification via `make verify-visual`

### Business Rules
- Model list is cached server-side for 1 hour (already implemented)
- The dropdown is lazy-loaded — only fetched when the user focuses the model field or opens the dropdown
- Free-text entry is always allowed (user may know a model ID not in the cached list)

### Out of Scope
- Model pricing comparison view
- Model capability filtering (vision, function calling, etc.)

## Technical Guidance

### Architecture Notes

- **No shadcn-vue `Combobox` component exists in this project.** `resources/js/components/ui/` has `popover/`, `select/`, `input/`, and `dropdown-menu/` — no `combobox/`. The correct implementation pattern is `Popover` + `Input` (already installed) — the existing hand-rolled dropdown in `AiSettings.vue` lines 726–802 is already the right shape. Refactor it into a reusable `ModelCombobox.vue` component rather than adding a new shadcn dependency.

- **The `filteredModels` computed (`AiSettings.vue` lines 144–163) and `POPULAR_MODELS` constant (lines 74–87) must be moved into the new `ModelCombobox.vue`** as internal state/props so both the OpenRouter custom-mode field and the preset model override field can reuse the same logic. `selectedModel` / `presetModelOverride` become `v-model` bindings passed from `AiSettings.vue`.

- **Lazy-load models only on first focus/open** — task §Business Rules: "only fetched when user focuses the model field or opens the dropdown." The existing `loadOpenRouterModels()` call in `onCustomProviderChange` (`AiSettings.vue` line 278) triggers too eagerly on provider switch. In `ModelCombobox.vue`, trigger `loadOpenRouterModels()` on the `@focus` or `@open` event of the combobox trigger, not on mount.

- **Free-text fallback is required** — task §Acceptance Criteria: "user can still type an arbitrary model ID not in the list." The `<input>` inside the combobox must bind `v-model` to the parent's value ref directly so typing a value not in `filteredModels` still propagates. Do NOT lock the input to list-only selection.

- **Preset mode model override field** (`AiSettings.vue` lines 590–600) currently renders a plain `<Input>`. It must be replaced by `ModelCombobox.vue` only when the active preset's provider is `openrouter`; for other preset providers (Ollama, etc.) keep the plain text input — task §Acceptance Criteria ("Non-OpenRouter custom mode — plain text input remains").

- **`loadOpenRouterModels()` must be idempotent / guard against double-fetch.** The existing guard `if (modelsLoading.value) return` (`AiSettings.vue` line 234) is correct but only works within a session. Add an `openRouterModels.value.length > 0` early-return guard too so re-opening the combobox does not re-trigger the API call, consistent with the "cached server-side for 1 hour" rule (task §Business Rules).

- **Tests: one feature test** covering `GET /api/settings/ai/models` returns 200 with the expected shape `[{ id, name, context_length, pricing }]`, and one test for the 1-hour cache header. Use PHPUnit 12 (`make:test --phpunit AiModelListTest`) — per ADR-009. No new backend work needed; the endpoint already exists.

### Gotchas

- **`selectedModelLabel` computed (`AiSettings.vue` line 165) and `modelDropdownOpen` / `modelSearch` refs (lines 100–101) must be removed from `AiSettings.vue`** when extraction is complete; they live inside `ModelCombobox.vue`. Leaving them in the parent creates a ghost state that will diverge from what the child renders.
- **The existing dropdown does not close on outside click.** The `popover/` components use Radix Vue `PopoverContent` which closes on `Escape` and outside-click automatically. When building `ModelCombobox.vue` on top of `Popover`, this behavior is free — do not re-implement it with manual `v-if="modelDropdownOpen"` conditional rendering (`AiSettings.vue` line 739 pattern), which has no outside-click dismissal.
- **`formatPrice` and `formatContext` helpers (`AiSettings.vue` lines 407–418) belong in `ModelCombobox.vue`** after extraction. They are purely display concerns for model list items; keeping them in `AiSettings.vue` creates a hidden coupling between parent and child rendering.
