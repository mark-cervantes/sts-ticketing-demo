# Task 09.01: AI Chat — Backend (Streaming Endpoint + Saved Conversations)

## Business Requirements

### Problem
Users can see AI-generated summaries on issues, but they can't ask the AI follow-up questions. When a summary says "Carol confirmed the gateway timeout," there's no way to ask "What timeout value?" or "Which gateway endpoint?" without reading the entire comment thread. Users want a conversational interface where they can chat with AI about the issue — right from the summary section.

### User Story
As a user viewing an issue, I want to chat with AI about the issue so I can drill into details, and optionally save the conversation so my team can see it too.

### Key UX Concept: Session-First, Save-on-Demand

**Chat is ephemeral by default.** Messages live only in the user's browser session. Nothing is persisted until the user explicitly saves the conversation. This keeps the feature lightweight and low-friction — users can ask throwaway questions without cluttering the issue.

**Saved conversations are a separate entity** from the AI synthesis. They appear below the summary as their own section on the issue, visible to anyone who can view the issue. Saved conversations can be continued by anyone with access.

### Architecture

#### Data Model
```
issue_conversations
├── id (bigint, PK)
├── issue_id (FK → issues.id, ON DELETE CASCADE)
├── saved_by (FK → users.id — who hit "Save")
├── title (string, nullable — auto-generated or user-provided)
├── created_at
└── updated_at

issue_conversation_messages
├── id (bigint, PK)
├── conversation_id (FK → issue_conversations.id, ON DELETE CASCADE)
├── user_id (FK → users.id, nullable — null = AI response)
├── role (enum: 'user' | 'assistant')
├── content (text)
├── created_at
└── updated_at
```

#### API Endpoints
| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/issues/{issue}/chat` | Stateless — send message + chat history in body, get streamed AI response. Nothing persisted. |
| `POST` | `/api/issues/{issue}/conversations` | Save a conversation — accepts `{ title?, messages: [{role, content}] }`, persists to DB |
| `GET` | `/api/issues/{issue}/conversations` | List saved conversations for an issue (newest first) |
| `GET` | `/api/issues/{issue}/conversations/{conversation}` | Load a saved conversation's messages |
| `POST` | `/api/issues/{issue}/conversations/{conversation}/continue` | Append a new message to a saved conversation + get streamed AI response |

#### Streaming Strategy
The `POST /chat` endpoint is **stateless** — the frontend sends the full message history (session messages) in the request body. The backend builds the LLM prompt from issue context + provided messages, calls `{base_url}/chat/completions` with `stream: true`, and pipes tokens back as SSE.

For continuing saved conversations (`POST /conversations/{id}/continue`), the backend loads existing messages from DB, appends the new user message, streams the AI response, and saves both the user message and AI response to DB.

#### Context Window
The AI receives:
1. **System prompt** — configurable from DB (`chat_system_prompt` on `ai_settings`)
2. **Issue context** — title, description, category, priority, status
3. **Recent comments** — last N comments with author names and timestamps
4. **Chat history** — from request body (session chat) or DB (saved conversation)

### Business Rules
- Stateless chat: the backend does NOT store anything on `POST /chat` — purely ephemeral
- Saved conversations are a **separate section** on the issue — they do NOT replace or modify the AI synthesis
- Anyone who can view the issue can see saved conversations
- Anyone who can view the issue can continue a saved conversation
- The user who saves gets attributed (`saved_by`)
- Rate limit applies to both stateless chat and continue endpoints
- If no AI provider configured, `POST /chat` and `/continue` return 503

### Out of Scope
- Frontend UI (task 09.02)
- Conversation deletion or editing
- Multi-turn function calling / tool use
- File attachments in chat

### Architecture Notes

- **ADR-002 compliance — `ChatService` mirrors `TriageService`**: Create `app/Services/Ai/ChatService.php` injecting `Illuminate\Http\Client\Factory as HttpFactory`. All HTTP calls to the LLM endpoint (`POST {effective_base_url}/chat/completions`) must live exclusively in this service — never in controllers, Form Requests, or jobs. Read `app/Services/Ai/TriageService.php` for the constructor and `AiSetting::current()` call pattern. — *ADR-002, `app/Services/Ai/TriageService.php`*

- **Streaming: use the `IssueSseController` SSE shape exactly**: The chat streaming response must be a `Symfony\Component\HttpFoundation\StreamedResponse` (already used in `app/Http/Controllers/IssueSseController.php`) with headers `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `X-Accel-Buffering: no`. Each token chunk writes `data: {token}\n\n` and calls `flush()`. Terminal event is `data: [DONE]\n\n`. On mid-stream LLM failure, emit `data: {"error":"<message>"}\n\n` then `data: [DONE]\n\n` before breaking — never leave the stream open on error. — *`app/Http/Controllers/IssueSseController.php`*

- **LLM streaming differs from summary: `stream: true` + SSE chunked read**: Unlike `LlmDriver` which POSTs and parses a complete JSON response, `ChatService` must POST with `stream: true` and read the response body incrementally. Use `Http::withOptions(['stream' => true])` to get a streaming Guzzle response, then iterate `getBody()` line-by-line, parsing OpenAI SSE format (`data: {...}`) to extract `choices[0].delta.content` tokens. — *`app/Services/Summary/Drivers/LlmDriver.php` (non-streaming baseline), `SRS §EI-01`*

- **`AiSetting` readiness check before opening stream**: Before emitting any SSE bytes, call `AiSetting::current()` and check `effective_driver === 'rules'` or `api_key` empty. If the provider is not configured, return a plain `JsonResponse` with HTTP 503 — do NOT open a `StreamedResponse`. This avoids leaving the client with a hanging stream. — *`app/Models/AiSetting.php::getEffectiveDriverAttribute()`, task Business Rules*

- **`chat_system_prompt` column follows the `effective_*` accessor pattern**: The migration adds a nullable `chat_system_prompt` column to `ai_settings`. `AiSetting` gets a `getEffectiveChatSystemPromptAttribute()` accessor that returns `$this->chat_system_prompt ?? 'You are a helpful assistant...'` — the same DB-with-fallback pattern already used for `effective_system_prompt` and `effective_user_prompt`. Do NOT hardcode the default in the service. — *`app/Models/AiSetting.php::getEffectiveSystemPromptAttribute()`*

- **Context assembly cap — 20 messages, 10 comments**: `ChatService::buildMessages()` constructs the `messages` array: `[system]` + `[user context block]` + `[history]` + `[new user message]`. Cap `history` at the last 20 messages (slice from tail) and cap comments at the last 10 sorted by `created_at`. The issue context block is a single `user`-role message containing title, description, category, priority, status, and formatted comments — modelled after the prompt construction in `LlmDriver::generate()`. — *`app/Services/Summary/Drivers/LlmDriver.php` lines 43–59*

- **Authorization: reuse `IssuePolicy@view` — no new policy**: All five endpoints call `$this->authorize('view', $issue)`. `IssuePolicy::view()` already handles owner / share / public resolution per `SRS §BR-04`. No new policy method is required. — *`app/Policies/IssuePolicy.php::view()`, `SRS §BR-04`*

- **Rate limiting: named limiter, not inline throttle middleware**: Define a named limiter in `AppServiceProvider` (or a dedicated `RateLimitServiceProvider`) using `RateLimiter::for('chat', ...)` keyed on `auth()->id() . ':' . $issue->id`, limit 10 per hour. Apply via `->middleware('throttle:chat')` on the two streaming routes (`/chat` and `/conversations/{conversation}/continue`). This avoids duplicating the limit logic in both controllers and makes it testable in isolation. — *`routes/api.php` (existing `throttle` usage pattern), task Business Rules*

- **`/conversations` nested routes follow `issues.shares` shallow resource pattern**: Register the five endpoints in `routes/api.php` as an explicit `Route::middleware('auth')->group()` — not as `apiResource` because the continue endpoint is not a standard REST verb. Place them after the existing AI group so route ordering does not shadow `issues/{issue}` wildcard. — *`routes/api.php` lines 33–47*

- **`continue` endpoint: save user message first, then stream**: On `POST /conversations/{conversation}/continue`, persist the user's `IssueConversationMessage` row (role=user, user_id=auth id) to DB **before** opening the SSE stream. After the stream completes (all tokens received), persist the assistant's message (role=assistant, user_id=null). This ensures the user message is durable even if the stream errors mid-way. — *task Architecture §Streaming Strategy*

- **Bulk insert for `save` endpoint — avoid N+1 inserts**: `POST /conversations` receives potentially many messages. Use `IssueConversationMessage::insert([...])` (mass insert, no model events) rather than looping `create()`. Wrap the conversation creation and message insert in `DB::transaction()` to keep them atomic. — *`SRS §NFR-01` (no N+1 on any endpoint)*

### Gotchas

- **`Http::fake()` does not intercept streaming responses by default**: When testing `ChatService` with `Http::fake()`, the fake response must be constructed so that `getBody()` iteration works. Use `Http::fake(['*/chat/completions' => Http::response("data: {\"choices\":[{\"delta\":{\"content\":\"hello\"}}]}\n\ndata: [DONE]\n\n")])` and structure the service to call `$response->body()` line-by-line in tests (or inject a `StreamedResponseInterface` seam). Alternatively, extract the context-building into a separate testable method and test the streaming path with a true integration mock. — *`app/Services/Summary/Drivers/LlmDriver.php` (Http::fake pattern), `SRS §8.5`*

- **`ob_end_clean()` must NOT be called inside `StreamedResponse`**: `IssueSseController` documents this explicitly — Laravel's test harness wraps `sendContent()` in `ob_start()` and calling `ob_end_clean()` destroys the capture buffer. `ChatService`'s streaming callback must only call `flush()` between chunks, matching the pattern in `IssueSseController`. — *`app/Http/Controllers/IssueSseController.php` lines 17–23*

- **`role` column on `issue_conversation_messages` must be a DB enum or `varchar` with a cast**: If using a PHP-backed `Enum` (e.g., `App\Enums\ChatRole`), add it to `$casts` on `IssueConversationMessage` and use `->type('enum', ['user', 'assistant'])` or a `string` column with validation. Mismatched enum casts silently produce `null` on retrieval — follow the existing `Priority` and `Visibility` enum cast pattern in `app/Models/Issue.php`. — *`app/Models/AiSetting.php` (cast pattern reference)*

- **`saved_by` is `constrained()` on `users`, NOT `nullable`**: The task data model shows `saved_by` as a required FK (the user who hit "Save"). However, for `continue`, any viewer can continue — `saved_by` refers to the original saver, not the continuer. Do not confuse `saved_by` on `issue_conversations` with `user_id` on `issue_conversation_messages`. — *task Architecture §Data Model*

- **Rate limiter key must include issue ID, not just user ID**: A user chatting on issue A must not consume quota against issue B. Key the limiter as `"chat:{$user->id}:{$issue->id}"` so quotas are per-user-per-issue. — *task Business Rules: "10 messages per user per issue per hour"*

- **`GET /conversations` list query needs `withCount('messages')` to avoid N+1**: The list response includes message count per conversation. Use `IssueConversation::where('issue_id', $issue->id)->with('savedBy')->withCount('messages')->latest()->get()` — a single query. Do not loop and call `$conversation->messages()->count()`. — *`SRS §NFR-01`*
