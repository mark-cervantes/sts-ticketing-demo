# Task 10.02: Resolved Column — Auto-Collapse, Pagination, Archive Strategy

## Problem
Resolved tickets pile up indefinitely in the "Resolved" column. After a few weeks of use, this column becomes an unreadable wall of cards that slows down the board and makes it impossible to find anything.

## Solution

### Collapsed-by-default Resolved Column
The Resolved column renders **collapsed** by default, showing only the count and a "Show" toggle:
```
┌─────────────────────────┐
│ ✓ Resolved        (47)  │
│    [Show resolved ▾]    │
└─────────────────────────┘
```
Clicking expands it to show the most recent N cards with "Load more" pagination.

### Pagination
All columns should support pagination (they may already via `loadMore`), but Resolved especially:
- Show the most recent 10 resolved tickets by default
- "Load more" button at the bottom loads the next 10
- Total count always visible in the column header

### Time-based Grouping (Optional Enhancement)
Within the expanded Resolved column, group by time:
- "Today" / "This week" / "This month" / "Older"
- Only show "Today" and "This week" expanded, older collapsed

### Acceptance Criteria
- [ ] Resolved column is collapsed by default on page load
- [ ] Expand/collapse toggle with smooth animation
- [ ] Collapsed state shows issue count badge
- [ ] Expanded state shows most recent 10 tickets with "Load more"
- [ ] Column state (expanded/collapsed) persists in localStorage
- [ ] Non-resolved columns are unaffected
- [ ] `make verify-visual` passes

### Architecture Notes
- **Resolved column identification** — `IssueStatus` has no `is_resolved` flag (confirmed from migration and `app/Models/IssueStatus.php`). Identify the resolved column exclusively by matching `column.status === 'resolved'` in `KanbanBoard.vue`. Do NOT add a new DB column for this; the slug `'resolved'` is the stable, seeded identifier. If the slug must become configurable in a future sprint, that's a separate ADR.
- **`KanbanColumnDef` needs `isResolved` and `totalCount`** — add both fields to the interface in `resources/js/types/issue.ts`. `isResolved` is derived from `status.slug === 'resolved'` in the `columns` computed of `resources/js/composables/useKanbanBoard.ts`. `totalCount` should be populated from `result.meta.total` on initial load (already in the `PaginatedResponse` shape at `types/issue.ts` lines 131–139), so the collapsed header can show the real count rather than the loaded-page count (`issues.length`).
- **Per-column `per_page` on initial load** — `PER_PAGE = 15` is a single module constant in `useKanbanBoard.ts` (line 13). To give the Resolved column a smaller initial page (10), pass `perPage` as a parameter to `fetchColumn` rather than reading the constant directly. `loadInitial` already loops over `statuses.value.map(...)` (lines 163–166) — use `s.slug === 'resolved' ? 10 : PER_PAGE` there. Do not change `PER_PAGE` globally.
- **`collapsed` prop and localStorage read-on-mount** — add a `defaultCollapsed` boolean prop to `KanbanColumn.vue` (alongside the existing `KanbanColumnProps` interface at lines 46–52). Internal collapsed state should be a `ref<boolean>` initialized from `localStorage.getItem('kanban-col-${column.status}-collapsed')`, falling back to `defaultCollapsed`. Write back on every toggle. Use `onMounted` for the read so SSR-safe hydration is possible in future.
- **Expand/collapse `<Transition>`** — Vue's built-in `<Transition name="kanban-expand">` wrapping the card list is sufficient; Tailwind's `transition-all` is already on the column root (`KanbanColumn.vue` line 258). A CSS height transition from `max-height: 0` to `max-height: var(--radix-collapsible-content-height)` is an option, but the simpler path is a `v-show` toggle on the `VueDraggable` and the "Load more" block, wrapped in `<Transition>`.
- **Collapsed drop-target — the critical drag-drop constraint** — the `VueDraggable` element at `KanbanColumn.vue` line 383 is the SortableJS drop zone. If `v-show="!collapsed"` is applied to the `VueDraggable`, SortableJS can still receive drops to a `display:none` element because SortableJS fires the `end` event on the source and reads `evt.to.dataset.status`. Verify this works by dragging a card onto the collapsed header during manual QA. If SortableJS drops fail when the target list is hidden, the fallback is to keep the `VueDraggable` rendered but zero-height (`max-h-0 overflow-hidden`) instead of `v-show`. Do not use `v-if` on `VueDraggable` — that destroys the SortableJS group registration and breaks cross-column drag entirely.
- **`KanbanBoard.vue` wires `defaultCollapsed`** — the `v-for` loop over `localColumns` at line 210 is where `defaultCollapsed` is passed. Use `:default-collapsed="col.isResolved"` — no other plumbing needed. The board owns the "which column starts collapsed" policy; the column owns the "am I currently collapsed" state.
- **Count badge accuracy when collapsed** — the existing count badge in `KanbanColumn.vue` (line 302) displays `column.issues.length`, which is the loaded page count, not the server total. For the collapsed Resolved column this is misleading (shows `10`, not `47`). Render `column.totalCount` instead once that field is wired. Other columns can continue using `issues.length` since they load fully for now.
- **`useKanbanBoard.ts` singleton scope** — `columnMap`, `paginationState`, and `columnLoading` are all module-level (not per-composable-call). Any per-column `perPage` override must be tracked per-slug as well (e.g., `const columnPerPage = ref<Record<string, number>>({})`), or simply hardcode the slug check inline in `loadInitial` — the simpler approach is preferred given SPEC §4.2 does not describe configurable per-column page sizes.

### Gotchas
- **`v-if` on `VueDraggable` will break cross-column drag** — SortableJS registers the group on mount; unmounting the element removes it from the `'kanban'` group. Use `v-show` or CSS height clamping only. This is the single highest-risk implementation mistake for this task.
- **`column.issueCount` vs `column.totalCount`** — `issueCount` in `KanbanColumnDef` (line 144 of `useKanbanBoard.ts`) is already `issues.length` (loaded count). A new `totalCount` field is needed; do not reuse `issueCount` by redefining its meaning — that would silently break the migration-dialog copy in `KanbanColumn.vue` line 452 which reads `column.issueCount`.
- **localStorage key collision** — if the user renames the "Resolved" status, the slug changes (slug is auto-generated on `creating` in `IssueStatus::booted()`). The localStorage key `kanban-col-resolved-collapsed` becomes orphaned and the column resets to `defaultCollapsed`. This is acceptable behaviour for this sprint; document it as a known limitation.
- **`PER_PAGE` is module-scope** — changing the constant affects all columns globally and all in-flight `loadMore` calls. Parameterize `fetchColumn` rather than mutating the constant.
- **`meta.total` availability** — `PaginatedResponse.meta.total` is already typed in `resources/js/types/issue.ts` (line 138) and Laravel's paginator always includes it. No backend change needed.
