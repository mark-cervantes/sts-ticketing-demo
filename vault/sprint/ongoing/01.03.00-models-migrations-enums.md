## What To Build

Create all Eloquent models, migrations, and PHP enums for the data model defined in SPEC §4.

### Enums (app/Enums/)
- `Status`: open, in_progress, resolved (backed string enum)
- `Priority`: low, medium, high, critical (backed string enum, with `needsAttention(): bool`)
- `Visibility`: private, public (backed string enum)
- `Permission`: view, comment, edit (backed string enum, with `canComment(): bool`, `canEdit(): bool` — ladderized)

### Models + Migrations
- **User** — extend existing Breeze user (no changes needed unless adding fields)
- **Issue** — all fields from SPEC §4.2. Relationships: belongsTo(User), belongsTo(Category), hasMany(Comment), hasMany(IssueShare). SoftDeletes. Casts for enums.
- **Comment** — SPEC §4.4. Relationships: belongsTo(Issue), belongsTo(User).
- **Category** — SPEC §4.3. Auto-generate slug from name in model boot. Relationship: hasMany(Issue).
- **IssueShare** — SPEC §4.5. Relationships: belongsTo(Issue), belongsTo(User). Unique constraint on (issue_id, user_id).

### Indexes
Per SPEC §4.6:
```
issues:       [user_id], [category_id], [status], [priority], [visibility]
              composite: [status, priority], [user_id, status]
comments:     [issue_id], [user_id]
issue_shares: unique [issue_id, user_id]
categories:   unique [slug]
```

### Model Accessors / Methods
- `Issue::needsAttention()`: computed from priority + deadline (see ADR-005)
- `Category`: auto slug generation on creating event

## Technical Guidance

### 1. Migration Order & Column Definitions

Run migrations in strict FK dependency order. **No user migration needed** — existing Breeze `users` table (SPEC §4.1) already has `id`, `name`, `email`, `password`, `timestamps`.

**Migration 1 — categories** (no FK dependencies):
```bash
./vendor/bin/sail artisan make:migration create_categories_table --create=categories
```
Columns per SPEC §4.3:
- `$table->id()` — bigint PK
- `$table->string('name')->unique()` — case-insensitive uniqueness enforced at app layer (FR-08), DB unique handles exact match
- `$table->string('slug')->unique()` — auto-generated, unique index satisfies §4.6
- `$table->timestamp('created_at')->nullable()` — no `updated_at` per SPEC §4.3 (only `created_at`)

⚠️ Do NOT use `$table->timestamps()` here — SPEC §4.3 only specifies `created_at`, not `updated_at`. Use a single `$table->timestamp('created_at')->nullable()` column instead.

**Migration 2 — issues** (FK → users, categories):
```bash
./vendor/bin/sail artisan make:migration create_issues_table --create=issues
```
Columns per SPEC §4.2:
- `$table->id()`
- `$table->foreignId('user_id')->constrained()->onDelete('cascade')->onUpdate('cascade')` — owner deleted → issues cascade (user account deletion removes their data)
- `$table->string('title')` — max 255 default, required enforced in Form Request
- `$table->text('description')` — required, used for summary generation
- `$table->string('priority')` — **string column, NOT native enum** (see §7 Pitfalls); cast via PHP enum `Priority`; default not set in migration (required input per FR-02)
- `$table->foreignId('category_id')->constrained()->onDelete('restrict')->onUpdate('cascade')` — **restrict** enforces BR-05 (cannot delete category with issues) at DB level
- `$table->string('status')->default('open')` — BR-01 default
- `$table->string('visibility')->default('private')` — BR-01 default
- `$table->text('summary')->nullable()` — null until job completes
- `$table->text('suggested_next_action')->nullable()`
- `$table->string('summary_status')->default('pending')` — BR-01 default (note: `summary_status` is NOT in our 4 enum classes — it only needs `$casts` if we add a `SummaryStatus` enum later; for now string is sufficient since no methods needed on it)
- `$table->boolean('needs_attention')->default(false)` — computed on save
- `$table->timestamp('deadline_at')->nullable()` — optional user-set deadline
- `$table->softDeletes()` — adds nullable `deleted_at`
- `$table->timestamps()` — `created_at` + `updated_at`

Single indexes per §4.6 (FKs `user_id` and `category_id` already get automatic indexes from `foreignId()->constrained()`):
- `$table->index('status')`
- `$table->index('priority')`
- `$table->index('visibility')`

Composite indexes per §4.6:
- `$table->index(['status', 'priority'])` — status leading, correct for "all open issues filtered by priority" query pattern
- `$table->index(['user_id', 'status'])` — user leading, correct for "my issues by status" pattern

**Migration 3 — comments** (FK → issues, users):
```bash
./vendor/bin/sail artisan make:migration create_comments_table --create=comments
```
Columns per SPEC §4.4:
- `$table->id()`
- `$table->foreignId('issue_id')->constrained()->onDelete('cascade')->onUpdate('cascade')` — issue deleted (hard or forced) → comments cascade; soft-deleted issues retain comments since soft deletes don't fire FK cascades
- `$table->foreignId('user_id')->constrained()->onDelete('cascade')->onUpdate('cascade')` — user deleted → their comments cascade
- `$table->text('body')` — required, non-empty enforced at app layer
- `$table->timestamp('created_at')->nullable()` — no `updated_at` per SPEC §4.4

⚠️ Same as categories: SPEC §4.4 only lists `created_at`. Use `$table->timestamp('created_at')->nullable()`, NOT `$table->timestamps()`.

Indexes per §4.6:
- `$table->index('issue_id')` — already from `foreignId`
- `$table->index('user_id')` — already from `foreignId`

Both FK indexes are auto-created by `foreignId()->constrained()`, so no additional explicit index calls needed for comments.

**Migration 4 — issue_shares** (FK → issues, users):
```bash
./vendor/bin/sail artisan make:migration create_issue_shares_table --create=issue_shares
```
Columns per SPEC §4.5:
- `$table->id()`
- `$table->foreignId('issue_id')->constrained()->onDelete('cascade')->onUpdate('cascade')` — issue deleted → shares cascade
- `$table->foreignId('user_id')->constrained()->onDelete('cascade')->onUpdate('cascade')` — user deleted → shares cascade
- `$table->string('permission')` — string column, cast to `Permission` enum
- `$table->timestamp('created_at')->nullable()` — SPEC §4.5 only lists `created_at`

Unique constraint per §4.6 — **must be a real DB unique index**, not just model validation:
- `$table->unique(['issue_id', 'user_id'])` — one share per user per issue; upsert logic in service layer (ADR-007)

### 2. Enum Design (app/Enums/)

All four enums: backed string enums per PHP 8.4, explicit return types, PHPDoc blocks.

**`Status`** — cases: `Open = 'open'`, `InProgress = 'in_progress'`, `Resolved = 'resolved'`. No methods needed.

**`Priority`** — cases: `Low = 'low'`, `Medium = 'medium'`, `High = 'high'`, `Critical = 'critical'`.
- `needsAttention(): bool` — returns `true` for `High` and `Critical` only (the priority component of BR-03; deadline component lives on Issue model per ADR-005).
- Implementation: `return $this === self::High || $this === self::Critical;`

**`Visibility`** — cases: `Private = 'private'`, `Public = 'public'`. No methods needed.

**`Permission`** — cases: `View = 'view'`, `Comment = 'comment'`, `Edit = 'edit'`.
- Ladder: `view < comment < edit` (SPEC §3.2).
- `canComment(): bool` — true for `Comment` and `Edit`: `return $this === self::Comment || $this === self::Edit;`
- `canEdit(): bool` — true for `Edit` only: `return $this === self::Edit;`
- Explicit equality comparisons are fine here — three cases don't warrant a numeric ordering system. The explicitness is self-documenting.

⚠️ **Discrepancy flag:** SPEC §6.5 says the share API only accepts `view` or `edit` (not `comment`). ADR-007 also only mentions `view | edit`. But SPEC §4.5 schema and §3.2 authorization define the full three-value ladder including `comment`. **Recommendation: implement all three in the enum and schema (SPEC §4.5 is the column definition ground truth), but note that the share API endpoint may restrict input to `view` and `edit` only via Form Request validation.** Resolve with PM before building the share endpoint (later task).

### 3. Model Design (app/Models/)

Generate with factories:
```bash
./vendor/bin/sail artisan make:model Issue --factory
./vendor/bin/sail artisan make:model Comment --factory
./vendor/bin/sail artisan make:model Category --factory
./vendor/bin/sail artisan make:model IssueShare --factory
```

**`Issue`** model:
- Traits: `HasFactory`, `SoftDeletes`
- Use `#[Fillable]` attribute (matches existing `User` model convention): `#[Fillable(['title', 'description', 'priority', 'category_id', 'status', 'visibility', 'summary', 'suggested_next_action', 'summary_status', 'needs_attention', 'deadline_at'])]` — whitelist matching FR-02 input + server-set fields. Do NOT use `$guarded = []`.
- `casts()` method (Laravel 13 style, matching existing `User::casts()`):
  ```php
  protected function casts(): array
  {
      return [
          'priority' => Priority::class,
          'status' => Status::class,
          'visibility' => Visibility::class,
          'deadline_at' => 'immutable_datetime',
          'needs_attention' => 'boolean',
      ];
  }
  ```
  Note: `summary_status` left as string intentionally — no enum class for it in this task (no methods needed; add `SummaryStatus` enum later if methods arise).
- Relationships: `user(): BelongsTo`, `category(): BelongsTo`, `comments(): HasMany`, `shares(): HasMany` (to `IssueShare`).
- **`computeNeedsAttention()` static helper** per ADR-005 + BR-03:
  ```php
  public static function computeNeedsAttention(Priority $priority, ?CarbonImmutable $deadlineAt): bool
  {
      if ($priority->needsAttention()) {
          return true;
      }
      if ($deadlineAt !== null) {
          $threshold = config('issues.attention_threshold_minutes', 60);
          return $deadlineAt->lte(now()->addMinutes($threshold));
      }
      return false;
  }
  ```
- **Boot `saving` event** to auto-set `needs_attention`:
  ```php
  protected static function booted(): void
  {
      static::saving(function (Issue $issue): void {
          $issue->needs_attention = self::computeNeedsAttention(
              $issue->priority,
              $issue->deadline_at,
          );
      });
  }
  ```
  This fires on both create and update, satisfying BR-03's "recomputed on issue create/update" requirement.

**`Comment`** model:
- `#[Fillable(['body'])]` — only user-supplied field; `issue_id` and `user_id` set in controller/service.
- Relationships: `issue(): BelongsTo`, `user(): BelongsTo`.
- No `$table->timestamps()` in migration, so set `public $timestamps = false;` and add `const CREATED_AT = 'created_at'; const UPDATED_AT = null;` — OR simply set `const UPDATED_AT = null;` and keep `$timestamps = true` which will only write `created_at`. The cleaner approach: `$timestamps = true` with `const UPDATED_AT = null;`.

**`Category`** model:
- `#[Fillable(['name', 'slug'])]`
- Relationship: `issues(): HasMany`
- No `updated_at` — same pattern as Comment: `$timestamps = true` with `const UPDATED_AT = null;`
- **Slug auto-generation** on `creating` event in `booted()`:
  ```php
  protected static function booted(): void
  {
      static::creating(function (Category $category): void {
          $category->slug = static::generateUniqueSlug($category->name);
      });
  }
  ```
- **Collision-handling algorithm** for `generateUniqueSlug(string $name): string`:
  1. Base slug = `Str::slug($name)` (e.g., "Bug Reports" → "bug-reports")
  2. Query `categories` for `slug LIKE '{base}%'` to find existing matches
  3. If base doesn't exist → use base
  4. If base exists → find highest numeric suffix, increment: `bug-reports-2`, `bug-reports-3`, etc.
  5. Edge case: first collision is `-2` (not `-1`), matching conventional behavior

**`IssueShare`** model:
- `#[Fillable(['issue_id', 'user_id', 'permission'])]`
- Relationships: `issue(): BelongsTo`, `user(): BelongsTo`
- `casts()`: `'permission' => Permission::class`
- No `updated_at` — same `const UPDATED_AT = null;` pattern
- The unique constraint `(issue_id, user_id)` is enforced at DB level (migration). Service layer will handle upsert logic (ADR-007) — not needed in this task.

**Soft-delete cascade note:** Laravel `SoftDeletes` sets `deleted_at` without firing DB-level `ON DELETE CASCADE`. When an issue is soft-deleted, its comments and shares **remain intact** in the DB — this is correct per ADR-007 (comments preserved for audit trail). Hard-delete via `forceDelete()` triggers the DB cascade and removes comments + shares. No additional cascade logic needed in models.

### 7. Pitfalls

- **Postgres native `enum` type is an ALTER-headache.** Postgres `CREATE TYPE ... AS ENUM` requires `ALTER TYPE ... ADD VALUE` to add new values later — which can't run inside a transaction. Use `string` columns with PHP enum `$casts` instead. Laravel handles hydration/dehydration transparently. The `$table->enum()` blueprint method creates a `CHECK` constraint on some DBs but a native type on Postgres — **avoid it entirely**; use `$table->string()`.
- **`SoftDeletes` + unique indexes:** If you ever need uniqueness on a soft-deleted column (e.g., unique title per user), you'd need a partial unique index (`WHERE deleted_at IS NULL`). Not needed now (no unique constraints on issue columns), but flag for future.
- **`category_id` must be `NOT NULL` + `onDelete('restrict')`** — SPEC §4.2 says required, BR-05 says categories with issues can't be deleted. `foreignId()->constrained()` is non-nullable by default. The `restrict` enforces at DB level as a safety net beyond the app-layer guard.
- **`deadline_at` is nullable** — tests must cover both branches (`null` → only priority signal, non-null → check threshold). Ensure factory has `deadline_at => null` as default with a state for `withDeadline`.
- **`casts()` method form** — Laravel 13 uses the `casts()` method (not `$casts` property). Existing `User` model already uses this pattern. Follow it consistently.
- **Composite index column order** — `[status, priority]` is correct: status has ~3 values (lower cardinality but it's the leading filter in "show all open, then sort by priority"). This matches the dominant query pattern per FR-03.
- **`IssueShare` unique constraint must be a DB-level unique index** (`$table->unique(['issue_id', 'user_id'])`), not just model-level validation. The upsert for re-sharing (ADR-007) relies on this constraint for `updateOrCreate` correctness.
- **`comments` and `issue_shares` have no `updated_at`** per SPEC §4.4 and §4.5 — set `const UPDATED_AT = null;` on those models. Using `$table->timestamps()` will create both columns; use `$table->timestamp('created_at')->nullable()` instead in migrations.
- **Factories are required** for all 4 models — `IssueFactory`, `CommentFactory`, `CategoryFactory`, `IssueShareFactory`. The `--factory` flag on `make:model` generates the stub. Populate with realistic defaults matching SPEC §4 columns. `IssueFactory` should default to `Priority::Low`, `Status::Open`, `Visibility::Private`, `needs_attention => false`, `deadline_at => null`, with states like `highPriority()`, `withDeadline()`, `public()`.
