## What To Build

Customize Breeze auth and implement Laravel Policies for the ownership + sharing authorization model.

### Auth Customization
- Breeze already provides login/register — verify it works with our User model
- Ensure authenticated user data is shared via Inertia (HandleInertiaRequests middleware)
- No roles — just authenticated vs. not

### Policies (app/Policies/)

**IssuePolicy:**
- `viewAny()`: any authenticated user (sees own + shared + public in list)
- `view($user, $issue)`: owner OR shared(any level) OR public
- `create($user)`: any authenticated user
- `update($user, $issue)`: owner OR shared(edit)
- `comment($user, $issue)`: owner OR shared(comment or edit)
- `delete($user, $issue)`: owner only
- `share($user, $issue)`: owner only
- `restore($user, $issue)`: owner only

**CommentPolicy:**
- `create($user, $issue)`: delegates to IssuePolicy::comment

### Policy Registration
- Register in AuthServiceProvider (or auto-discovery)
- Apply in controllers via `$this->authorize()` or middleware

### Scopes (on Issue model)
- `scopeAccessibleBy($query, $user)`: returns issues where user is owner OR shared OR public
- Used in list endpoint to filter correctly

### 1. `IssuePolicy` design (`app/Policies/IssuePolicy.php`)

Generate with `sail artisan make:policy IssuePolicy --model=Issue --no-interaction`.

Methods and their access rules per ADR-004 §Access Resolution + ADR-007 §Access Resolution (ordered):

| Method | Rule | Ladder method used |
|---|---|---|
| `viewAny(User $user)` | Always `true` — any authenticated user; the list is filtered by `scopeAccessibleBy`, not the policy | — |
| `view(User $user, Issue $issue)` | Owner → true; any share row exists → true; `Visibility::Public` → true; else false | — (any share grants view) |
| `create(User $user)` | Always `true` — any authenticated user can create | — |
| `update(User $user, Issue $issue)` | Owner → true; share exists AND `$share->permission->canEdit()` → true; else false | `Permission::canEdit()` |
| `comment(User $user, Issue $issue)` | Owner → true; share exists AND `$share->permission->canComment()` → true; else false | `Permission::canComment()` |
| `delete(User $user, Issue $issue)` | Owner only | — |
| `share(User $user, Issue $issue)` | Owner only | — |
| `restore(User $user, Issue $issue)` | Owner only (for soft-deleted issues) | — |

Pseudocode for the two non-trivial methods:

```php
// view — per-issue check; fine for detail endpoints, NOT for list loops
public function view(User $user, Issue $issue): bool
{
    if ($issue->user_id === $user->id) return true;            // owner
    $share = $issue->shares()->where('user_id', $user->id)->first();
    if ($share) return true;                                    // any share grants view
    if ($issue->visibility === Visibility::Public) return true; // public
    return false;
}

// update — share must have edit permission
public function update(User $user, Issue $issue): bool
{
    if ($issue->user_id === $user->id) return true;
    $share = $issue->shares()->where('user_id', $user->id)->first();
    return $share !== null && $share->permission->canEdit();
}
```

Extract the share lookup into a `private function getShare(Issue $issue, User $user): ?IssueShare` to avoid repetition across `view`/`update`/`comment`.

**`comment` is non-standard** — not in Laravel's default policy ability list. It must be invoked explicitly via `$user->can('comment', $issue)`. `authorizeResource()` will NOT auto-map it. Note this for Sprint 02 controller work.

### 2. `CommentPolicy` design (`app/Policies/CommentPolicy.php`)

Generate with `sail artisan make:policy CommentPolicy --no-interaction` (no `--model` — the `create` method takes `Issue`, not `Comment`).

- `create(User $user, Issue $issue): bool` — delegate to `IssuePolicy::comment` via `Gate::allows('comment', $issue)` or `$user->can('comment', $issue)`. Keep one source of truth; do NOT reimplement the ladder check.
- No other methods (delete/update on Comment are out of scope per task brief).

### 3. `Issue` model scope (`app/Models/Issue.php`)

Add `scopeAccessibleBy(Builder $query, User $user): Builder` — SPEC §3.2, ADR-007 §Access Resolution:

```php
use Illuminate\Database\Eloquent\Builder;

public function scopeAccessibleBy(Builder $query, User $user): Builder
{
    return $query->where(function (Builder $q) use ($user) {
        $q->where('user_id', $user->id)
          ->orWhereHas('shares', fn (Builder $sq) => $sq->where('user_id', $user->id))
          ->orWhere('visibility', Visibility::Public->value);
    });
}
```

- `orWhereHas` generates a correlated EXISTS subquery — fine for Sprint 1 demo scale. If N grows, switch to UNION ALL.
- SoftDeletes already applies `whereNull('deleted_at')` on the query before this scope runs; no extra check needed.
- Use this scope in list endpoints — **never** call `view()` policy per-row in a loop.

### 4. Policy registration

- Laravel 13 auto-discovers policies in `app/Policies/` by naming convention (`Issue` → `IssuePolicy`). No manual registration in `AuthServiceProvider` needed.
- The custom `comment` ability lives as a method on `IssuePolicy` — `$user->can('comment', $issue)` resolves through policy auto-discovery. No `Gate::define()` needed.
- `CommentPolicy` does NOT auto-resolve to `Comment` model for the `create` ability because its signature takes `Issue` not `Comment`. The controller must call `$user->can('create', [Comment::class, $issue])` or `Gate::allows('comment', $issue)` directly. Leave this wiring for the comment CRUD task.

### 8. Pitfalls

- `Permission::canComment()` returns true for `Comment` AND `Edit` (ladder). `canEdit()` returns true for `Edit` only. Test cases must reflect this asymmetry (`app/Enums/Permission.php:20-32`).
- ADR-007 §Decision shows only 2 permission values (`view|edit`); SPEC §4.5 has 3 (`view|comment|edit`). SPEC is the column ground truth and `Permission` enum already has all 3. Policies handle all 3 cases.
- Soft-deleted issues: route-model-binding excludes them by default; the policy itself doesn't need a `deleted_at` check. Leave 404-via-binding tests for the controller/CRUD task.
- `view(User, Issue)` hits the DB per call — acceptable for single-issue endpoints. For list endpoints, use `scopeAccessibleBy` instead. Never loop `view()` over a collection.
- `comment` is not a Laravel default policy method — `authorizeResource()` won't auto-map it. Controllers must call `$this->authorize('comment', $issue)` or `$user->can('comment', $issue)` explicitly.
- Future controller note: `$this->authorize('viewAny', Issue::class)` (class, not instance) vs `$this->authorize('view', $issue)` (instance). Get the argument shape right.
- Form Request restriction (only `view`/`edit` on the share API endpoint) is a SHARE-API task concern — do NOT add that constraint in the policy layer.
