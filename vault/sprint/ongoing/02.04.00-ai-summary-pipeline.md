## What To Build

The complete async AI summary generation subsystem (ADR-002, SPEC §7, SRS §7).

### Architecture
```
app/
├── Contracts/SummaryGeneratorInterface.php
├── Services/Summary/
│   ├── SummaryManager.php           ← extends Illuminate\Support\Manager
│   ├── Drivers/
│   │   ├── LlmDriver.php           ← OpenAI-compatible HTTP client
│   │   └── RulesDriver.php         ← Deterministic fallback
│   └── SummaryResult.php            ← Immutable value object
├── Facades/Summary.php
├── Jobs/GenerateSummaryJob.php
├── Events/SummaryCompleted.php
├── Exceptions/SummaryGenerationException.php
└── Providers/SummaryServiceProvider.php
```

### SummaryGeneratorInterface
```php
interface SummaryGeneratorInterface
{
    /** @throws SummaryGenerationException */
    public function generate(Issue $issue): SummaryResult;
}
```

### SummaryResult (Value Object)
```php
final readonly class SummaryResult
{
    public function __construct(
        public string $summary,
        public string $suggestedNextAction,
        public string $driver,
    ) {}
}
```

### SummaryManager
- Extends `Illuminate\Support\Manager`
- `getDefaultDriver()` → `config('summary.default')`
- `createLlmDriver()` → LlmDriver with HTTP client + config
- `createRulesDriver()` → RulesDriver
- Auto-fallback: if `llm` selected but no API key → return rules driver

### LlmDriver
- Laravel HTTP client (injectable, mockable)
- POST `{base_url}/chat/completions`
- Temperature: 0.3, response_format: json_object
- Timeout: configurable (default 30s)
- Throws SummaryGenerationException on any failure

### RulesDriver
- Category + priority matrix → domain-specific summaries
- Description analysis: extract lead sentence
- Action suggestions: concrete, category-aware
- MUST produce genuinely useful output

### GenerateSummaryJob
- tries: 3, backoff: [10, 30, 90]
- Sets summary_status = processing on start
- Calls Summary::generate($issue)
- On success: updates summary + next_action, status = ready
- On final retry failure: fallback to rules driver
- On permanent failure: status = failed
- Fires SummaryCompleted event

### Config
```php
// config/summary.php
return [
    'default' => env('SUMMARY_DRIVER', 'rules'),
    'drivers' => [

];
```

### Implementation notes for coder-backend
- **SummaryManager auto-fallback:** in `getDefaultDriver()`, check `config('summary.drivers.llm.api_key')` — if null/empty AND default is `'llm'`, return `'rules'` (SRS §7.2 line 353). This is the config-time fallback, distinct from the job-level retry fallback.
- **LlmDriver HTTP call:** inject `Illuminate\Http\Client\Factory` via constructor (not `Http` facade directly) so unit tests can mock it without `Http::fake()` global state. Send to `{base_url}/chat/completions` with `response_format: {"type":"json_object"}` (SRS §7.3). Parse the `choices[0].message.content` JSON string — validate it has both `summary` and `suggested_next_action` keys, throw `SummaryGenerationException` if missing.
- **RulesDriver category coverage:** the 6 seeded categories are `billing`, `technical`, `account`, `general`, `bug`, `feature-request` (see `database/seeders/CategorySeeder.php`). The driver must load `$issue->category` (eager-load or lazy-load) and branch on `$issue->category->name`. Include a generic fallback for user-created categories. Priority-aware action text uses `Priority` enum cases: `Low`, `Medium`, `High`, `Critical` (see `app/Enums/Priority.php`).
- **GenerateSummaryJob `handle()` flow (SRS §7.5):** (1) set `summary_status=Processing`, save; (2) try `Summary::generate($issue)`; (3) on `SummaryGenerationException`: if `$this->attempts() < $this->tries` → rethrow to let Laravel retry; if `>=` → call `Summary::driver('rules')->generate($issue)` as explicit fallback; (4) write `summary`, `suggested_next_action`, set `status=Ready`, save; (5) fire `SummaryCompleted($issue)`.
- **GenerateSummaryJob `failed()` method:** set `summary_status=Failed`, save, `Log::error()`. This fires only when Laravel exhausts retries AND the final-attempt fallback itself threw — which should not happen with the deterministic rules driver, but must be handled (ADR-002 line 71).
- **`$issue->refresh()` before writing:** the job runs async; the issue may have been updated between dispatch and execution. Call `$this->issue->refresh()` at the top of `handle()` to get current DB state.

### Pitfalls
- **Do not change the `GenerateSummaryJob` constructor signature** — `IssueService` (line 42, 95) and existing tests (line 140 of `IssueCrudApiTest.php`) depend on `new GenerateSummaryJob($issue)`.
- **`SummaryServiceProvider` must be registered** in `bootstrap/providers.php`, not `config/app.php` — Laravel 13 uses the providers array file.
- **Facade accessor:** `Summary::getFacadeAccessor()` should return the `SummaryManager::class` string (or a custom binding key), matching what `SummaryServiceProvider` registers as a singleton.
- **`summary_status` string vs enum:** `IssueService::create()` line 39 sets `'summary_status' => 'pending'` as a raw string, while `IssueService::update()` line 82 uses the `SummaryStatus::Pending` enum. The job should use the enum consistently (model casts handle it, but be explicit).
- **Category eager-loading in job:** `GenerateSummaryJob` receives a serialized `Issue`. When it unserializes, relations are lost. The job must `$this->issue->load('category')` before passing to the driver, or the RulesDriver will get `null` from `$issue->category->name`.
