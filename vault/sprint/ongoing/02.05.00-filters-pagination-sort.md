## What To Build

Enhance the issue list endpoint with proper filtering, pagination, and sorting.

### Filtering (SPEC §5.6)
Query params — all optional, combinable:
- `?status=open` — exact match on status enum
- `?priority=high` — exact match on priority enum
- `?category=billing` — match on category **slug** (resolve via Category::where('slug', ...))
- Invalid values → ignored silently (no 422)

### Pagination
- Default 15 per page
- `?per_page=25` (optional, max 50)
- `?page=2`
- Response includes pagination metadata (total, current_page, last_page, etc.)

### Sorting
- `?sort=created_at` (default)
- Allowed sort fields: created_at, updated_at, priority, deadline_at
- `?direction=desc` (default: desc)
- Default sort: needs_attention desc, then priority desc, then created_at desc

### Implementation
- Build query scopes on Issue model: `scopeFilterByStatus`, `scopeFilterByPriority`, `scopeFilterByCategory`
- Chain scopes in controller/service
- Use Eloquent `paginate()` with `appends()` to preserve filter params in pagination links

### Remaining Gaps

**G1 — `?per_page=N` query param (functional gap)**
- Controller hardcodes `paginate(15)`. No support for `?per_page=25`.
- Need: read `$request->integer('per_page', 15)`, clamp to `min(1, max($val, 50))`, pass to `paginate()`.
- Files: `IssueController::index()` L77.
- Test: `?per_page=5` with 8 issues → page 1 has 5, page 2 has 3.

**G2 — `?sort=field` query param (functional gap)**
- Controller hardcodes the 3-column default sort. No user-selectable sort.
- Need: accept `?sort=` from allowlist `[created_at, updated_at, priority, deadline_at]`, fall back to default multi-column sort when absent. `priority` sort must use the `CASE` expression (already computed as `priority_order` alias at L42).
- Files: `IssueController::index()` L71-75.
- Tests: `?sort=priority` returns priority-ordered results; `?sort=updated_at` returns updated_at-ordered results; invalid `?sort=title` falls back to default.

**G3 — `?direction=asc|desc` query param (functional gap)**
- Controller hardcodes `orderByDesc`. No direction toggling.
- Need: accept `?direction=asc|desc`, default `desc`, apply to the user-selected sort field only. Validate against `['asc', 'desc']`, ignore invalid.
- Files: `IssueController::index()` L71-75.
- Tests: `?sort=created_at&direction=asc` returns oldest first; `?direction=invalid` falls back to desc.

**G4 — `appends()` on pagination links (functional gap)**
- Pagination links do not preserve filter/sort query params. Clients following `next_page_url` would lose filters.
- Need: `$issues->appends($request->query())` or explicit `appends(['status', 'priority', 'category', 'sort', 'direction', 'per_page'])` before returning.
- Files: `IssueController::index()` between L77-79.
- Test: assert `links.next` URL contains the filter param when a filter is active.

**G5 — Named model scopes (code-quality, not functional)**
- Task spec asks for `scopeFilterByStatus`, `scopeFilterByPriority`, `scopeFilterByCategory` on `Issue` model, but the current inline `where()` clauses in the controller are functionally equivalent.
- This is a **refactor for reusability**, not a bug. The scopes would let other parts of the app (e.g., export, dashboard) reuse filter logic.
- Files: `Issue.php` (add scopes), `IssueController::index()` (replace inline wheres with scope calls).
- No new tests needed — existing filter tests cover the behavior regardless of where the logic lives.

**G6 — Missing sort-specific tests**
- No test asserts the default sort order (needs_attention → priority → created_at).
- No test for `?sort=priority` or `?sort=deadline_at`.
- No test for `?direction=asc` vs `?direction=desc`.
- Files: `IssueCrudApiTest.php` — add 3-4 tests in the list section.

### Technical Guidance

- `per_page` clamp: use `clamp(intval($request->query('per_page', 15)), 1, 50)` — Laravel 13 / PHP 8.4 has no native `clamp()`, use `min(50, max(1, ...))`.
- Sort allowlist: define `const SORTABLE = ['created_at', 'updated_at', 'priority', 'deadline_at']` on the controller or model; use `in_array()` guard, not validation (match silent-ignore pattern from filters).
- When `?sort=priority`, reuse the existing `priority_order` alias from L42 selectRaw — `orderBy('priority_order', $direction)`.
- `appends()` must come BEFORE `IssueResource::collection()` since `collection()` reads the paginator's links at transform time.
- G5 scopes should accept the raw query string value and do the `tryFrom()` / slug resolution internally, keeping the controller truly thin.
- For `?sort=deadline_at`, issues with `null` deadline should sort last; use `orderByRaw("deadline_at IS NULL, deadline_at {$direction}")` (PostgreSQL-compatible `NULLS LAST` equivalent).
