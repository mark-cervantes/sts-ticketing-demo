# Task 03.01.00: Design System + App Layout Shell

## Business Requirements

### Problem
The app currently uses Breeze's default layout. Sprint 03 requires a dashboard-first Kanban UI with header, sidebar, and main content area — the structural shell everything else builds on.

### User Story
As a user, I want a polished, responsive dashboard layout with a sidebar for filters and a main content area for the Kanban board, so I can navigate the app efficiently.

### Acceptance Criteria
- [ ] App layout component with header (logo, user menu), sidebar, and main content area
- [ ] CSS custom properties for theming — primary color changeable in ONE file
- [ ] Dark mode toggle (Tailwind `dark:` variant, class-based)
- [ ] Responsive: sidebar collapses on mobile (< 768px)
- [ ] shadcn-vue components used where applicable (Button, DropdownMenu, Avatar, etc.)
- [ ] Skeleton loader components available for future use
- [ ] Dashboard route (`/dashboard`) renders the shell with placeholder content
- [ ] Toast notification system wired (shadcn-vue Sonner or Toast)

### Business Rules
- Header shows user name + avatar initial, logout action
- Sidebar has filter sections: Status, Priority, Category (placeholder UI only — wiring in 03.02)
- Stats section in sidebar (placeholder)
- "+ New Issue" button in sidebar

### Out of Scope
- Kanban columns (03.02)
- Issue create modal (03.03)
- Actual filter wiring to API
- SSE / real-time

## Technical Guidance

### Architecture Notes

- **Do NOT modify `AuthenticatedLayout.vue`** — Breeze auth pages (`Profile`, `Welcome`) still use it. Create a new `AppLayout.vue` in `resources/js/Layouts/` that replaces the Breeze shell for dashboard-class pages. `Dashboard.vue` switches its layout import to `AppLayout`.
- **Inertia layout convention:** layouts are NOT declared via `app.ts` resolve — each page component imports its layout directly (`import AppLayout from '@/Layouts/AppLayout.vue'`). Inertia's `defineOptions({ layout: AppLayout })` or wrapping in `<template>` both work; pick one pattern and use it consistently across all Sprint 3 pages.
- **Dark mode is already configured** — `resources/css/app.css` line 14 defines `@custom-variant dark (&:is(.dark *))` (class-based on `<html>`). The toggle composable must add/remove the `dark` class on `document.documentElement` and persist preference to `localStorage`. No Tailwind config file changes needed (v4 CSS-only config).
- **CSS custom properties are already wired** — `app.css` `:root` / `.dark` blocks are THE single source for color tokens (SPEC §9.1). The `@theme inline` block maps `--primary` → `--color-primary` for Tailwind utilities. Do NOT add a `tailwind.config.js` — this project uses Tailwind v4's CSS-based configuration.
- **Component composition:** `AppLayout.vue` composes `AppHeader.vue`, `AppSidebar.vue`, and a `<main>` slot. Keep each under 150 lines. Use shadcn-vue `Button` (`@/components/ui/button`), `Skeleton` (`@/components/ui/skeleton`), and `Sheet` (`@/components/ui/sheet` — for mobile sidebar drawer).
- **shadcn-vue style is `reka-nova`** (`components.json`). Import alias is `@/components/ui/<name>`. Components already installed: `button`, `badge`, `dialog`, `sheet`, `select`, `input`, `skeleton`, `sonner`, `tooltip`. Install `dropdown-menu` and `avatar` via `./vendor/bin/sail npx shadcn-vue@latest add dropdown-menu avatar`.
- **Sonner is already installed** at `@/components/ui/sonner`. Wire `<Toaster />` into `AppLayout.vue` (or `app.ts` root) so toasts work globally from Sprint 3 onward.

### Technical DoD

#### Visual
- Dashboard renders with header (logo + user name + avatar initial + logout), sidebar (placeholder filters, stats, "+ New Issue" button), and main content area
- Dark mode toggle works in header — page re-renders with dark tokens from `app.css` `.dark` block
- Sidebar collapses to a `Sheet` (slide-over drawer) on viewports `< 768px`; hamburger button in header triggers it
- Sonner `<Toaster />` renders (verify with a test toast from devtools or a temporary button)
- Screenshots in `test-results/playwright/smoke/dashboard.png` show the new shell (not Breeze default)

#### Code
- `AppLayout.vue` uses semantic token classes (`bg-background`, `text-foreground`, `bg-sidebar`, etc.) — no hardcoded color values
- Breeze's `Components/` directory (`@/Components/`) is NOT imported from the new layout — use shadcn-vue equivalents from `@/components/ui/`
- `useDarkMode` composable is a standalone file in `@/composables/`, not inline script in a component
- All new `.vue` files pass `vue-tsc` — `<script setup lang="ts">` with typed props/emits where applicable

### Gotchas

- **Case-sensitive imports:** Breeze uses `@/Components/` (capital C); shadcn-vue uses `@/components/ui/` (lowercase). Both directories coexist. New layout code must import from lowercase `@/components/ui/` only.
- **`@theme inline` is required** — `app.css` already uses it. If you try plain `@theme` the dark mode CSS variable overrides won't cascade at runtime. Do not change this.
- **Inertia persistent layouts:** if using `defineOptions({ layout: AppLayout })`, the layout is persistent across navigations (sidebar state preserved). If wrapping in `<template>`, it remounts on every navigation. Persistent is preferred for the sidebar UX.
- **Sheet vs custom sidebar:** on mobile, use shadcn-vue `Sheet` (already installed) with `side="left"` rather than a custom CSS sidebar. This gives accessible overlay + focus trap for free.
- **Avatar component:** needs to be installed (`npx shadcn-vue@latest add avatar`). It renders initials when no image URL is provided — use `$page.props.auth.user.name.charAt(0)` for the fallback.
- **Sonner positioning:** default is bottom-right. SPEC doesn't specify — keep default unless design says otherwise. Import `Toaster` from `@/components/ui/sonner` and place in layout root.
- **Sidebar width:** use a CSS custom property (e.g. `--sidebar-width: 16rem`) defined in `:root` so downstream tasks (03.02 Kanban columns) can account for it in their grid calculations.
