# Task 04.01.00: Sharing + Visibility Backend and Frontend

## Business Requirements

### Problem
Users need to share issues with other users at specific permission levels (view, comment, edit) and control whether an issue is publicly visible or private.

### User Story
As an issue owner, I want to share my issue with specific users at different permission levels, so I can collaborate while controlling access.

### Acceptance Criteria
- [ ] `POST /api/issues/{issue}/shares` — share an issue with a user (email + permission level)
- [ ] `DELETE /api/issues/{issue}/shares/{share}` — remove a share
- [ ] `PATCH /api/issues/{issue}/shares/{share}` — update share permission level
- [ ] `GET /api/issues/{issue}/shares` — list shares for an issue
- [ ] Only issue owner can manage shares (IssuePolicy::share already enforced)
- [ ] Permission levels: view, comment, edit (Permission enum already exists)
- [ ] Visibility toggle: PATCH `/api/issues/{id}` with `visibility` field already works
- [ ] Shared issues appear in the recipient's Kanban board (scopeAccessibleBy already handles this)
- [ ] All tests pass

### Business Rules
- Cannot share with yourself
- Cannot share with same user twice (upsert permission instead)
- Removing a share revokes all access for that user
- Public issues are viewable by any authenticated user (already handled by IssuePolicy::view)

### Out of Scope
- Share UI in slide-over (04.04)
- Email notifications on share

## Technical Guidance

- Create `ShareController` as an apiResource nested under issues: `Route::middleware('auth')->apiResource('issues.shares', ShareController::class)->shallow()` in `routes/api.php` — follows existing pattern (`CommentController` nests under issues)
- Use `updateOrCreate(['issue_id' => ..., 'user_id' => ...], ['permission' => ...])` in the store action to implement upsert behavior — the `unique(['issue_id','user_id'])` constraint in `2026_05_22_194120_create_issue_shares_table` enforces DB-level uniqueness
- Resolve the target user via `User::where('email', $request->email)->firstOrFail()` — return 422 if unregistered, not 404, to avoid leaking whether an email exists (match `StoreCommentRequest` convention of delegating authz to the controller via `$this->authorize('share', $issue)`)
- Create `StoreShareRequest` (validates `email` + `permission`) and `UpdateShareRequest` (validates `permission` only) in `app/Http/Requests/` — validate `permission` with `Illuminate\Validation\Rules\Enum(Permission::class)`
- Self-share prevention: add `$request->email !== $request->user()->email` validation rule in `StoreShareRequest` — this is a business rule (SPEC §4.5), not a DB constraint
- Authorize all four actions via `IssuePolicy::share()` (already owner-only at `app/Policies/IssuePolicy.php:116`) — the controller calls `$this->authorize('share', $issue)` before each mutation
- The `IssueShare` model (`app/Models/IssueShare.php`) has `UPDATED_AT = null` — never set `$share->updated_at`; upsert via `updateOrCreate` will only touch `permission` and `created_at` on insert
- FK cascade: `issue_shares.issue_id` has `cascadeOnDelete()` per migration — deleting an issue auto-removes shares; no orphan cleanup needed in `ShareController::destroy`
- New files: `app/Http/Controllers/ShareController.php`, `app/Http/Requests/StoreShareRequest.php`, `app/Http/Requests/UpdateShareRequest.php`, `tests/Feature/ShareControllerTest.php`
- Quality gates: `./vendor/bin/sail test --compact --filter=ShareController` then full suite; `vendor/bin/pint --dirty --format agent` before final commit
