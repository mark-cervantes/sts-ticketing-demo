# Task 04.03.00: Optimistic Locking UI Feedback

## Business Requirements

### Problem
When two users edit the same issue simultaneously, the second save gets a 409 conflict. The current UI shows a toast, but should offer better UX — show what changed and let the user retry.

### User Story
As a user, I want to see a clear conflict message when my edit conflicts with another user's, so I can review the changes and retry.

### Acceptance Criteria
- [ ] On 409 from PATCH, show a conflict dialog (not just toast)
- [ ] Conflict dialog shows: "This issue was updated by another user. Your changes were not saved."
- [ ] "Reload & Retry" button: fetches fresh data, keeps the slide-over open
- [ ] "Discard" button: fetches fresh data, discards user's pending changes
- [ ] After reload, `updated_at` is refreshed so next save works
- [ ] Works for both slide-over edits and drag-drop status changes
- [ ] All tests pass

### Business Rules
- 409 conflict should never silently lose data
- The fresh reload should update the entire issue (another user may have changed multiple fields)

### Out of Scope
- Showing a diff of what the other user changed (too complex for this sprint)
- Real-time conflict prevention (would need websockets)

## Technical Guidance

- In `useIssueDetail.ts:patchIssue` (line 49–53), replace the `toast.error` + silent `fetchIssue` with a return value that signals "conflict" to the caller — e.g. return `'conflict'` instead of `false`, so `IssueDetailSheet.vue` can open a dialog instead of silently refreshing.
- Add a `conflictDialogOpen` ref in `IssueDetailSheet.vue` alongside the existing `deleteDialogOpen` (line 90); reuse the already-imported `AlertDialog*` components (lines 29–37) — no new UI dependency needed.
- The "Reload & Retry" action must call `fetchIssue(issue.id)` then `syncLocalFields()` (line 179) so `localTitle` / `localDescription` pick up the fresh server state while the sheet stays open.
- The "Discard" action is identical to "Reload & Retry" except it does NOT preserve pending edits — just `fetchIssue` + `syncLocalFields` is sufficient since `syncLocalFields` overwrites locals from `issue.value`.
- For drag-drop conflicts in `useKanbanBoard.ts:moveIssue` (line 199–205), keep the revert + toast pattern — a modal dialog during drag-drop is disorienting; but change the toast copy to match the new conflict message from the AC.
- `patchIssue` already calls `fetchIssue(currentIssue.id)` on 409 (line 51), which updates `issue.value` and triggers the `watch(issue, …)` at `IssueDetailSheet.vue:187` — verify that `updated_at` is refreshed on the ref after dialog dismiss so the next save sends the current token.
- Every `patchIssue` call site in `IssueDetailSheet.vue` (lines 214, 219, 224, 229, 234, 239, 247, 251) must handle the new conflict return — a single wrapper or `watch` on a `conflictDetected` ref avoids touching all 8 call sites individually.
- No backend changes required — the 409 response and `updated_at` locking are already implemented in `IssueController::update`.
