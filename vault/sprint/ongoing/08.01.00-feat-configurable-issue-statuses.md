# Task 08.01: Configurable Issue Statuses

## Business Requirements

### Problem
Issue statuses are hardcoded as a PHP enum (`open`, `in_progress`, `resolved`) and a TypeScript union type. Users cannot rename, add, or remove statuses to match their own workflow. Teams with different processes (e.g., "Waiting on Customer", "Escalated", "QA Review") are stuck with the three defaults.

### User Story
As an admin, I want to manage issue statuses (add, rename, delete, reorder) so that the Kanban board and issue lifecycle match my team's actual workflow.

### Acceptance Criteria
- [ ] Status management UI accessible from Settings (like Category management)
- [ ] Default statuses (Open, In Progress, Resolved) are seeded on fresh install
- [ ] Admin can **add** a new status with a custom name and display color
- [ ] Admin can **rename** an existing status
- [ ] Admin can **delete** a status that has no issues (409 if issues exist, with count)
- [ ] Admin can **reorder** statuses (drag or up/down arrows) — order determines Kanban column order
- [ ] Kanban board dynamically renders columns from the `statuses` table (not hardcoded)
- [ ] Issue detail sheet status dropdown populated from `statuses` table
- [ ] Sidebar filter checkboxes populated from `statuses` table
- [ ] Existing issues keep their status after migration (data preserved)
- [ ] API: `GET /api/statuses` returns all statuses ordered by `sort_order`
- [ ] API: `POST /api/statuses` creates a new status
- [ ] API: `PUT /api/statuses/{id}` updates name/color/sort_order
- [ ] API: `DELETE /api/statuses/{id}` deletes if no issues reference it
- [ ] Visual verification via `make verify-visual`

### Business Rules
- Statuses are global (shared across all users), like categories
- Status names must be unique (case-insensitive)
- A status with existing issues cannot be deleted (show count in error)
- Each status has: `id`, `name`, `slug` (auto-generated), `color` (hex), `sort_order`, `is_default` (used as initial status for new issues)
- Exactly one status must be marked `is_default` at any time
- The default status cannot be deleted
- `sort_order` determines Kanban column left-to-right order

### Data Migration
- Create `statuses` table
- Seed three rows: Open (order 0, default), In Progress (order 1), Resolved (order 2)
- Change `issues.status` from string enum to `status_id` foreign key
- Migrate existing issue status values to the corresponding status IDs
- Remove the `Status` PHP enum after migration

### Out of Scope
- Status transition rules (e.g., "can only go from Open → In Progress")
- Per-user or per-role status visibility
- Status-based automations or triggers

### Affected Areas
- **Backend:** Migration, `Status` model, `StatusController`, `IssueResource`, `Issue` model (cast change), `UpdateIssueRequest` validation, `RecalculateAttentionCommand`, seeder
- **Frontend:** `types/issue.ts` (remove hardcoded type), `useKanbanBoard.ts`, `useIssueFilters.ts`, `IssueDetailSheet.vue`, `AppSidebar.vue`, `KanbanColumn.vue`, new `StatusSettings.vue` page
- **Tests:** Existing tests referencing `Status::Open` etc. need updating

### Architecture Notes

1. **Three-migration strategy required** — do NOT attempt a single migration. Migration 1: create `statuses` table and seed the three rows. Migration 2: add `status_id` FK to `issues` (nullable) and populate it via `UPDATE issues SET status_id = (SELECT id FROM statuses WHERE slug = issues.status)`. Migration 3: drop the old `issues.status` string column and drop the composite indexes `[status, priority]` and `[user_id, status]`, then recreate them using `status_id`. Running these as three discrete, reversible migrations prevents data loss if any step fails during `make migrate`. See: `database/migrations/2026_05_22_194119_create_issues_table.php` (existing indexes to recreate).

2. **`Status` model follows `Category` model exactly** — slug auto-generated in a `creating` boot hook, `generateUniqueSlug` pattern, `#[Fillable]` attribute, `is_default` toggle protected by a `beforeDelete` guard that returns 409 when `is_default = true`. See: `app/Models/Category.php` (source pattern for slug and deletion guard logic structure).

3. **`StatusController` mirrors `CategoryController` with additions** — `index`, `store`, `destroy` (identical pattern from `app/Http/Controllers/CategoryController.php`) plus `update` (PATCH for rename/color/sort_order). No Policy needed — statuses are open to all authenticated users, same rationale as categories (`SRS §FR-08`). Register in `routes/api.php` as `Route::middleware('auth')->apiResource('statuses', StatusController::class)`.

4. **`StoreStatusRequest` must use case-insensitive uniqueness check on `statuses` table** — copy the `LOWER(name)` raw-query pattern from `app/Http/Requests/StoreCategoryRequest.php` verbatim; PostgreSQL's UNIQUE index is case-sensitive. Also validate `color` as a 7-char hex string (`#[A-Fa-f0-9]{6}`). `UpdateStatusRequest` applies the same uniqueness rule but excludes the current record by ID.

5. **`Issue` model: replace `status` cast and scope** — remove `'status' => Status::class` from `casts()` in `app/Models/Issue.php`. Add `public function status(): BelongsTo` returning `$this->belongsTo(IssueStatusModel::class, 'status_id')`. Replace `scopeFilterByStatus` (currently uses `Status::tryFrom`) to query `status_id` via a subquery on the `statuses` table: `->where('status_id', fn($q) => $q->select('id')->from('statuses')->where('slug', $value))`. Do not use the enum class name `Status` — it will be deleted.

6. **`RecalculateAttentionCommand` filter is the highest-risk breakage** — `app/Console/Commands/RecalculateAttentionCommand.php:41` currently does `Issue::whereIn('status', [Status::Open, Status::InProgress])`. After migration this must become `Issue::whereHas('status', fn($q) => $q->whereIn('slug', ['open', 'in_progress']))` or `Issue::whereIn('status_id', Status::whereIn('slug', ['open', 'in_progress'])->pluck('id'))`. Both reference `SRS §FR-13`: "For all open/in_progress issues: recompute needs_attention." The slug values `open` and `in_progress` are now DB values, not enum cases — they survive only because the seeder creates them. Do NOT hardcode IDs.

7. **`IssueResource` shape change** — `app/Http/Resources/IssueResource.php:31` currently emits `'status' => $this->status->value`. After migration emit `'status' => $this->status->slug` and `'status_id' => $this->status_id` and include the full status object: `'status_obj' => ['id' => $this->status->id, 'name' => $this->status->name, 'color' => $this->status->color, 'slug' => $this->status->slug]`. Add `status` to the `with()` call in `IssueController::index()` alongside `category` and `user`. The frontend currently expects `issue.status` as a string slug — preserve that field for backward compat during transition.

8. **`IssueFactory` and `IssueSeeder` must be updated before any test runs** — `database/factories/IssueFactory.php` has `'status' => Status::Open` in `definition()` and status state methods (`open()`, `inProgress()`, `resolved()`) that use `Status::*` enum cases. All of these must switch to `'status_id' => fn() => IssueStatusModel::where('slug', 'open')->value('id')` — or use a factory state that accepts a slug. `database/seeders/IssueSeeder.php` references `->open()`, `->inProgress()`, `->resolved()` factory states — these will continue to work once the factory states are updated. Update factory states before touching any other file to keep `make test` green throughout.

9. **Frontend: `IssueStatus` type becomes a DB-backed shape, not a union** — `resources/js/types/issue.ts:6` defines `export type IssueStatus = 'open' | 'in_progress' | 'resolved'`. Replace with `export interface IssueStatus { id: number; name: string; slug: string; color: string; sort_order: number; is_default: boolean }`. `STATUS_CONFIG` (line 141) and the hardcoded `KanbanColumnDef.status: IssueStatus` shape must become runtime values fetched from `GET /api/statuses`. The `Issue` interface's `status` field becomes `status: string` (slug, for backward compat) plus `status_obj: IssueStatus`. All downstream consumers (`useKanbanBoard.ts`, `useIssueFilters.ts`, `AppSidebar.vue`, `IssueDetailSheet.vue`) reference the `IssueStatus` type — all need updating.

10. **`useKanbanBoard.ts` hardcodes three status slugs in seven places** — `columnMap`, `paginationState`, `columnLoading` (lines 16–33) are all `Record<IssueStatus, ...>` keyed by literal string. `loadInitial` (line 118) hardcodes `['open', 'in_progress', 'resolved']`. These must be replaced by a `useStatuses()` composable (parallel to `useCategories()`) that fetches from `GET /api/statuses` on first use and exposes a reactive `statuses` array. `useKanbanBoard` initializes its maps dynamically from `statuses.value` after the fetch resolves. `columns` computed must order by `sort_order` from the API, not hardcoded array index.

### Gotchas

- **`IssueFactory` must be fixed in Phase 1 step 7, before any other test-touching work.** The factory has `'status' => Status::Open` as the default definition AND uses `Status::*` in seven state methods (`open()`, `inProgress()`, `resolved()`, `status()`, `needsAttention()`). Every test that creates an `Issue` via factory will fail with "Class App\Enums\Status not found" the moment the enum is deleted — even tests that don't touch status at all. Fix the factory first; delete the enum last.

- **`tests/Unit/Enums/StatusTest.php` tests a class that will be deleted.** This test asserts `Status::cases()` returns exactly 3 values. It will fail as soon as the enum is gone. Delete it only after replacing it with `tests/Unit/Models/StatusTest.php` that covers slug generation and the deletion guard — per `SRS §8.6 rule 3`: "Do NOT modify existing tests unless SPEC explicitly changed." The deletion of the enum IS a SPEC change (task 08.01 §Data Migration), so deletion is authorized.

- **Migration 2 (data population) must use a raw UPDATE, not Eloquent.** Running `Issue::all()` during a migration is an anti-pattern — if the `Issue` model's `casts()` references `Status::class` (which it still does at migration time), loading issues will throw. Use `DB::statement("UPDATE issues SET status_id = s.id FROM statuses s WHERE s.slug = issues.status")` directly. See `database/migrations/2026_05_22_194119_create_issues_table.php` for the existing migration style.

- **`RecalculateAttentionCommand` scope uses `Status::Open` and `Status::InProgress` as enum values passed to `whereIn`.** When Eloquent casts `status` to the `Status` enum (pre-migration), passing the enum to `whereIn` works because the cast converts to a string. Post-migration, the column is `status_id` (integer) and the enum is gone — `whereIn('status_id', ...)` needs integer IDs from the DB. Using `IssueStatus::whereIn('slug', ['open', 'in_progress'])->pluck('id')` is safe and slug-stable. See `app/Console/Commands/RecalculateAttentionCommand.php:41`.

- **`moveIssue` in `useKanbanBoard.ts` sends `status: toStatus` (a string slug) in the PATCH body.** After migration the API validates `status_id` (integer) on `UpdateIssueRequest`. The PATCH body must change to `status_id: toStatus.id` where `toStatus` is now an `IssueStatus` object, not a string. All four optimistic-revert paths in `moveIssue` and `revertMove` compare `issue.status` (string) — after refactor they compare `issue.status_id` (number). Update the type guard and the revert logic together.

- **`useIssueFilters` initializes with `statuses: ['open', 'in_progress', 'resolved']` as string slugs.** The Kanban board uses this to decide which columns are visible (filter toggle). After the refactor, the filter must track status IDs (integers), not slugs, because the API accepts `?status_id=1` not `?status=open`. The `clearFilters()` function must re-initialize from `useStatuses()` — but statuses may not be loaded yet on first render. Fetch statuses before initializing filter state, or initialize to "all" and let the reactive `useStatuses` data hydrate it.

- **`AppSidebar.vue` has `hasActiveFilters` computed that hardcodes `filters.value.statuses.length < 3`.** After the refactor, "all selected" means `filters.value.statuses.length === allStatuses.value.length` — change the comparison to use the dynamic status count. See `resources/js/Layouts/partials/AppSidebar.vue:61`.

- **`is_default` enforcement is a DB-level invariant.** The business rule says exactly one status must be `is_default` at all times. The seeder sets Open as default. The `StatusController::destroy` must block deletion of the default status with 409 before checking issue count. The `StatusController::update` must handle the `is_default` toggle: when setting a new status as default, clear the flag from all others in a transaction. No raw `is_default = true` updates outside the service-layer transaction.

- **Slug stability for existing API consumers.** The `GET /api/issues` filter currently accepts `?status=open`. After migration the canonical filter is `?status_id=1`. Keep `scopeFilterByStatus` accepting a slug string (resolving to `status_id`) for backward compatibility during transition — the frontend can move to `status_id` filter in Phase 2, but the backend scope must remain slug-resolvable so existing integration tests (e.g., `tests/Feature/IssueCrudApiTest.php`) continue to pass without rewriting every API call.

- **`vue-tsc` will fail immediately on build if any component still references `STATUS_CONFIG[status].label` with `status` typed as a string.** `STATUS_CONFIG` in `resources/js/types/issue.ts:141` is keyed by the old `IssueStatus` string union. After the type change, all six reference sites (`AppSidebar.vue`, `IssueDetailSheet.vue`, `useKanbanBoard.ts` ×3, `KanbanColumn.vue`) must be updated before `make verify-visual` will pass.
