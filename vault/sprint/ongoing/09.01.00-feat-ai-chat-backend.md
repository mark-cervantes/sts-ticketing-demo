# Task 09.01: AI Chat ‚Äî Backend (Streaming Endpoint + Saved Conversations)

## Business Requirements

### Problem
Users can see AI-generated summaries on issues, but they can't ask the AI follow-up questions. When a summary says "Carol confirmed the gateway timeout," there's no way to ask "What timeout value?" or "Which gateway endpoint?" without reading the entire comment thread. Users want a conversational interface where they can chat with AI about the issue ‚Äî right from the summary section.

### User Story
As a user viewing an issue, I want to chat with AI about the issue so I can drill into details, and optionally save the conversation so my team can see it too.

### Key UX Concept: Session-First, Save-on-Demand

**Chat is ephemeral by default.** Messages live only in the user's browser session. Nothing is persisted until the user explicitly saves the conversation. This keeps the feature lightweight and low-friction ‚Äî users can ask throwaway questions without cluttering the issue.

**Saved conversations are a separate entity** from the AI synthesis. They appear below the summary as their own section on the issue, visible to anyone who can view the issue. Saved conversations can be continued by anyone with access.

#### API Endpoints
| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/issues/{issue}/chat` | Stateless ‚Äî send message + chat history in body, get streamed AI response. Nothing persisted. |
| `POST` | `/api/issues/{issue}/conversations` | Save a conversation ‚Äî accepts `{ title?, messages: [{role, content}] }`, persists to DB |
| `GET` | `/api/issues/{issue}/conversations` | List saved conversations for an issue (newest first) |
| `GET` | `/api/issues/{issue}/conversations/{conversation}` | Load a saved conversation's messages |
| `POST` | `/api/issues/{issue}/conversations/{conversation}/continue` | Append a new message to a saved conversation + get streamed AI response |

#### Streaming Strategy
The `POST /chat` endpoint is **stateless** ‚Äî the frontend sends the full message history (session messages) in the request body. The backend builds the LLM prompt from issue context + provided messages, calls `{base_url}/chat/completions` with `stream: true`, and pipes tokens back as SSE.

For continuing saved conversations (`POST /conversations/{id}/continue`), the backend loads existing messages from DB, appends the new user message, streams the AI response, and saves both the user message and AI response to DB.

#### Context Window
The AI receives:
1. **System prompt** ‚Äî configurable from DB (`chat_system_prompt` on `ai_settings`)
2. **Issue context** ‚Äî title, description, category, priority, status
3. **Recent comments** ‚Äî last N comments with author names and timestamps
4. **Chat history** ‚Äî from request body (session chat) or DB (saved conversation)

### Business Rules
- Stateless chat: the backend does NOT store anything on `POST /chat` ‚Äî purely ephemeral
- Saved conversations are a **separate section** on the issue ‚Äî they do NOT replace or modify the AI synthesis
- Anyone who can view the issue can see saved conversations
- Anyone who can view the issue can continue a saved conversation
- The user who saves gets attributed (`saved_by`)
- Rate limit applies to both stateless chat and continue endpoints
- If no AI provider configured, `POST /chat` and `/continue` return 503

### Out of Scope
- Frontend UI (task 09.02)
- Conversation deletion or editing
- Multi-turn function calling / tool use
- File attachments in chat
