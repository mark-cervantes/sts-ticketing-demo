# Task 09.02: AI Chat — Frontend (Session Chat + Save + Continue)

## Business Requirements

### Problem
The backend AI chat API exists (task 09.01), but there's no UI. Users need to chat with AI directly from the summary area on the issue detail sheet, with the ability to save important conversations.

### User Story
As a user viewing an issue summary, I want to ask the AI follow-up questions right from the summary, and save the conversation if it's worth sharing with my team.

### UX Design

#### Entry Point: The Summary Section
The chat opens **from the AI synthesis section** on the issue detail sheet. A small "Ask AI" button/input appears below the summary. Clicking or typing opens the chat interface inline — the user is already looking at the summary when they have a question, so the chat should be right there.

#### Session Chat (Ephemeral)
```
┌─────────────────────────────────────┐
│  AI Synthesis                       │
│  Billing portal returns 502 during  │
│  peak hours...                      │
│  • Carol Chen: confirmed at 30s...  │
│  • Alice Johnson: recommends...     │
├─────────────────────────────────────┤
│  ⚠ This chat is not saved.         │
│  It will be lost when you leave     │
│  this page.                  [Save] │
├─────────────────────────────────────┤
│                                     │
│         You: What timeout value     │
│         did Carol mention?          │
│                                     │
│  AI: Carol confirmed the gateway    │
│  hard-times out at 30 seconds       │
│  during 09:00–11:00 UTC.            │
│                                     │
│         You: Is there a retry?      │
│                                     │
│  AI: Alice recommended a local      │
│  retry queue with...█               │  ← streaming
│                                     │
├─────────────────────────────────────┤
│  [Ask about this issue...    ] [⏎]  │
└─────────────────────────────────────┘
```

#### ⚠ Ephemeral Warning (MUST be prominent)
A **persistent, highly visible banner** at the top of the chat area:
- Yellow/amber warning background: "⚠ This chat is not saved. It will be lost when you close the browser."
- "Save" button on the same banner line
- The warning stays visible AT ALL TIMES while the chat is unsaved
- After saving: banner changes to "✓ Conversation saved" (green, with link to view)

#### Session Persistence
Unsaved chat messages are stored in **`sessionStorage`** (keyed by issue ID), NOT component state. This means:
- ✅ Survives page navigation within the app (Inertia visits, back/forward)
- ✅ Survives page refresh (F5)
- ❌ Cleared when the browser tab is closed
- Each issue has its own independent session chat

#### Saved Conversations Section
Below the AI synthesis (and below the chat input), a "Saved Conversations" section lists previously saved conversations on this issue:
```
┌─────────────────────────────────────┐
│  💬 Saved Conversations             │
├─────────────────────────────────────┤
│  "Timeout investigation"            │
│  Saved by Alice · 5 messages · 2h   │
│  [Continue]                         │
├─────────────────────────────────────┤
│  "Retry queue options"              │
│  Saved by Carol · 3 messages · 1d   │
│  [Continue]                         │
└─────────────────────────────────────┘
```

Clicking "Continue" opens that conversation in the chat area (replacing any unsaved session chat — warn first if messages exist).

#### Save Flow
1. User clicks "Save" on the ephemeral warning banner
2. Optional: prompt for a title (auto-suggest from first message, e.g., "Timeout investigation")
3. POST to `/api/issues/{issue}/conversations` with the session messages
4. Banner changes to green "✓ Saved" state
5. Conversation appears in the Saved Conversations list
6. Further messages in this chat now auto-save (it's a saved conversation now)

### Business Rules
- Chat panel only shows if user has view permission (`can.view`)
- Session messages are stored in `sessionStorage` (keyed by issue ID) — survives navigation/refresh, cleared on tab/browser close
- Saved conversations are visible to anyone who can view the issue
- Saved conversations are continuable by anyone who can view the issue
- The "Save" action is the only path from ephemeral → persisted
- Conversations do NOT replace the AI synthesis — they're a separate section below it

### Out of Scope
- Conversation deletion or renaming
- File attachments in chat
- @mentions
- Chat export
- Typing indicators

## Technical Guidance

### Architecture Notes

- **Problem class: Frontend wiring** — new composable + component subtree injected into an existing sheet. The sheet's `<SheetContent>` already uses `overflow-y-auto`; the chat panel must NOT add a second scroll context inside it, or the auto-scroll sentinel will be unreachable. Keep the panel as a normal block inside the existing scroll container (`IssueDetailSheet.vue` line 355).

- **Composable scope: do NOT use module-level singleton state** (contrast `useKanbanBoard.ts` lines 19–22 which are module-scoped singletons). `useIssueChat` is instance-scoped — each call gets its own `ref` objects — because two different issues can be open in sequence within the same session. Follows the `useCategories.ts` pattern (lines 5–7: refs inside the factory, not outside).

- **SSE streaming via `fetch()` + `ReadableStream`, NOT `EventSource`**: `EventSource` only supports GET (MDN spec). The stateless chat endpoint is `POST /api/issues/{issue}/chat` (09.01 task, API table). Use `apiPost` from `resources/js/composables/useApiFetch.ts` (line 37) which returns a raw `Response` — then call `response.body.getReader()`. Parse lines prefixed `data: `, accumulate content into `streamingContent` ref, finalize on `data: [DONE]`. Mirror the CSRF token pattern from `useApiFetch.ts` line 43 (`X-XSRF-TOKEN: getCsrfToken()`).

- **sessionStorage key must be issue-scoped**: Use `ai-chat-${issueId.value}` as the key. On composable init (inside `watch(issueId, ...)` — not `onMounted`, because `issueId` can change while the sheet stays open), attempt `sessionStorage.getItem(key)` and parse JSON. On every send/receive, call `sessionStorage.setItem(key, JSON.stringify(messages.value))`. This matches the task's session persistence guarantee (task §Session Persistence).

- **Streaming request body shape** matches 09.01 backend contract: `{ message: string, history: Array<{role: 'user'|'assistant', content: string}> }` for `POST /api/issues/{issue}/chat`; `{ message: string }` for `POST /api/issues/{issue}/conversations/{conversation}/continue`. These are two distinct fetch paths — `activeConversationId` in the composable determines which endpoint to hit.

- **No `Alert` component exists** in `resources/js/components/ui/` — only `AlertDialog`. The ephemeral warning banner must be a hand-crafted `<div>` with Tailwind classes. Model it after the existing `needs_attention` banner in `IssueDetailSheet.vue` lines 443–448 (amber border/background pattern using `border-amber-200 bg-amber-50 dark:border-amber-900 dark:bg-amber-950/50 dark:text-amber-200`). The `AlertDialog` from `resources/js/components/ui/alert-dialog/` (already imported in `IssueDetailSheet.vue` lines 30–38) is the correct component for the "unsaved messages — switch?" confirm.

- **No markdown library is installed** — `package.json` has no `marked`, `showdown`, or `dompurify`. Add `marked` (lightweight, ~5 kB min+gzip) via `./vendor/bin/sail npm install marked`. Do NOT use `v-html` on raw AI content without sanitization — XSS risk. Either pair `marked` with `DOMPurify` (`./vendor/bin/sail npm install dompurify @types/dompurify`), or use `marked`'s built-in `sanitize` option (deprecated in v5+, prefer DOMPurify). AI messages are the only place `v-html` is acceptable after sanitization; user messages must render as plain text.

- **`IssueDetailSheet.vue` is already large (772 lines)**. Encapsulate the entire chat feature in `resources/js/components/issues/IssueChatPanel.vue`. Pass `issueId` and `canView` as props. `IssueDetailSheet` adds only a single `<IssueChatPanel>` after the AI Summary `</div>` (line 682) and before `<ShareSection>` (line 685). Sub-components `ChatMessage.vue`, `ChatInput.vue`, and `SavedConversationsList.vue` live under `resources/js/components/issues/` to stay consistent with `CommentThread.vue` and `ShareSection.vue`.

- **`@vueuse/core` is available** (v14.3.0 in `package.json`). Use `useStorage` from `@vueuse/core` instead of raw `sessionStorage` calls — it provides reactive sync and handles JSON serialization. Pass `sessionStorage` as the storage argument. This eliminates manual `getItem`/`setItem` calls and keeps the ref automatically in sync.

- **Permission gate**: Only render `<IssueChatPanel>` when `issue.value?.can?.view` is true (same pattern as `CommentThread` prop `can-comment` at `IssueDetailSheet.vue` line 691). The `can` map on the Issue type is already populated by the backend's `IssueResource`.

- **Saved-conversation auto-persist**: Once `activeConversationId` is non-null (saved mode), every call to `send()` must POST to `/api/issues/{issue}/conversations/{id}/continue` instead of `/api/issues/{issue}/chat`. The backend persists both the user message and AI response (09.01 spec §Streaming Strategy). The composable does NOT need to manually append to `messages` after the stream ends — it should append the finalized `streamingContent` as an `assistant` message when `[DONE]` is received, regardless of mode.

### Gotchas
- **`SheetContent` is the scroll root** (`overflow-y-auto` at `IssueDetailSheet.vue` line 355): do not add `overflow-y-auto` or a fixed `max-height` on `IssueChatPanel` itself, or the auto-scroll `scrollIntoView` on the sentinel element will target the wrong scroll ancestor. The chat message list should expand naturally; use `max-h-64 overflow-y-auto` only on the message list sub-div, keeping the sentinel outside it so it scrolls the sheet.
- **`activeConversationId` watch**: when `issueId` prop changes (user closes and reopens a different issue), `useIssueChat` must reset all state — `messages`, `isSaved`, `activeConversationId`, `streamingContent`, `isStreaming` — and attempt sessionStorage restore for the new ID. A `watch(issueId, ...)` with `immediate: true` handles this; do NOT use `onMounted` alone, because the composable outlives a single issue open/close cycle while the sheet component stays mounted.
- **`[DONE]` parsing**: the backend (09.01 spec §Streaming) emits `data: [DONE]` as the final SSE line. The `ReadableStream` reader yields raw bytes; use a `TextDecoder` to convert chunks, then split on `\n` to extract `data: ` lines. Accumulate partial lines across chunks — a `data: ` line can be split across two `read()` calls if the chunk boundary falls mid-line.
- **Rate limit (429) response**: the streaming endpoint returns a non-streaming JSON error when rate-limited (09.01 spec §Acceptance Criteria). Check `response.ok` before calling `response.body.getReader()`. If `response.status === 429`, parse `response.json()` and surface the `retry_after` seconds inline in the chat — do NOT attempt to read the stream body.
- **Title dialog for Save**: the task calls for an optional title prompt. Implement this as a small inline `<Dialog>` (not `AlertDialog`) pre-filled with the first user message truncated to 60 chars. The `Dialog` component is not yet in `resources/js/components/ui/` — install it via `./vendor/bin/sail npx shadcn-vue@latest add dialog` before using it.
