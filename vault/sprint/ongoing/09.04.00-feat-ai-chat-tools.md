# Task 09.04: AI Chat Tools — Extensible Tool Architecture + Discovery UX

## Business Requirements

### Problem
Users chatting with AI about an issue often want to take action — "create a ticket for the retry queue." Currently the chat is text-only. There's also no way for users to discover that the AI can do more than answer questions.

### User Story
As a user chatting with AI, I want the AI to create tickets based on our conversation, and I want to naturally discover this capability without reading documentation.

## Part A: Backend — Extensible Tool System

### Architecture: Tool Registry Pattern

```
app/Services/Ai/Tools/
├── ChatToolInterface.php      ← contract every tool implements
├── ChatToolResult.php         ← value object for tool execution results
├── ChatToolRegistry.php       ← discovers + registers all tools, provides schema to LLM
└── CreateTicketTool.php       ← first tool
```

**`ChatToolInterface`:**
```php
interface ChatToolInterface
{
    public function name(): string;                    // e.g., 'create_ticket'
    public function description(): string;             // for LLM system prompt
    public function parameterSchema(): array;          // OpenAI function calling JSON schema
    public function execute(array $parameters, Issue $issue, User $user): ChatToolResult;
    public function requiresConfirmation(): bool;      // true = render card, wait for user click
    public function suggestionChip(): ?string;         // e.g., "Create a follow-up ticket" — null if no chip
}
```

The `suggestionChip()` method lets each tool register its own discovery prompt for the frontend empty state.

**`ChatToolResult`** value object:
```php
class ChatToolResult
{
    public function __construct(
        public readonly string $toolName,
        public readonly bool $success,
        public readonly string $message,
        public readonly ?array $data = null,        // e.g., { id: 42, title: "...", url: "/issues/42" }
        public readonly bool $pendingConfirmation = false,
    ) {}
}
```

**`ChatToolRegistry`:**
- Registered via service provider (explicit, not auto-discovery — predictable ordering)
- `getToolDefinitions(): array` — OpenAI function calling format for LLM request
- `getSuggestionChips(): array` — collects non-null chips from all registered tools
- `executeTool(string $name, array $params, Issue $issue, User $user): ChatToolResult`

**`CreateTicketTool`:**
- Name: `create_ticket`
- Description: "Create a new support ticket based on the conversation context."
- Parameters: `{ title: string, description: string, priority: enum, category: string }`
- Requires confirmation: YES
- Suggestion chip: "Create a follow-up ticket"
- Execute: Creates `Issue` via existing pipeline, returns ID + title + URL

### ChatService Updates
- Include `tools` param in `/chat/completions` request when tools are registered
- When LLM responds with `tool_calls`: parse, execute via registry, feed result back as `tool` role message
- **Fallback for non-function-calling models:** Include tool descriptions in system prompt, instruct LLM to output `{"tool_call": {"name": "...", "arguments": {...}}}` JSON block, parse response for this pattern
- Stream tool call events as distinct SSE: `data: {"type":"tool_call","tool":"create_ticket","arguments":{...},"requires_confirmation":true}\n\n`

### New Endpoint
- `POST /api/issues/{issue}/chat/tool-confirm` — accepts `{ tool: string, arguments: object }`, executes via registry, returns `ChatToolResult`

### API: Suggestion Chips
- `GET /api/chat/suggestions` — returns `ChatToolRegistry::getSuggestionChips()` as JSON array. Frontend calls once on load.

## Part B: Frontend — Tool Discovery UX (4 Layers)

### Layer 1: Empty State Suggestion Chips (highest impact)
When the chat panel has no messages, show clickable suggestion chips instead of plain placeholder text:

```
┌─────────────────────────────────────┐
│  💡 Try asking:                     │
│  [Summarize the discussion]         │
│  [What's the root cause?]           │
│  [Create a follow-up ticket]  📋    │
│  [What should we do next?]          │
├─────────────────────────────────────┤
│  [Ask about this issue...    ] [⏎]  │
└─────────────────────────────────────┘
```

- **Static chips** (hardcoded): "Summarize the discussion", "What's the root cause?", "What should we do next?"
- **Tool chips** (from API): fetched from `GET /api/chat/suggestions`, each gets a subtle icon
- Clicking a chip = sends that text as the first message
- Chips disappear once the first message is sent

### Layer 2: Contextual Nudge After AI Insight
When AI response contains action-oriented language (keywords: "recommend", "suggest", "should", "need to", "follow-up"), show a subtle hint below the message:

```
💡 You can ask me to create a ticket for this.
```

- Rendered as a muted, italic line below the AI message — NOT a button
- Only shows on the first qualifying AI message per session (don't spam)
- Tracked via a `hasShownNudge` ref in the composable

### Layer 3: Input Placeholder Rotation
The chat input placeholder rotates between hints every 8 seconds:
- "Ask about this issue..."
- "Try: create a ticket for this"
- "Ask anything about this issue..."

Subtle — teaches through repetition. Stops rotating once the user starts typing.

### Layer 4: First-Time Tooltip (One-Time Education)
On the user's first-ever chat panel open (tracked via `localStorage` key `ai-chat-onboarded`):
- Show a small popover/tooltip above the input:
  > "💡 You can ask the AI to create tickets and more. Just describe what you need."
- Auto-dismisses after 8 seconds or on click
- Never shows again

### Tool Call Rendering in Chat
When AI responds with a tool call (SSE type `tool_call`):

**For confirmation-required tools (like CreateTicketTool):**
```
┌──────────────────────────────────────┐
│ 📋 Create Ticket                     │
│                                      │
│ Title: Add retry queue for gateway   │
│ Priority: High · Category: Backend   │
│                                      │
│ Description:                         │
│ Implement a local retry queue to     │
│ buffer peak-hour gateway timeouts... │
│                                      │
│        [Create]  [Edit & Create]     │
└──────────────────────────────────────┘
```

- **"Create"** → POST to `/api/issues/{issue}/chat/tool-confirm`, show success: "✓ Created — #42" with link
- **"Edit & Create"** → opens `CreateIssueDialog` with `prefill` prop (already exists)
- Card updates in-place after creation (no re-render flicker)

### Component Changes
- `IssueChatPanel.vue` — add suggestion chips in empty state, first-time tooltip, placeholder rotation
- `ChatMessage.vue` — add tool call card rendering, contextual nudge detection

- Confirmation-required tools NEVER auto-execute

- Suggestion chips are extensible — each tool provides its own chip text
- Future tools to support (architecture only, don't implement):

### Architecture Notes

1. **Service provider registration** — add `ChatToolRegistry` binding in `AppServiceProvider::register()`, not `boot()`. Use `$this->app->singleton(ChatToolRegistry::class, fn () => tap(new ChatToolRegistry(), fn ($r) => $r->register(new CreateTicketTool())))`. This is the only place tools are registered; no auto-discovery. `app/Providers/AppServiceProvider.php` `register()` is currently empty — safe to add. — `app/Providers/AppServiceProvider.php`

2. **ChatService tool injection** — `ChatService::streamChat()` builds the POST body at line 73 of `app/Services/Ai/ChatService.php`. When `ChatToolRegistry` has registered tools, add a `tools` key to that payload containing `$registry->getToolDefinitions()`. Inject `ChatToolRegistry` via constructor (type-hint; container resolves singleton). Do not pass `tools` when the registry is empty — some providers reject an empty tools array.

3. **Streaming tool_call detection** — at line 115 of `app/Services/Ai/ChatService.php`, the generator currently reads `$parsed['choices'][0]['delta']['content']`. For function-calling responses, `delta` contains `tool_calls` instead of `content`. Check `$parsed['choices'][0]['delta']['tool_calls'] ?? null` first; when present, accumulate the full tool_call JSON across chunks (arguments arrive fragmented), then yield a **distinct structured event** — `{"type":"tool_call","tool":"create_ticket","arguments":{...},"requires_confirmation":true}` — rather than a plain token string. The controller loop at `IssueChatController.php` line 61 must emit this verbatim as `data: <json>\n\n` without the `{"token":"..."}` wrapper.

4. **Fallback for non-function-calling models (Ollama)** — mirror the `TriageService` branching pattern (`app/Services/Ai/TriageService.php` lines 35–43): detect driver via `$settings->effective_driver`. When `tools` param is unsupported, append tool descriptions to the system prompt in `buildMessages()` and instruct the model to respond with `{"tool_call":{"name":"...","arguments":{...}}}` on a standalone line. Parse the streamed full-response text for this exact JSON pattern using `preg_match('/\{"tool_call":\s*\{.*?\}\s*\}/s', $fullText, $m)` before yielding the final content delta. Yield the extracted tool_call as a `type:tool_call` event and the remaining text as normal tokens.

5. **CreateTicketTool execution path** — `CreateTicketTool::execute()` must call `IssueService::create($user, $data)` directly — **never call `IssueController::store()` from inside a service**. Validate the tool parameters manually against the same constraints as `StoreIssueRequest` (`app/Http/Requests/StoreIssueRequest.php`): title max:255, description required, priority in `Priority` enum values, category_id exists in categories table. Resolve `category_id` from the LLM-supplied category name via `Category::where('name', $params['category'])->value('id')`. Return a `ChatToolResult` with `pendingConfirmation=true` when `requiresConfirmation()` is true — the tool does NOT execute; it only validates and returns the pre-flight card data. Actual creation happens at the `tool-confirm` endpoint.

6. **tool-confirm endpoint authorization** — `POST /api/issues/{issue}/chat/tool-confirm` must call `$this->authorize('view', $issue)` (same as chat) **and** `$this->authorize('create', Issue::class)` (same as `IssueController::store()`). `IssuePolicy::create()` returns true for any authenticated user, but this double-gate future-proofs the endpoint and makes the intent auditable. — `app/Policies/IssuePolicy.php` lines 52–58 + ADR-007

7. **GET /api/chat/suggestions routing** — declare this route **before** the `apiResource('issues', ...)` wildcard in `routes/api.php`, following the same precedent as the `triage-suggest` route at line 39. A `Route::get('chat/suggestions', ...)` registered after the apiResource would be shadowed by the `{issue}` wildcard. Use a dedicated `ChatSuggestionsController` or add a method to `IssueChatController`. — `routes/api.php` lines 37–39

8. **SSE event shape contract** — `IssueChatController` currently emits `data: {"token":"<string>"}\n\n` (line 62). The extended contract must preserve backward compatibility: regular tokens continue as `{"token":"..."}`, tool call events emit as `{"type":"tool_call","tool":"...","arguments":{...},"requires_confirmation":<bool>}`. The `consumeStream()` in `useIssueChat.ts` (line 130) already reads `parsed.token ?? parsed.content ?? ''` — add a `parsed.type === 'tool_call'` branch before that fallback to route to `pendingToolCalls` ref without contributing to `streamingContent`. — `resources/js/composables/useIssueChat.ts` lines 129–137

9. **ChatMessage type extension** — `resources/js/types/chat.ts` `ChatMessage.role` is `'user' | 'assistant'`. Tool call cards in the message stream need a third role: `'tool_call'`. Add `role: 'user' | 'assistant' | 'tool_call'` and a `toolCall?: { tool: string; arguments: Record<string, unknown>; requiresConfirmation: boolean; status?: 'pending' | 'confirmed' | 'rejected' }` field. `ChatMessage.vue` currently renders everything non-user as assistant markdown (line 43); add an explicit `v-if message.role === 'tool_call'` branch that delegates to `ToolCallCard.vue`.

10. **Placeholder rotation interval cleanup** — `setInterval` in `useIssueChat.ts` or the panel component must be cleared in `onUnmounted()`. The composable is instance-scoped (not module-level — confirmed by design note at line 6 of `useIssueChat.ts`), so the interval belongs in the panel component which owns the template, cleared via `onUnmounted`. Stop rotating when `messages.value.length > 0` (user has sent a message) or when the input is focused — do not stop on `isStreaming`. — `resources/js/components/issues/IssueChatPanel.vue`

### Gotchas
- **Tool_call argument streaming is fragmented** — OpenAI streams `arguments` as multiple delta chunks (e.g., `{"title":"Add"`, `" retry"`, `"..."}`) that must be concatenated before JSON.parse. Ollama may emit tool_calls differently or not at all; the rules-driver path must be tested with the fallback text-parse strategy. — `app/Services/Ai/ChatService.php` streaming loop
- **tool-confirm is NOT idempotent** — clicking "Create" twice will create two issues. The `ToolCallCard.vue` must set a `loading` ref immediately on click and disable the button. On success, replace the card with a link; on failure, re-enable. Do not rely on network latency to prevent double-submit.
- **CreateTicketTool dispatches GenerateSummaryJob** — `IssueService::create()` at line 43 of `app/Services/IssueService.php` always dispatches `GenerateSummaryJob`. This is correct and expected (new issue gets a summary), but test assertions should account for the queued job — use `Queue::fake()` in `CreateTicketToolTest`.
- **Suggestion chips cache** — `GET /api/chat/suggestions` returns the registry's chips, which are static for the lifetime of the process. Cache the response in the composable after the first successful fetch using a module-level variable (outside the factory function), not per-instance — chips don't change between panel opens on the same page.
- **`CreateIssueDialog` prefill watch fires on open** — `watch(() => props.open, ...)` at line 78 of `CreateIssueDialog.vue` applies prefill only when `isOpen` is true and `props.prefill` is defined. The `ToolCallCard.vue` "Edit & Create" button must set `prefill` before setting `open=true`; both must be in the same tick to avoid a race between the watcher firing with stale prefill.
- **StoreIssueRequest requires `category_id` (integer)**, not a category name string — `CreateTicketTool` must resolve name → ID before calling `IssueService::create()`. If the category name from the LLM doesn't match any row, return a `ChatToolResult` with `success=false` and a human-readable error message rather than throwing. — `app/Http/Requests/StoreIssueRequest.php` line 37
