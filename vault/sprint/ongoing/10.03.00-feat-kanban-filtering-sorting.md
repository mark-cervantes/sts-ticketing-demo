# Task 10.03: Kanban Filtering (My Tickets) + Column-Wide Sorting

## Problem
Users can't quickly find their own tickets on a busy board. There's no way to filter by ownership or sort columns by urgency/recency. On a board with 50+ tickets, this makes the Kanban unusable for individual contributors.

## Solution

### Sidebar Filter: "My Tickets" Toggle
Add a prominent toggle in the existing filter sidebar:
```
┌─────────────────────────────┐
│  👤 MY TICKETS              │
│  [Show only mine]  ○ / ●   │
│                             │
│  ⊙ STATUS                  │
│  ...existing filters...     │
│                             │
│  ⚠ PRIORITY                │
│  ...existing filters...     │
└─────────────────────────────┘
```
- Toggle ON: only show tickets where `issue.user.id === auth.user.id`
- This is client-side filtering (data is already loaded with the user relation)
- Position: top of the sidebar, above STATUS filters (it's the most-used filter)

### Sidebar Filter: Assignee (Future-Ready)
For now, "My Tickets" is a simple owner toggle. The architecture should support a future "Assigned to" dropdown when we add issue assignment.

### Column-Wide Sorting
Each column header gets a sort dropdown (or cycle button):
```
┌──────────────────────────────────┐
│ Open (12)         [↕ Priority]   │
├──────────────────────────────────┤
│  Critical tickets first...       │
│  High tickets next...            │
│  Medium...                       │
│  Low...                          │
└──────────────────────────────────┘
```

Sort options:
- **Priority** (default): Critical → High → Medium → Low
- **Newest first**: by `created_at` descending
- **Recently updated**: by `updated_at` descending
- **Oldest first**: by `created_at` ascending

Sorting is:
- Per-column (each column can have a different sort)
- Client-side (all data is already loaded)
- Persisted in localStorage per column slug

## Technical Guidance

### Grounding
Files read: `resources/js/composables/useIssueFilters.ts`, `resources/js/Layouts/partials/AppSidebar.vue`, `resources/js/components/kanban/KanbanBoard.vue`, `resources/js/components/kanban/KanbanColumn.vue`, `resources/js/composables/useKanbanBoard.ts`, `resources/js/types/issue.ts`, `app/Http/Resources/IssueResource.php`.

### Architecture Notes

- **"My Tickets" state belongs in `useIssueFilters.ts` as a module-scoped `ref<boolean>`**, following the identical singleton pattern used by `filters`. Add `myTickets: boolean` to the `IssueFilters` interface in `resources/js/types/issue.ts` or keep it as a separate sibling ref — either is acceptable, but the exported API must expose `toggleMyTickets()` and `clearFilters()` must reset it to `false`.

- **"My Tickets" is pure client-side — do NOT add it to the `watch` in `useKanbanBoard.ts`.** The existing watch at line 359 of `useKanbanBoard.ts` triggers `loadInitial()` (a full API re-fetch) on priority/category change. "My Tickets" filters `columnMap` data already in memory; wiring it to that watch would cause a needless round-trip and clear any locally-sorted order.

- **Apply "My Tickets" filtering in the `columns` computed in `useKanbanBoard.ts` (lines 127–146), not inside `KanbanColumn.vue`.** The computed already gates issues through `columnMap.value[status.slug]`. Filtering there means `column.issues` and `column.issueCount` (which drives the header badge) automatically reflect the filtered count — the column component receives the already-filtered array with no changes needed.

- **Auth user ID comes from `usePage().props.auth.user.id`** (Inertia shared prop). Read it once at module or composable init level — do not call `usePage()` inside the `columns` computed loop to avoid stale-ref risk.

- **Priority field is a plain string enum** (`'low' | 'medium' | 'high' | 'critical'`). `IssueResource.php` line 30 emits `$this->priority->value` (the enum's string value). `PRIORITY_CONFIG` in `resources/js/types/issue.ts` (line 166) already maps each value to an `order: number` (critical=0 … low=3). The sort comparator must use `PRIORITY_CONFIG[issue.priority].order` — there is no wrapping object to unwrap.

- **Column sort state belongs in a new `useColumnSort.ts` composable** with a `Map<slug, SortKey>` ref, exposing `getSortKey(slug)` and `setSortKey(slug, key)`. `SortKey` should be typed as `'priority' | 'newest' | 'updated' | 'oldest'`. Using a composable (rather than inline `ref` in `KanbanColumn.vue`) keeps sort state alive across column re-renders triggered by `columns` computed updates and allows `KanbanBoard.vue` to pass sorted issues down.

- **Sort application point**: Apply sorting inside the `columns` computed in `useKanbanBoard.ts`, after the "My Tickets" filter and before returning `issues` in the `KanbanColumnDef`. Sorting must happen on the raw `columnMap` array before it reaches `localIssues` in `KanbanColumn.vue` — the column's `watch` (lines 82–91) only resets `localIssues` when the ID set changes; a sort-only reorder won't trigger a reset, meaning VueDraggable's drag state is preserved correctly.

- **localStorage keys**: `kanban-filter-my-tickets` (boolean, stringify to `'true'`/`'false'`) and `kanban-sort-${slug}` per column slug. Read on module init (for `myTickets`) and on composable init or `onMounted` (for sort). Both must be read via a safe wrapper that returns the default (`false` / `'priority'`) if the key is missing or unparseable.

- **`Switch` component exists** at `resources/js/components/ui/switch/Switch.vue`. Use it for the "My Tickets" toggle in `AppSidebar.vue`. Import from `@/components/ui/switch`. `DropdownMenu` and `Badge` components also exist and are the correct primitives for the sort control and filter indicator respectively.

- **`hasActiveFilters` computed in `AppSidebar.vue` (line 62) must be updated** to include `myTickets.value === true` in its condition — otherwise "Clear filters" disappears when only "My Tickets" is active.

### Gotchas
- **`localIssues` reset trap**: `KanbanColumn.vue` watches `props.column.issues` and resets `localIssues` only when the issue ID set changes (lines 82–91). If sort is applied inside the `columns` computed (i.e., same IDs, different order), the watcher will NOT reset `localIssues` — meaning the first render after a sort change won't propagate. To handle this, `KanbanColumn.vue` must also watch for sort key changes (via a prop or a composable call) and apply sorting to `localIssues` directly when the sort key changes, without rebuilding the full array.
- **"My Tickets" filtered counts vs. "Load more"**: `column.issueCount` will drop below `column.hasMore` thresholds when "My Tickets" is active. Display "X of Y" (filtered vs. total in `columnMap`) in the column header badge to prevent confusion. Total count is `columnMap.value[slug].length` before the "My Tickets" filter.
- **Module-scoped singleton risk**: `useIssueFilters` state is module-scoped (not per-component). The `myTickets` ref must also be module-scoped (declared alongside `filters` at the top of the file), or the toggle state will reset on every component mount — inconsistent with how `filters` behaves.
- **`usePage()` must be called at setup time**, not inside computed callbacks. Call `const page = usePage()` once in the exported function body; use `page.props.auth.user.id` inside the computed — do not call `usePage()` inside the `columns` computed itself.
