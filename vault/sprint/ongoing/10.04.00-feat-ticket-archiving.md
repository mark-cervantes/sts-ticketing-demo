# Task 10.04: Ticket Archiving

## Problem
Resolved tickets pile up on the Kanban board indefinitely, making columns noisy and hard to scan. Collapsing the resolved column helps, but doesn't solve the root problem — old tickets never leave the active board. Users need a way to archive resolved tickets so the board stays clean, while still being able to access archived tickets when needed.

## Solution

### Archive Model
- `archived_at` nullable timestamp column on `issues` table
- Archived tickets are excluded from the Kanban board by default
- A toggle in the sidebar lets users show/hide archived tickets
- Archived tickets can be unarchived (restored to the board)

### Archive Actions
1. **Archive single ticket**: Button on the issue detail sheet (only for resolved tickets)
2. **Bulk archive**: "Archive all resolved" button on the resolved column header
3. **Auto-archive** (optional enhancement): Tickets resolved for > 7 days auto-archive via scheduled command
4. **Unarchive**: Button on archived ticket detail sheet

### Kanban Integration
- By default, archived tickets are **excluded** from the board API response
- Sidebar toggle: "Show archived" — when ON, archived tickets appear with a muted/dimmed visual style
- Archived tickets are **not draggable** (they're frozen)
- Column counts exclude archived tickets by default

### Archive View (Optional)
A dedicated `/archive` or `/settings/archive` page listing all archived tickets with search/filter. This is a nice-to-have — the sidebar toggle is the MVP.

## Acceptance Criteria

### Backend
- [ ] Migration: add `archived_at` nullable timestamp to `issues`
- [ ] `Issue` model: `scopeActive()` (where archived_at is null), `scopeArchived()`, `archive()`, `unarchive()` methods
- [ ] `PATCH /api/issues/{id}/archive` — sets `archived_at = now()`
- [ ] `PATCH /api/issues/{id}/unarchive` — sets `archived_at = null`
- [ ] `POST /api/issues/bulk-archive` — archives all resolved tickets (or accepts an array of IDs)
- [ ] Kanban API: default to `scopeActive()` — exclude archived. Accept `?include_archived=1` query param to include them.
- [ ] Authorization: only issue owner or users with edit permission can archive/unarchive
- [ ] PHPUnit tests: archive, unarchive, bulk archive, API filtering, authorization

### Frontend
- [ ] Issue detail sheet: "Archive" button visible on resolved tickets (muted, bottom of sheet)
- [ ] Resolved column header: "Archive all" button (with confirmation dialog)
- [ ] Sidebar: "Show archived" toggle (below "My Tickets" filter)
- [ ] Archived tickets on the board: dimmed card style (opacity, subtle "Archived" badge), not draggable
- [ ] Unarchive button on archived ticket detail sheet
- [ ] Counts: column headers exclude archived by default, show separate count when "Show archived" is on
- [ ] `make verify-visual` passes
- [ ] vue-tsc clean

### Business Rules
- Only resolved tickets can be archived (archiving an open/in-progress ticket makes no sense)
- Archived tickets retain all data (comments, AI synthesis, saved conversations) — nothing is deleted
- Unarchiving restores the ticket to its original status (resolved)
- Bulk archive only affects resolved tickets, not open/in-progress
- Auto-archive: a scheduled artisan command (`issues:auto-archive`) runs daily, archives tickets that have been in a configurable "terminal" column for > N days. Default: resolved column, 7 days. The column and threshold should be configurable via `config/issues.php` or similar:
  ```php
  'auto_archive' => [
      'enabled' => true,
      'statuses' => ['resolved'],  // slugs of columns to auto-archive from
      'after_days' => 7,
  ]
  ```

### Architecture Notes

1. **Migration must add index on `archived_at`**: The `scopeActive()` scope (`WHERE archived_at IS NULL`) and the auto-archive command query (`WHERE archived_at IS NULL AND status IN (...) AND updated_at < ?`) run on every Kanban load and daily schedule. Add `->index()` on the column in `database/migrations/`. — `app/Models/Issue.php` / SPEC §4.2

2. **`archived_at` must be added to `#[Fillable]` and `casts()`**: The existing model uses PHP 8 attribute-style `#[Fillable([...])]` at line 24 of `app/Models/Issue.php`. Add `'archived_at'` there. Cast to `'immutable_datetime'` to match `deadline_at` — not `'datetime'` — so it returns `CarbonImmutable` consistently.

3. **`archive()` / `unarchive()` are model convenience methods, not service methods**: The service (`app/Services/IssueService.php`) handles multi-step logic (optimistic locking, summary re-dispatch). Archive/unarchive are single-column writes with no side-effects — put them as methods directly on `app/Models/Issue.php`, following the existing `computeNeedsAttention` pattern for pure model logic. Do NOT dispatch `GenerateSummaryJob` on archive/unarchive — SPEC §5.3 says regen triggers only on description change.

4. **`PATCH /api/issues/{issue}/archive` and `/unarchive` authorize via `update` ability**: `IssuePolicy::update()` (line 68, `app/Policies/IssuePolicy.php`) already checks owner OR `share->permission->canEdit()`. This is the correct gate — no new policy method needed. Call `$this->authorize('update', $issue)` in the new controller methods, consistent with `IssueController::update()`.

5. **`POST /api/issues/bulk-archive` MUST be declared before `apiResource` in `routes/api.php`**: Lines 43–46 of `routes/api.php` show the established precedent — `triage-suggest` is declared before the `apiResource` wildcard to prevent `{issue}` from consuming it. Apply identical ordering for `bulk-archive`. The controller method lives in `IssueController` (or a dedicated `IssueArchiveController`).

6. **`IssueController::index()` must apply `scopeActive()` by default, toggled by `?include_archived=1`**: The query in `IssueController::index()` (line 47) starts with `Issue::query()->...->accessibleBy($user)`. Chain `->active()` there by default; when `$request->boolean('include_archived')` is true, skip it. This is the only place the scope needs to be applied — `loadMore` and `fetchColumn` in `useKanbanBoard.ts` both call this endpoint.

7. **`bulk-archive` should return `{ archived_count: N }` in its JSON response**: The frontend will display `"Archived N issues"` in a toast (see `moveIssue` toast pattern in `useKanbanBoard.ts` lines 389–390). Without the count, the toast is uninformative. The controller method queries `Issue::query()->accessibleBy($user)->active()->whereHas('status', fn($q) => $q->whereIn('slug', $statuses))->update(['archived_at' => now()])` and returns the affected count.

8. **`auto_archive` config block appends to existing `config/issues.php`**: The file currently only holds `attention_threshold_minutes`. Append the `auto_archive` block from the task Business Rules section. The artisan command reads `config('issues.auto_archive.enabled')`, `config('issues.auto_archive.statuses')`, and `config('issues.auto_archive.after_days')`. Register as `./vendor/bin/sail artisan make:command --command=issues:auto-archive` and add `Schedule::command('issues:auto-archive')->daily()` to `routes/console.php` following the `issues:recalculate-attention` precedent at line 11.

9. **`showArchived` in `useIssueFilters.ts` MUST trigger `loadInitial()` in `useKanbanBoard.ts`**: Unlike `myTickets` (client-side only, intentionally excluded from the watch at line 431 comment), `showArchived` changes the API query param `?include_archived=1`. It must be added to the `watch()` dependency array at line 433–439 of `useKanbanBoard.ts`. Failing to add it means toggling "Show archived" has no effect until a manual page refresh.

10. **`archived_at: string | null` must be added to the `Issue` interface in `resources/js/types/issue.ts`**: The field is emitted by `IssueResource` and consumed by `IssueDetailSheet.vue` (to show/hide archive button) and `IssueCard.vue` (to apply dimmed styling). Without the type, `vue-tsc` will error on property access. Add it after `updated_at` at line 98 of `resources/js/types/issue.ts`.

### Gotchas

- **`showArchived` is NOT a client-side filter like `myTickets`**: The existing comment at line 431 of `useKanbanBoard.ts` explicitly documents why `myTickets` is excluded from the `watch()` — it is client-side only. `showArchived` changes the server query; the coder must add it to the watch so `loadInitial()` re-fetches all columns. Missing this will ship a non-functional toggle.

- **`archive()` model method must NOT touch `updated_at` or trigger the `saving` observer**: `IssueService::update()` (line 88–93 of `app/Services/IssueService.php`) advances `updated_at` manually and disables timestamps to avoid observer conflicts. `archive()` / `unarchive()` should use `Issue::where('id', $this->id)->update(['archived_at' => now()])` (direct query) or `$this->timestamps = false; $this->save()` to avoid the `booted()` `saving` observer recalculating `needs_attention` unnecessarily and updating `updated_at`.

- **`bulk-archive` route naming collision**: `routes/api.php` uses `apiResource('issues', ...)` which registers `issues/{issue}` with an `{issue}` implicit binding. A static route `POST /api/issues/bulk-archive` MUST be declared before this `apiResource` line — exactly as `triage-suggest` is at line 46 — or Laravel will attempt to resolve `"bulk-archive"` as an issue ID and return 404.

- **Auto-archive command must guard against non-existent status slugs in config**: `config('issues.auto_archive.statuses')` is user-editable. The command should resolve slugs to status IDs via `IssueStatus::whereIn('slug', $statuses)->pluck('id')` before querying, and skip silently (with a log warning) if no matching statuses are found — same defensive pattern as `scopeFilterByStatus()` in `app/Models/Issue.php` line 141.

- **`IssueResource` `can` block does not include an `archive` ability yet**: `IssueDetailSheet.vue` needs to know whether to show the archive button. Add `'archive' => $request->user() ? Gate::allows('update', $this->resource) : false` to the `can` block in `app/Http/Resources/IssueResource.php` line 89–94, reusing `update` ability per Architecture Note 4. The frontend reads `issue.can?.archive` — the type must also add this to the `can` shape in `resources/js/types/issue.ts`.

- **Archived card `no-drag` class**: `KanbanColumn.vue` already uses the `.no-drag` CSS class (line 538) to block SortableJS dragging for view-only issues. Apply the same class to archived cards by checking `issue.archived_at !== null` in the `:class` binding at line 538. Do not add a new SortableJS filter — reuse the existing `.no-drag` mechanism.
